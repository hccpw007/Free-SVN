## 最高优先级规则

以下 4 条规则均为最高优先级，本次对话必须无条件遵守。越靠前优先级越高：

> **严格遵循本文件全部要求** — 每次对话必须严格遵守本 `CLAUDE.md` 的所有规则和约定。在「表一：各阶段文档引用」列明的时机到达时，必须读取该时机对应的文档，不得跳过或凭记忆替代。本文件是 AI 行为的总约束，任何时候不得以任何理由偏离。每个时机对应的文档必须在该时机到达时读取，不得提前跳过或事后补读。

> **每次对话结束前评估死代码** — 每次对话结束前，都要评估本次对话的修改内容是否产生死代码。如果有，及时清除，不管是否有死代码，都需要告诉我结果。

> **每次对话结束前执行 git 提交** — 每次对话结束前，执行 `git add -A && git commit -m "本次修改的中文总结"`，无需询问直接执行。

> **引用文件必须读取，不准凭记忆回答** — 在对话中如果要引用文件信息（路径、代码、配置、文档等），必须先读取文件内容。不因担心上下文过长而省略关键信息或跳过必要的文件读取。上下文费用不是约束，输出错误信息的代价远高于 token 消耗。

## 编码准则

1. **有执行步骤时严格按步骤编码** — 不自行引入设计变更；无执行步骤时以对话要求为准
2. **只写最简代码** — 不引入未请求的特性或抽象层
3. **外科手术式修改** — 不改动不相关的代码
4. **超限文件及时拆分** — 对话中如果修改的文件超过文件拆分标准（`doc/01-项目规范/` 下 01-Vue前端开发规范 §08 和 02-Rust后端开发规范 §文件拆分标准），必须在本次对话内拆分。拆分后确保编译/测试通过。
5. **可验证的成功标准** — 定义目标并循环直到确认完成
