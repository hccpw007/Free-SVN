## 项目概述

Free-SVN — 基于 Tauri + Vue 的跨平台 SVN 图形化工具（macOS / Windows / Linux）。

### 架构概要

**前后端通信**：前端通过 Tauri `invoke` 调用 Rust command（每种 SVN 操作一个 command），长操作（更新/检出）通过 Tauri event 推送进度到前端。

**前端**：Vue 3 + TypeScript + Vite + Tailwind CSS 4 + Element Plus + Pinia（4 个 store：workspace、fileList、settings、svn）+ Vue Router + vue-i18n。差异对比使用 `diff2html` + `highlight.js`。

**后端**：Tauri 2（Rust）+ serde/serde_json 序列化 + thiserror 错误处理 + tauri-plugin-store 持久化。所有 SVN 操作调用系统 `svn` CLI 并解析 `--xml` 输出，60 秒超时，异步执行（`spawn_blocking`）。

## 前端样式规范

仅使用 Tailwind CSS 编写样式，**禁止**写 `<style>` 标签或任何自定义 CSS 文件。

## 编码准则（Karpathy）

1. **先想清楚再编码** — 主动暴露假设和不确定性，不做默认选择
2. **只写最简代码** — 不引入未请求的特性或抽象层
3. **外科手术式修改** — 不改动不相关的代码
4. **可验证的成功标准** — 定义目标并循环直到确认完成

> **注**："先想清楚"在 `doc/02-设计文档/` 和 `doc/03-执行计划/` 阶段完成。进入编码阶段后严格按计划步骤执行，不再做设计决策。

## AI 编程模式

本项目是 **AI 主导编程** 项目，所有代码由 AI 生成和维护。

执行流程：`doc/02-设计文档/` 定义功能需求+拆分方案 → `doc/03-执行计划/` 创建执行步骤+代码细节 → AI 执行编码。

文档分工与引用规则：

### doc/02-设计文档/ — 设计文档（仅用于生成开发步骤）

设计文档定义"做什么"，每个功能定义必须包含充分的执行信息（如输入/输出/行为/边界等）。设计文档仅用于生成 `doc/03-执行计划/` 中的开发步骤。

### doc/03-执行计划/ — 开发步骤（AI 编码的唯一参考）

开发步骤是设计文档以 AI 能具体执行代码生成的细化文档，定义"怎么做"。**写步骤时必须逐条对照设计文档，确认覆盖了全部输入/输出/行为/边界，不得遗漏**。

AI 编码时以开发步骤为准**而非设计文档**。编码时如果发现开发步骤和设计文档不一致，**以开发步骤为准**。如果开发步骤描述不够具体无法编码，**不要自行回读设计文档补充细节**——直接向用户提问。

每个执行步骤必须包含以下字段：

| 字段 | 要求 |
|------|------|
| 涉及文件 | 列出所有新建/修改的文件路径，标注操作类型 |
| 操作细节 | 具体代码结构、函数签名、修改位置、逻辑描述。**禁止仅写需求目标** |
| 验收标准 | 可验证的完成条件（如"登录表单包含账号/密码输入框和登录按钮"） |

### 编码执行规则

编码前必须读取 `doc/01-开发规范/01-Vue前端开发规范.md` 和 `doc/01-开发规范/02-Rust后端开发规范.md`。

1. **一次只执行一个步骤** — 每次只读取一个步骤文件，执行完成后验证通过才能读取下一步
2. **每步执行后逐条验收** — 对照该步骤的"验收标准"逐条打勾确认并在当前步骤末尾输出验收结果：
   ```
   ✅ 全部通过 → 进入下一步
   ❌ 不通过 → 先自行修正代码再验收，重复最多 1 次
   ❌ 仍不通过 → 停止并展示失败项，向用户提问
   ```
3. **步骤字段不完整或矛盾必须追问** — 以下情况必须向用户提问，不得编码：
   - "涉及文件"中没有文件路径
   - "操作细节"中没有代码结构或函数签名
   - "验收标准"中没有可验证的条件
   - "涉及文件"的文件类型与"操作细节"描述不一致（如涉及文件是 `.vue` 但操作细节写的是 Rust 后端逻辑）

报告类文档（`doc/99-评估报告/`）不受此约束。

## 文档规范

所有文档存放在 `doc/` 目录下，按类别分目录、以序号开头：

| 目录 | 用途 |
|------|------|
| `doc/01-开发规范/` | 规范、规则 |

| `doc/03-执行计划/` | 开发计划 |

- 文件名以 `01-` 序号开头，**使用中文命名**

### 文档版本命名规则（设计文档 & 评估报告）

**核心原则：所有设计文档和评估报告必须从 v1 开始就放在子文件夹中，后续修改一律生成新版本文件，不得原地修改。**

```
# 正确示例
doc/02-设计文档/
├── 01-业务功能设计/
│   ├── 01-业务功能设计-v1.md       # 初始版本
│   ├── 01-业务功能设计-v2.md       # 修改时新增，不修改 v1
│   └── 01-业务功能设计-v3.md       # 再次修改时继续新增

doc/99-评估报告/
├── 03-设计文档评估报告/
│   ├── 03-设计文档评估报告-v1.md
│   ├── 03-设计文档评估报告-v2.md
│   └── ...

# 错误示例 — 禁止
doc/02-设计文档/
├── 01-业务功能设计.md              # ✗ 平级文件，无子文件夹
├── 02-其他设计/
│   └── 02-其他设计.md              # ✗ 无版本号后缀
```

规则：
1. **子文件夹带序号** — 文件夹以序号开头，如 `01-业务功能设计/`
2. **统一基底文件名** — 子文件夹内所有文件共用同一基底名称（含序号），如 `01-业务功能设计-v1.md`
3. **版本号后缀** — 依次追加 `-v1`、`-v2`、`-v3`（v1 不省略，始终标注版本号）
4. **首次即子文件夹** — 即使当前只有 v1，也放在子文件夹中，为后续版本预留空间。禁止平级单文件（如 `01-业务功能设计.md`）
5. **新增版本文件，不修改旧版本** — 对某份文档的内容修改时，**不得修改当前文档**，而是在同目录下新建 `-v2`（或后续版本号）文件。原始 v1 文件保持只读，用于追溯变更历史
6. **版本号连续递增** — 新版本号在目录中最大版本号基础上 +1，不得跳号
