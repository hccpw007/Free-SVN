## 最高优先级规则

> **规则一（最高）：每次对话结束前评估死代码** — 每次对话结束前，都要评估本次对话的修改内容是否产生死代码。如果有，及时清除，不管是否有死代码，都需要告诉我结果。此规则优先于其他所有规则。

> **规则二：每次对话结束前执行 git 提交** — 每次对话结束前，执行 `git add -A && git commit -m "本次修改的中文总结"`，无需询问直接执行。

> **规则三（最高）：引用文件必须读取，不准凭记忆回答** — 在对话中如果要引用文件信息（路径、代码、配置、文档等），必须先读取文件内容。不因担心上下文过长而省略关键信息或跳过必要的文件读取。上下文费用不是约束，输出错误信息的代价远高于 token 消耗。

## 编码准则

1. **严格按执行步骤编码** — 不自行引入设计变更，按步骤文档执行
2. **只写最简代码** — 不引入未请求的特性或抽象层
3. **外科手术式修改** — 不改动不相关的代码
4. **可验证的成功标准** — 定义目标并循环直到确认完成

## AI 编程模式

本项目是 **AI 主导编程** 项目，所有代码由 AI 生成和维护。各文档角色定义详见 `doc/01-项目规范/04-文档生成规范.md`。

### 编码执行规则

1. **先通读后单步执行** — 编码前先读整体开发计划了解本期范围，然后一个步骤文件一个步骤文件执行
2. **逐条验收** — 对照当前步骤的"验收标准"逐条打勾：
   - ✅ 全部通过 → 进入下一步
   - ❌ 存在失败项 → 修正后重新验收
   - ❌ 修正后仍不通过 → 展示失败项并向用户提问
3. **字段不完整或矛盾必须追问** — 步骤缺少"涉及文件"/"操作细节"/"验收标准"或内容与文件类型矛盾时，不得自行脑补，直接提问
4. **步骤细节不足时，不回读设计文档补充** — 步骤是编码唯一参考，细节不足说明步骤本身不完整，应修复步骤而非绕过它
5. **步骤与设计文档不一致时，以步骤为准执行，并在验收结果中标注不一致之处**
