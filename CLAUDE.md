## 最高优先级规则

> **规则一（最高）：每次对话结束前评估死代码** — 每次对话结束前，都要评估本次对话的修改内容是否产生死代码。如果有，及时清除，不管是否有死代码，都需要告诉我结果。此规则优先于其他所有规则。

> **规则二：每次对话结束前执行 git 提交** — 每次对话结束前，执行 `git add -A && git commit -m "本次修改的中文总结"`，无需询问直接执行。

> **规则三（最高）：引用文件必须读取，不准凭记忆回答** — 在对话中如果要引用文件信息（路径、代码、配置、文档等），必须先读取文件内容。不因担心上下文过长而省略关键信息或跳过必要的文件读取。上下文费用不是约束，输出错误信息的代价远高于 token 消耗。

## 编码准则

1. **严格按执行步骤编码** — 不自行引入设计变更，按步骤文档执行
2. **只写最简代码** — 不引入未请求的特性或抽象层
3. **外科手术式修改** — 不改动不相关的代码
4. **可验证的成功标准** — 定义目标并循环直到确认完成

## AI 编程模式

本项目是 **AI 主导编程** 项目，所有代码由 AI 生成和维护。

执行流程：`doc/02-设计文档/` 定义功能需求+拆分方案 → `doc/03-执行计划/` 创建执行步骤文件 → AI 按步骤文件逐一编码。各文档的角色定义和步骤文件格式详见 `doc/01-项目规范/04-文档生成规范.md`。

### 编码执行规则

编码前必须读取 `doc/01-项目规范/01-Vue前端开发规范.md`、`doc/01-项目规范/02-Rust后端开发规范.md` 和 `doc/01-项目规范/03-代码注释规范.md`。

0. **编码前通读整体开发计划** — 读取 `doc/03-执行计划/01-整体开发计划/01-开发计划/` 中的最新版本计划文档，了解当前阶段要实现的全部功能范围，然后按步骤逐一执行
1. **一次只执行一个步骤** — 步骤文件位于 `doc/03-执行计划/01-整体开发计划/02-执行步骤/phase-N/` 目录下，每次只读取一个步骤文件，执行完成后验证通过才能读取下一步
2. **每步执行后逐条验收** — 对照该步骤的"验收标准"逐条打勾确认并在当前步骤末尾输出验收结果：
   ```
   ✅ 全部通过 → 进入下一步
   ❌ 不通过 → 先自行修正代码再验收，重复最多 1 次
   ❌ 仍不通过 → 停止并展示失败项，向用户提问
   ```
3. **步骤字段不完整或矛盾必须追问** — 以下情况必须向用户提问，不得编码：
   - "涉及文件"中没有文件路径
   - "操作细节"中没有代码结构或函数签名
   - "验收标准"中没有可验证的条件
   - "涉及文件"的文件类型与"操作细节"描述不一致（如涉及文件是 `.vue` 但操作细节写的是 Rust 后端逻辑）

报告类文档（`doc/99-评估报告/`）不受此约束。
