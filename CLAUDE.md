## 最高优先级规则

每次对话结束前，执行 `git add -A && git commit -m "本次修改的中文总结"`，无需询问直接执行。

## 项目概述

Free-SVN — 基于 Tauri + Vue 的跨平台 SVN 图形化工具（macOS / Windows / Linux）。

## 前端样式规范

仅使用 Tailwind CSS 编写样式，**禁止**写 `<style>` 标签或任何自定义 CSS 文件。

## 编码准则（Karpathy）

1. **先想清楚再编码** — 主动暴露假设和不确定性，不做默认选择
2. **只写最简代码** — 不引入未请求的特性或抽象层
3. **外科手术式修改** — 不改动不相关的代码
4. **可验证的成功标准** — 定义目标并循环直到确认完成

> **注**："先想清楚"在 `doc/design/` 和 `doc/plan/` 阶段完成。进入编码阶段后严格按计划步骤执行，不再做设计决策。

## AI 编程模式

本项目是 **AI 主导编程** 项目，所有代码由 AI 生成和维护。

执行流程：`doc/design/` 定义功能需求+拆分方案 → `doc/plan/` 创建执行步骤+代码细节 → AI 执行编码。

文档分工与引用规则：

### doc/design/ — 设计文档（仅用于生成开发步骤）

设计文档定义"做什么"，每个功能定义必须包含充分的执行信息（如输入/输出/行为/边界等）。设计文档仅用于生成 `doc/plan/` 中的开发步骤。

### doc/plan/ — 开发步骤（AI 编码的唯一参考）

开发步骤是设计文档以 AI 能具体执行代码生成的细化文档，定义"怎么做"。**写步骤时必须逐条对照设计文档，确认覆盖了全部输入/输出/行为/边界，不得遗漏**。

AI 编码时以开发步骤为准**而非设计文档**。编码时如果发现开发步骤和设计文档不一致，**以开发步骤为准**。如果开发步骤描述不够具体无法编码，**不要自行回读设计文档补充细节**——直接向用户提问。

每个执行步骤必须包含以下字段：

| 字段 | 要求 |
|------|------|
| 涉及文件 | 列出所有新建/修改的文件路径，标注操作类型 |
| 操作细节 | 具体代码结构、函数签名、修改位置、逻辑描述。**禁止仅写需求目标** |
| 验收标准 | 可验证的完成条件（如"登录表单包含账号/密码输入框和登录按钮"） |

### 编码执行规则

1. **一次只执行一个步骤** — 每次只读取一个步骤文件，执行完成后验证通过才能读取下一步
2. **每步执行后逐条验收** — 对照该步骤的"验收标准"逐条打勾确认并在当前步骤末尾输出验收结果：
   ```
   ✅ 全部通过 → 进入下一步
   ❌ 未通过 → 修正代码后再验收
   ```
3. **步骤字段不完整必须追问** — 以下情况必须向用户提问，不得编码：
   - "涉及文件"中没有文件路径
   - "操作细节"中没有代码结构或函数签名
   - "验收标准"中没有可验证的条件

报告类文档（`doc/report/`）不受此约束。

## 文档规范

所有文档存放在 `doc/` 目录下，按类别分目录、以序号开头：

| 目录 | 用途 | 命名示例 |
|------|------|----------|
| `doc/design/` | 设计文档（定义功能需求：输入/输出/行为/边界 + 文件拆分方案） | `01-架构设计.md` |
| `doc/plan/` | 开发计划（按阶段分文件夹，每步独立文档，含具体执行步骤 + 操作命令 + 实际代码细节） | `01-账号模块/1.1-登录.md` |
| `doc/rule/` | 规范、规则 | `01-Vue前端开发规范.md` 等 |
| `doc/report/` | 评估报告 | `01-架构评估报告.md` |

- 文件名以 `01-`、`02-` 等序号开头，**使用中文命名**（如 `01-账号模块/1.1-登录.md`）
- 主题内容较多时创建子文件夹（如 `02-数据库设计/schema.md`）
- `doc/plan/` 按开发阶段划分子文件夹，**每个执行步骤一个独立文件**，步骤编号含阶段号（如 `01-账号模块/1.2-注册.md`、`02-核心功能/2.1-文件列表.md`）
