## 操作细节

### 函数签名

在 `executor.rs` 中新增异步函数 `run_svn_with_progress`：

```rust
pub async fn run_svn_with_progress(
    args: &[&str],
    cwd: &str,
    credentials: Option<&crate::svn::types::SvnCredentials>,
    app_handle: tauri::AppHandle,
    operation: &str,
) -> Result<String, AppError>
```

### 双管道线程模型

在 `tokio::task::spawn_blocking` 闭包内使用两个 `std::thread::spawn` 线程分别读取 stdout 和 stderr，通过 `std::sync::mpsc::channel` 发送行数据到主处理循环：

```
spawn_blocking 外层（Tauri 异步兼容）
  ├─ stdout 线程（thread::spawn）→ mpsc::Sender<String> → 主循环（解析文件行 + emit operation:line）
  ├─ stderr 线程（thread::spawn）→ mpsc::Sender<String> → 主循环（解析百分比 + emit operation:progress，带 throttle）
  └─ 主循环（在 spawn_blocking 闭包内同步运行）
```

说明：`spawn_blocking` 确保 Tauri 异步运行时不被阻塞，内部 `thread::spawn` 实现真正的并行 I/O 读取。

### 线程逐行读取

每个线程内：
1. 获取 pipe reader（`BufReader::new(stdout.take().unwrap())`）
2. 每次 `read_line` 前检查 `CANCELLED` 标志（`Ordering::SeqCst`）
3. `catch_unwind` 包裹读取逻辑
4. 读取到行后通过 channel 发送

### 后端 Throttle（200ms）

- 首条进度行立即发送
- 后续 200ms 窗口内最多发送一条
- 使用 `Instant::now()` 时间戳比较

### 文件行解析

根据 operation 类型（checkout/update/commit/switch/merge/export）选择不同的正则：

| 操作 | 匹配模式 | 文件路径提取 |
|------|---------|------------|
| checkout | `^(A)\s+(.+)$` | 第 2 组 |
| update | `^(U|A|D)\s+(.+)$` | 第 2 组 |
| commit | `^Sending\s+(.+)$` | 第 1 组 |
| switch | `^(U|A|D)\s+(.+)$` | 第 2 组 |
| merge | `^(U|A|D|C)\s+(.+)$` | 第 2 组 |
| export | `^(A)\s+(.+)$` | 第 2 组 |

解析出 filePath 后 emit `operation:line` 事件。

### 进度行解析

stderr 行匹配两种模式：
- 标准进度：`^\s*(\d+)%\s`
- commit Transmitting 阶段：`^\s*\.\.\.\s(\d+)%\s`

提取百分比后 emit `operation:progress`。

### 取消协调

- stdout/stderr 线程每次读取一行前检查 `is_cancelled()`
- 主循环每次处理一行后检查 `is_cancelled()`
- 取消后不再 emit 新事件
- 取消事件使用 `CancelledPayload` 结构体（在 `types.rs` 中定义，含 `reason: String` 字段），通过 `operation:cancelled` 事件推送

### stdout 管道关闭标记

stdout 线程检测到 BufReader 返回 `Ok(0)`（EOF/管道关闭）时，push `"__STDOUT_CLOSED__:{operation}"` 标记行。

### panic 恢复 + 5s 超时兜底

- 每线程用 `catch_unwind` 包裹
- panic 时关闭对应 channel
- 两个 receiver 都关闭后启动 5s 超时
- 5s 后若进程未退出，`kill_current_process()`

## 事件生命周期约束

事件生命周期（源自操作进度提示设计-v4 §5.1）：

```
operation:started → (operation:line × N + operation:progress 节流)
                  → operation:completed | operation:error | operation:cancelled
```

**约束说明：**
- `operation:cancelled` 仅在 `operation:started` 之后、`operation:completed`/`operation:error` 之前有效
- 操作完成或出错后，前端不应再监听 cancelled 事件
- `operation:line` 与 `operation:progress` 独立并发，不相互阻塞
