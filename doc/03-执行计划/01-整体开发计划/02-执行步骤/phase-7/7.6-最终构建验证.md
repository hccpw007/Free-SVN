## 操作细节

### 7.6A — 验证顺序

按顺序执行，逐项修复到零报告：

1. `cd src-tauri && cargo build` — 修复编译错误/警告（unused import/variable、unwrap）
2. `cd src-tauri && cargo test` — 修复测试失败
3. `cd src-tauri && cargo clippy` — 修复 clippy 警告（详见下方 clippy 警告处理策略）
4. `cnpm run build` — 修复 TS 类型错误和 Vite 错误
5. `cnpm run lint` — 修复 ESLint 错误
6. `cnpm run knip` — 删除未引用文件，重复 4-6 直到零报告
7. 最终命令：`cargo build && cargo test && cnpm run build && cnpm run lint`

### 7.6B — clippy 警告处理策略

`cargo clippy` 可能产生大量警告。以下策略确定哪些警告允许忽略、哪些必须修复：

**必须修复的 clippy 警告（导致 CI 失败）：**

| lint 规则 | 说明 | 修复方式 |
|-----------|------|---------|
| `clippy::all` 默认集 | 包含正确性/性能/可读性问题 | 按建议修改代码 |
| `clippy::unwrap_used` | 非必要 unwrap 可能 panic | 替换为 `?` / `unwrap_or` / 错误处理 |
| `clippy::expect_used` | 同上 | 同上 |
| `clippy::panic` | 禁止直接调用 panic! | 替换为错误返回 |
| `clippy::todo` | 未实现的占位代码 | 实现功能或改为 `unimplemented!()` 并记录 |
| `clippy::dbg_macro` | 残留调试代码 | 删除或改为 `log::debug!()` |
| `clippy::print_stdout` | 生产代码禁止直接 print | 替换为 `log::info!()` / `log::debug!()` |
| `clippy::needless_return` | 不必要的 return | 移除末尾 return |
| `clippy::unnecessary_cast` | 不必要的类型转换 | 移除 as 转换 |
| `clippy::redundant_closure` | 冗余闭包 | 简化为函数引用 |
| `clippy::useless_conversion` | 无用的 Into/From 转换 | 移除 |

**允许忽略的 clippy 警告（不阻塞构建，不需要修复）：**

| lint 规则 | 允许理由 |
|-----------|---------|
| `clippy::too_many_arguments` | 命令参数多（SVN CLI 参数多）是领域特性，拆分函数签名会降低可读性 |
| `clippy::type_complexity` | 复杂类型别名在 parser 层中不可避免 |
| `clippy::module_name_repetitions` | 模块前缀在大型项目中提高辨识度 |
| `clippy::similar_names` | SVN CLI 中存在大量相似词汇（如 revision/rev/merge） |
| `clippy::must_use_candidate` | 公共 API 函数需要全局决策是否都加 #[must_use]，非本轮范围 |
| `clippy::cast_lossless` | 数值类型转换精度损失在 SVN 版本号范围内可控（u64 <-> i64） |
| `clippy::cast_possible_truncation` | 同上（SVN 版本号不超出 u32 范围） |
| `clippy::cast_sign_loss` | 同上 |
| `clippy::enum_variant_names` | SVN 错误码枚举命名保持一致性的需要优先于 lint 建议 |

**配置方式（在 `src-tauri/Cargo.toml` 或 `src-tauri/src/lib.rs` 中添加 `#![allow(...)]`）：**

```toml
# Cargo.toml — 在 [lints.clippy] 中配置允许的 lint
[lints.clippy]
too_many_arguments = "allow"
type_complexity = "allow"
module_name_repetitions = "allow"
similar_names = "allow"
cast_lossless = "allow"
cast_possible_truncation = "allow"
cast_sign_loss = "allow"
enum_variant_names = "allow"
```

或者在代码根添加：
```rust
// src-tauri/src/lib.rs 顶部
#![allow(clippy::too_many_arguments, clippy::type_complexity)]
```

> **建议：** 使用 `Cargo.toml` 的 `[lints.clippy]` 配置方式（更集中、不影响内联代码）。仅在极少数需要例外的地方使用 `#[allow(...)]` 属性。

**执行顺序：** 先修复"必须修复"的警告类别至零报告，再决定是否处理"允许忽略"类别。验收标准要求 `cargo clippy` 零警告 —— 包括允许忽略的 lint 也最好清零（可在后续迭代中完成），但至少"必须修复"类别必须清零。

### 7.6C — 常见问题与修复指引

| 问题 | 修复指引 |
|------|---------|
| Rust 编译错误：未使用变量 | 删除变量或将引用改为 `_` 或 `_var` 前缀。例如 `let x = ...` → `let _x = ...` |
| Rust 编译错误：未使用 import | 删除 `use` 语句或在编辑器中使用 cargo fix --allow-dirty 自动清理 |
| Rust 编译错误：类型不匹配 | 检查函数签名和调用处的参数类型是否一致，重点检查 Option/Result 的 unwrap/expect 使用 |
| Rust 编译错误：generate_handler! 缺失 | 确保新增的 command 已在 `lib.rs` 的 `generate_handler!` 宏中注册 |
| Rust 编译错误：tauri 版本不匹配 | 检查 Cargo.toml 和 package.json 中的 tauri 版本号一致，运行 `cargo update` |
| Rust 编译错误：模块未声明 | 在 `src-tauri/src/lib.rs` 或 `mod.rs` 中添加 `pub mod module_name;` |
| Rust 编译错误：move 闭包捕获 | 在闭包外预 clone 需要的值，或在 move 闭包前 clone 引用。例如 `let handle = app.clone();` 后在 spawn 中使用 `handle` |
| Rust 编译错误：关联类型不匹配 | 检查 trait 实现是否正确，特别是 From/Into/Display/Deref 等 trait 的实现 |
| Clippy all 默认集 | 按 clippy 建议修改代码，使用 `cargo clippy --fix` 自动修复部分警告 |
| Clippy unwrap_used | 将 `.unwrap()` 替换为 `?` 操作符、`.unwrap_or()`、`.unwrap_or_default()` 或 `.ok_or(AppError::xxx)?` |
| Clippy print_stdout | 将 `println!()` / `print!()` 替换为 `log::info!()` / `log::debug!()` |
| Clippy needless_return | 移除函数末尾的 `return` 关键字 |
| TS 编译错误：类型不匹配 | 对齐 Rust 命令返回类型与 TypeScript 接口定义。检查 `types/svn.ts` 中的类型定义是否与 Rust 端的序列化结构体一致 |
| TS 编译错误：模块未找到 | 检查 import 路径是否正确（大小写敏感），确认文件存在 |
| TS 编译错误：未使用的变量 | 删除变量或在参数前加 `_` 前缀 |
| Vite 构建错误：静态资源缺失 | 检查 `public/` 目录下的资源文件是否存在，Vite 的 `public/` 路径与 `src/` 路径不同 |
| ESLint 错误：规则违反 | 按 ESLint 提示修改代码，或检查 `.eslintrc` / `eslint.config.*` 中的规则配置 |
| knip 报告：未使用文件 | 检查文件是否确实未使用，确认后删除。若是动态导入使用的文件，需检查 knip 配置中的 entry 文件列表 |
