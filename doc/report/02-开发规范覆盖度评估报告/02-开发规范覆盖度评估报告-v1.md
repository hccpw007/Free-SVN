## 三、本轮新增发现

在第二轮深入审查中，发现以下次要问题。这些问题不影响整体评级，但修正后可提升编码一致性。

### 3.1 问题 A：`cancel.rs` 示例代码违反错误码原则

**位置**：Rust 规范 §04 长操作取消模式（第 549 行）

```rust
state.app_handle.emit("operation:completed", "已取消").ok();
//                                            ^^^^^^^^
//                       违反了 §03 错误码规则：后端应返回错误码 key 而非中文文字
```

该行直接发射了中文字符串 `"已取消"` 作为 event payload，违反了 §03 "后端返回错误码 key（如 `SVN_CANCELLED`）而非中文文字"的原则。

**建议**：改为发送错误码：
```rust
let payload = serde_json::json!({ "result": "cancelled", "code": "SVN_CANCELLED" });
state.app_handle.emit("operation:completed", payload).ok();
```

### 3.2 问题 B：`check_network` 端口号硬编码

**位置**：Rust 规范 §04 网络可达性检测（第 560 行）

```rust
tokio::net::TcpStream::connect((host.as_str(), 3690))
//                                               ^^^^
//     硬编码 svnserve 默认端口，不支持 https (443) / svn+ssh (22)
```

SVN 服务器可能运行在 `svn://`（3690）、`https://`（443）、`http://`（80）、`svn+ssh://`（22）等协议上。硬编码 3690 会导致多数实际场景下的网络检测失败。

**建议**：解析 `server_url` 的协议部分，确定正确的端口号：
```rust
fn get_svn_port(server_url: &str) -> u16 {
    if server_url.starts_with("https://") { 443 }
    else if server_url.starts_with("http://") { 80 }
    else if server_url.starts_with("svn+ssh://") { 22 }
    else { 3690 } // svn:// 或未知
}
```

### 3.3 问题 C：`isOperationRunning` 状态归属不合理

**位置**：Vue 规范 §05 `fileListStore`

```ts
// stores/fileList.ts
const isOperationRunning = ref<boolean>(false)
```

`isOperationRunning` 控制着工具栏按钮、对话框打开、键盘快捷键等多个全局行为。将其放在 `fileListStore` 中意味着 `ToolBar.vue`、`dialogs/*` 都必须 import `fileListStore` 来判断操作状态——这不自然。

`isOperationRunning` 不属于任何一个业务 Store，它是**全局 UI 状态**。

**建议**：

方案 A（推荐）：在 `composables/useOperationGuard.ts` 中管理，通过 provide/inject 或简单模块级 ref 共享：
```ts
// composables/useOperationGuard.ts
import { ref } from 'vue'

const isRunning = ref(false)

export function useOperationGuard() {
  function start() { isRunning.value = true }
  function finish() { isRunning.value = false }
  return { isRunning, start, finish }
}
```

方案 B：新增 `stores/uiState.ts` 存放全局 UI 互斥状态。

### 3.4 问题 D：前端测试策略仍偏薄弱

**位置**：Vue 规范 §10

与 Rust 规范 §08 的充分补充相比，Vue 规范 §10 仍只有基础描述（使用 Vitest + Vue Test Utils），缺少：
- 组件测试模板（如 `mount(Component, { props })` → assert rendered output）
- Store 测试模板（直接调用 action → 验证 state）
- Tauri invoke 的 mock 策略（如何 mock `@tauri-apps/api`）
- 覆盖率目标

**建议**：补充至少一个 Store 测试代码示例和一个 Tauri invoke mock 示例。

### 3.5 问题 E：TypeScript 错误码枚举示例缺失

**位置**：Rust 规范 §03 统一错误码目录

规范要求"前端 `types/error-codes.ts` 中定义与后端完全相同的错误码常量集合"，但未给出前端应该怎么写。

**建议**：补充 TypeScript 侧的错误码定义示例：
```ts
// types/error-codes.ts
export const ErrorCodes = {
  SVN_EXEC_FAILED: 'SVN_EXEC_FAILED',
  SVN_PARSE_FAILED: 'SVN_PARSE_FAILED',
  // ... 完整 11 个
} as const

export type ErrorCode = typeof ErrorCodes[keyof typeof ErrorCodes]
```
