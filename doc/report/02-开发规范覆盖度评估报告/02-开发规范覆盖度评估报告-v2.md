## 二、P0 缺项修复验证

### ☑ P0-1：Tauri Event 协议（Rust 规范 §02）

**修改内容**：新增完整事件协议表

| 事件名 | Payload | 触发时机 |
|--------|---------|---------|
| `operation:started` | `{ operation: "checkout" }` | 长操作开始时 |
| `operation:progress` | `{ percent, stage, fileCount }` | 每处理 N 个文件或阶段变更 |
| `operation:completed` | `{ result, detail }` | 正常完成/取消/失败 |
| `operation:error` | `{ errorCode, message }` | 操作过程中发生错误 |

**验证结论**：✅ 4 个事件完整覆盖了"开始 → 进度 → 完成/失败"的完整生命周期。前后端 payload 类型定义（Rust struct + TypeScript interface）同步给出。前端监听示例代码包含。

## 四、本轮新发现

以下问题不影响"进入 Plan 阶段"的结论，建议编码过程中顺手修正。

### 4.1 示例代码违反错误码原则

**位置**：Rust 规范 §04 取消模式（`cancel.rs`）

```rust
state.app_handle.emit("operation:completed", "已取消").ok();
//                                            ^^^^^^^^
//            中文字符串，违反 §03 错误码规则
```

该行直接发射了中文字符串 `"已取消"` 作为 event payload，与 §03 "后端返回错误码 key 而非中文文字"的原则矛盾。建议改为：

```rust
let payload = serde_json::json!({
    "result": "cancelled",
    "code": "SVN_CANCELLED"
});
state.app_handle.emit("operation:completed", payload).ok();
```

**严重度**：轻微（规范示例代码，非生产代码）。

### 4.2 `check_network` 端口硬编码

**位置**：Rust 规范 §04 网络可达性检测

```rust
tokio::net::TcpStream::connect((host.as_str(), 3690))
```

硬编码 3690（svnserve 默认端口），但 SVN 服务器可通过 `https://`（443）、`http://`（80）、`svn+ssh://`（22）访问。HTTPS 是企业 SVN 的常见部署方式，硬编码会导致网络检测始终失败。

**建议**：解析 URL 协议确定端口：
```rust
fn get_svn_port(url: &str) -> u16 {
    if url.starts_with("https://") { 443 }
    else if url.starts_with("http://") { 80 }
    else if url.starts_with("svn+ssh://") { 22 }
    else { 3690 }
}
```

**严重度**：中等 — 可能影响 HTTPS SVN 用户的按钮启用逻辑。

### 4.3 `isOperationRunning` 归属不恰当

**位置**：Vue 规范 §05 `fileListStore`

```ts
const isOperationRunning = ref<boolean>(false) // 在 fileListStore 中
```

该状态控制工具栏、所有对话框、全部快捷键，属于全局 UI 互斥状态。放在 `fileListStore` 导致 `ToolBar.vue` 和所有 `dialogs/` 组件都必须 import `fileListStore`。

**建议**：移至 `composables/useOperationGuard.ts`：
```ts
const isRunning = ref(false)
export function useOperationGuard() {
  function start() { isRunning.value = true }
  function finish() { isRunning.value = false }
  return { isRunning, start, finish }
}
```

**严重度**：轻微（不影响正确性，但架构不干净）。

### 4.4 前端测试策略仍缺具体模板

**位置**：Vue 规范 §10

与 Rust 规范 §08 相比，Vue 规范 §10 缺少：

| 缺失项 | 影响 |
|--------|------|
| Tauri invoke mock 示例 | 测试 services/svn.ts 时无法隔离后端 |
| Store 测试模板 | 测试 fileListStore 时无从参考 |
| 组件渲染测试模板 | 测试 FileListTable 时不知怎么写 mount + assert |

**建议**：每个 Store 完成编码后及时补测试，避免一次性补。参考 Rust 规范 §08 的模板风格补充至少一个 Store 测试示例和一个 Tauri invoke mock 示例。

**严重度**：轻微。

### 4.5 TypeScript 错误码枚举示例缺失

**位置**：Rust 规范 §03

规范要求"前端 types/error-codes.ts 中定义与后端完全相同的错误码常量集合"，但仅在 Rust 规范中给出了 Rust 枚举代码，未给 TypeScript 对应代码。

**建议补充**：
```ts
// types/error-codes.ts
export const ErrorCodes = {
  SVN_EXEC_FAILED: 'SVN_EXEC_FAILED',
  SVN_PARSE_FAILED: 'SVN_PARSE_FAILED',
  // ... 11 个完整
} as const
export type ErrorCode = keyof typeof ErrorCodes
```

**严重度**：轻微。

## 六、总体结论

### 裁决：**可以进入 `doc/plan/` 阶段**

首轮 v1 报告的 5 项 P0、6 项 P1、3 项 P2 已全部合理修复。开发规范的覆盖度、一致性和编码就绪度已达到 Plan 阶段的门槛标准。

### 编码时建议关注的 5 点

1. **`cancel.rs` 示例代码**：`"已取消"` 改为错误码 `SVN_CANCELLED`，避免照搬示例导致违反规范
2. **`check_network` 端口**：增加协议解析，支持 HTTPS/HTTP/svn+ssh
3. **`isOperationRunning` 归属**：实现时放在 `useOperationGuard` composable 而非 `fileListStore`
4. **前端测试**：每完成一个 Store/Service 及时补测，勿积压
5. **错误码枚举**：编码 `types/error-codes.ts` 时直接对照 Rust 规范 §03 的目录表
