## 二、对设计师"归属 Plan 阶段"意见的独立判断

设计师认为以下 6 项属于 Plan 阶段而非 Design 阶段。以下逐个做独立判断：

### 1. Pinia Store 接口 → Plan 阶段

**设计师观点：** Store 的 actions/getters 属于"怎么做"，应归属 Plan。

**独立判断：✅ 正确**

- 设计文档已列出 4 个 store 的名称和职责（`workspace`、`fileList`、`settings`、`svn`）
- 这已在 Design 层面完成了"需要哪些状态管理模块"的定义
- 各 store 内部的 state 变量、actions 方法、getters 是编码细节，Plan 阶段定义即可
- 无争议

### 2. 核心类型定义 → Plan 阶段

**设计师观点：** 编码时自然产生。

**独立判断：✅ 正确（含一条提醒）**

- Design 文档不需要列出完整的 TypeScript 接口字段
- 但 `FileItem` 的结构（path、status、propsChanged 等）决定了文件列表的渲染和筛选逻辑
- **提醒：Plan 的第一步应定义核心类型**，否则步骤 1 定义的类型可能和步骤 5 的预期不一致，导致返工。但这一步确实在 Plan 中完成即可

### 3. Tauri Event 协议 → Plan 阶段

**设计师观点：** Rust 后端与前端通信细节。

**独立判断：✅ 正确**

- 设计文档已声明"长操作通过 Tauri event 推送进度到前端"（§6.2）
- 具体的事件名和 payload 结构是纯实现细节
- Plan 阶段定义时只需前后端约定一致

### 4. XML 解析结构 → Plan 阶段

**设计师观点：** Rust 内部实现细节。

**独立判断：✅ 正确**

- `svn status --xml` 的输出结构由 SVN 自身定义
- 文档已声明"通过 serde 解析 XML 输出"（§6.2），足矣
- Plan 中针对每个 command 定义对应的 Rust struct

### 5. "最近使用的日志"存储 → Plan 阶段

**设计师观点：** 提交面板的小功能。

**独立判断：✅ 正确**

- 存储方案（tauri-plugin-store 存最近 N 条 vs Pinia 运行时内存存）可由 Plan 决定
- 设计层面只需知道"有这个功能"

### 6. Services 拆分策略 → Plan 阶段

**设计师观点：** 纯实现决策。

**独立判断：⚠️ 原则正确，但有隐含硬约束**

- 从职责划分上，Services 拆分确实是实现决策
- **但有一个硬约束：** Vue 规范（`doc/rule/01-Vue前端开发规范.md §07`）规定 services 文件超过 200 行需拆分。19 个 invoke 方法 + 类型转换逻辑，单文件必然远超 200 行
- 如果 Plan 忽略此问题，AI 会写出一个超长文件，然后被 lint 检测触发拆分，浪费一轮编码
- **结论：** 设计师正确（Design 不需要处理），但 Plan 必须在第一步就决定拆分策略，不能"编码时自然发生"

### 汇总

| 序号 | 项目 | 判断 | 备注 |
|------|------|------|------|
| 1 | Pinia Store 接口 | ✅ Plan | 无争议 |
| 2 | 核心类型定义 | ✅ Plan | 提醒：Plan 第一步需定义 shape |
| 3 | Tauri Event 协议 | ✅ Plan | 无争议 |
| 4 | XML 解析结构 | ✅ Plan | 无争议 |
| 5 | 最近日志存储 | ✅ Plan | 无争议 |
| 6 | Services 拆分 | ✅ Plan（含硬约束） | Plan 必须处理，否则违反规范 |

**总判断：设计师 6 项判断中 5 项完全正确，1 项原则正确但有隐含约束（Plan 必须处理 Services 拆分，不能留到编码时自然发生）。**

首次评估（02 号报告）将这 6 项列为 Design 缺口是**判断过于严格**，本次评估予以正式修正。

## 五、最终结论

**评级：A-**

**设计文档已达到生成 AI 编程执行步骤（Plan）的条件。**

判断依据：
1. Design 层必须定义的 16 项内容已全部覆盖
2. 设计师关于 6 项归属 Plan 的意见经独立判断 **5 项完全正确、1 项原则正确**
3. 剩余细节性工作（类型 shape / Store 接口 / Event 协议等）合理归属 Plan 阶段
4. 项目从零启动，Plan 需完整覆盖前后端所有文件

**对 Plan 编写者的建议：**

1. **分阶段实施** — 按 MVP 优先级分步，不要一次性实现全部 19 个操作
2. **步骤 1 先定义类型** — TypeScript 核心类型和 Rust 结构体先写好，避免跨步骤不一致
3. **明确 Services 拆分策略** — 在 Plan 中决定 `services/svn.ts` 是否拆分为目录结构
4. **约定 Event 协议** — 在 Plan 中定义前后端共用的 Tauri event 名称和 payload

**对首次评估的反省：** 02 号报告中，本人将这 6 项归为 Design 缺口是判断过于严格，未能准确把握 Design 与 Plan 的边界划分。本次评估予以修正和澄清。
