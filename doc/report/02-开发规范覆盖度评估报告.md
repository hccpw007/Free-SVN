## 一、评估概述

本次评估的核心问题是：**当前两份开发规范是否足以支撑工程团队（或 AI 编码）将设计文档转换为可工作的代码？**

相比上一份架构评估报告（关注技术选型与架构合理性），本报告聚焦于 **规范对设计的覆盖度**——即业务设计 v4 和视觉设计 v4 中的每一处要求，是否都能在开发规范中找到对应的编码指导。

评估方法：逐节遍历业务功能设计（15 节）和交互视觉设计（8 章），对照开发规范检查覆盖率。

## 三、关键缺项分析（按严重程度排序）

### P0 — 必须补充，否则无法编码

#### 缺项 1：Tauri Event 协议未定义（业务 §8.3 / Rust 规范）

**描述**：长操作（checkout/update/merge/export）需要通过 Tauri event 推送进度到前端。目前无任何文档定义：

- 事件名称（如 `progress:update`、`progress:done`、`operation:completed`）
- 事件 payload 结构（进度百分比 / 阶段文字 / 文件计数 / 错误信息）
- 事件发射时机（开始/每 N 个文件/完成/失败/取消）

**影响**：前后端各自开发必然导致事件协议不匹配，集成时需返工。这是 IPC 中最容易出 bug 的环节。

**要求**：在 Rust 规范第 02 章 Command 中增加 Tauri Event 协议表，或在业务设计第 6 章中补充。

#### 缺项 2：错误码目录缺失（业务 §6.2）

**描述**：业务设计和 Rust 规范均强调"后端返回错误码 key，前端根据 locale 翻译"，但**没有任何文档列出全部错误码**。

当前散落在各处的错误码：

| 来源 | 错误码 |
|------|--------|
| Rust 规范 AppError | `SVN_EXEC_FAILED / SVN_PARSE_FAILED / INVALID_INPUT / IO_ERROR / REPO_ERROR / SVN_TIMEOUT / SVN_NOT_FOUND / SVN_NOT_WORKING_COPY / SVN_OP_IN_PROGRESS / SVN_CANCELLED / NETWORK_UNREACHABLE` |
| 业务设计 SvnError | `SVN_EXEC_FAILED / SVN_PARSE_FAILED / SVN_TIMEOUT / SVN_NOT_FOUND / SVN_NOT_WORKING_COPY / SVN_OPERATION_IN_PROGRESS / SVN_CANCELLED` |

两套枚举有差异（`IO_ERROR` vs 缺失、`REPO_ERROR` vs 缺失、命名不一致 `SVN_OP_IN_PROGRESS` vs `SVN_OPERATION_IN_PROGRESS`）。

**影响**：编码时前端无法预先知道需要翻译哪些 error code，后端和前端会各自定义一套，最终不匹配。

**要求**：建立一处单一错误码目录（建议放在 `types/error-codes.ts` 和 `models/error.rs` 同步维护），并在开发规范中列出完整表格。

#### 缺项 3：前端 Store 状态规范缺失（视觉 §3.2 / §3.4 / §4.5）

**描述**：视觉设计定义了大量交互状态（加载/空/错误/冲突/操作进行中），但 Vue 规范没有定义 **每个 Store 的状态结构**。关键缺失：

- **全局操作状态**：`isOperationRunning`（ProgressOverlay 展示期间的全局禁用）
- **网络状态**：`isOffline`（服务端操作按钮禁用）
- **选择状态传递**：主视图选择 → 提交面板继承 → 提交完成清除
- **自动刷新时机**：window focus / 提交后 / 切换工作副本后

**影响**：每个组件自行实现状态管理会导致实现不一致。

**要求**：在 Vue 规范第 05 章状态管理中，为每个 Store 定义最小状态接口（类似 Rust 的结构体定义）。

#### 缺项 4：测试策略形同虚设（Rust 规范 §08 / 业务 §10）

**描述**：当前测试规范仅一句话"单元测试写在模块末尾"。面对业务设计第 10 章中的具体性能要求（文件列表加载 ≤1 秒、差异渲染 ≤2 秒、10000+ 文件虚拟滚动），开发规范完全没有提供测试方法。

关键缺失：
- SVN 集成测试方法（临时仓库、svnadmin create）
- Tauri Command mock 测试
- 性能基准测试
- 错误场景注入测试

**影响**：无法验证非功能需求是否满足。无测试 = 无法重构。

**要求**：在 Rust 规范第 08 章测试策略中补充集成测试模板代码；在 Vue 规范第 10 章测试策略中补充性能测试方法。

#### 缺项 5：Composables 组合式函数缺失（Vue 规范）

**描述**：Vue 规范定义了 `pages/`、`components/`、`stores/`、`services/`、`utils/` 五层，但缺少 Vue 3 Composition API 的核心模式——**组合式函数（composables）**。

本项目中需要 composables 的场景：

| 场景 | 说明 | 不适合放在 |
|------|------|-----------|
| `useKeyboardShortcuts` | 全局快捷键注册与冲突管理 | stores（无全局状态）|
| `useFileSelection` | Shift+Click / Ctrl+Click 多选逻辑 | utils（有状态）|
| `useAutoRefresh` | window focus / 定时刷新 | stores（非共享状态）|
| `useOperationGuard` | ProgressOverlay 期间的全局禁用 | stores（非共享状态）|

**要求**：在 Vue 规范第 07 章文件拆分时机的 composables 拆分阈值中加入此目录。

### P1 — 重要，建议编码前补充

#### 缺项 6：IPC 接口契约未形式化（业务 §8.3）

**描述**：前后端通过 20+ 个 Tauri Command 通信，每个 command 有结构化的参数和返回值。目前没有一个**集中的 IPC 接口契约文件**。

风险：Rust 侧 `commands/status.rs` 定义 `StatusParams { path: String }`，前端 `services/svn.ts` 也要定义相同结构。同一份结构维护两次，改了一侧另一侧不同步时编译不报错。

**建议**：在 Rust 侧定义参数结构体后，自动或手动同步到 TypeScript 类型定义。至少在开发规范中增加一条"IPC 参数/返回类型必须在 Rust 和 TypeScript 两处同步定义"的规则。

#### 缺项 7：外部工具执行模式未定义（业务 §5.7 / §5.11）

**描述**：业务设计定义了外部差异/合并工具（VS Code、Beyond Compare 等），但 Rust 规范只有 `svn::executor` 用于执行 SVN 命令。外部工具执行是另一条路径：

```rust
// 在 Rust 规范中缺少的模式
pub fn run_external_tool(tool: &str, args: &[&str]) -> Result<(), AppError> {
    // 与 svn::executor::run_svn 类似但不同：
    // 1. 不设置超时（用户可能长时间编辑）
    // 2. 输出是工具窗口而非 stdout
    // 3. 需要返回工具进程的退出码
}
```

**建议**：在 Rust 规范第 04 章中增加 `external/` 模块或 `tools/` 模块，封装外部工具调用模式。

#### 缺项 8：网络检测的前后端协同未定义（业务 §10）

**描述**：业务设计第 10 章规定"服务端操作前检测网络可达性"，但：
- Rust 规范第 04 章有 `check_network` 函数签名但没有调用时机
- 前端如何响应网络不可达（按钮禁用 + tooltip "网络不可用"）未定义
- status 是本地操作，不受网络限制——此判断逻辑在哪里实现？

**建议**：在 Vue 规范中定义网络状态 store 或 composable；在 Rust 规范中明确每个 Command 是否调用 `check_network`。

#### 缺项 9：对话框互斥规则未定义（业务 §5）

**描述**：业务设计定义了 9 种对话框，但没说明哪些对话框可以同时出现、哪些互斥：
- 检出/切换/合并向导是否可以同时打开？
- ProgressOverlay 展示期间能否打开新的对话框？
- 多个 ConfirmDialog 能否叠加？

**建议**：在 Vue 规范第 04.5 章或业务设计中增加对话框互斥矩阵表。

#### 缺项 10：键盘快捷键冲突处理未定义（视觉 §4.4）

**描述**：11 个快捷键在不同页面和对话框中的作用域不同：
- `Ctrl+W` 在 Diff 面板 = 关闭面板，在对话框 = 关闭对话框，在提交面板 = ？
- `Esc` = 关闭对话框/面板，但如果同时打开了面板和对话框？
- `Ctrl+D` = 打开差异，但如果 Diff 面板已打开？

**建议**：定义快捷键优先级规则：对话框 > 面板 > 全局。`Esc` 先关闭顶层元素。

#### 缺项 11：二进制工具管理规范缺失（业务 §2.3）

**描述**：内置 svn 的路径获取、版本检测、许可证合规在业务设计 §2.3 中有详细说明，但 Rust 规范中没有对应章节管理"外部二进制工具"：
- `get_svn_path()` 返回值在哪里定义？
- 找不到内置 svn 时的诊断和提示策略？
- 未来可能增加其他二进制工具（如 git、diff 工具）的扩展策略？

**建议**：在 Rust 规范中增加工具管理规范，统一管理所有捆绑的二进制文件。

### P2 — 优化建议，可在编码中逐步完善

#### 缺项 12：虚拟滚动规范缺失（Vue 规范）

**描述**：业务设计第 10 章要求 10000+ 文件使用虚拟滚动，但 Vue 规范没有提及。
- 文件列表使用 `<el-table>` 的虚拟滚动还是自定义实现？
- 日志页面的虚拟滚动同样问题
- 虚拟行高、缓冲区大小等参数约定

**建议**：在 Vue 规范第 01.2 章组件使用约定中补充虚拟滚动场景和组件选择。

#### 缺项 13：日志搜索能力边界未在前端约束（视觉 §3.5）

**描述**：`svn log --search` 仅支持搜索提交信息，不支持按作者或版本号过滤。视觉设计中的 placeholder="按提交信息搜索..." 已体现此约束，但 Vue 规范没有要求后端屏蔽不支持的操作，前端搜索 UI 也未定义如何告知用户此边界。

**建议**：增加 svn CLI 能力边界表，让前端 UI 提前响应用户输入的无效过滤条件。

#### 缺项 14：自动刷新策略未在 Pinia 规范中定义

**描述**：多处提到"窗口 focus 自动刷新"、"提交后自动刷新"、"切换工作副本后刷新"，但刷新由谁触发、刷新与当前选中状态的关系未定义。
- 自动刷新是否保持文件选中状态？
- 自动刷新期间用户正在进行的操作（如编辑提交日志）是否保留？
- 刷新失败时的回退策略？

**建议**：在 Vue 规范第 05 章状态管理中定义 Store 的 refresh() 约定。

#### 缺项 15：Store 重置策略未定义

- 切换工作副本时哪些 Store 需要重置？（workspace、fileList → 重置；settings → 保留）
- 应用退出时哪些需要持久化？

**建议**：在 Vue 规范第 05 章中定义 `reset()` 约定和调用时机。

#### 缺项 16：版本兼容性策略缺失

- store 配置文件的跨版本兼容性（当前规范只说"不存在的字段用默认值"）
- Tauri event 协议的版本兼容
- 内置 svn 版本变更对命令输出的影响

**建议**：简单版本策略——Store Schema 加版本号；Event payload 加 version 字段。

### 5.1 六维评分

### 5.2 代码生成就绪度评估

就目前规范而言，AI 编码面临以下障碍：

1. **🟢 可直接编码**（无需追问）：组件结构、路由配置、设计 Token、Tailwind 样式、SVN 命令骨架
2. **🟡 需补充信息**（需追问）：Store 状态接口、IPC 参数类型、对话框互斥规则
3. **🔴 无法编码**（需先补充规范）：Tauri Event 协议定义、错误码目录、网络检测交互流程、外部工具执行模式、集成测试模板

**估计**：直接编码可完成约 60% 的代码，其余 40% 需要补充决策或返工。

## 七、总结

两份开发规范的质量整体不错，组件结构、路由、设计系统、SVN 命令等核心部分与业务/视觉设计高度匹配，可以直接指导编码。

但存在几处 **关键缺口** 导致无法完全依赖规范进行编码：

1. **Tauri Event 协议** 未定义——这是前后端通信的"暗线"，长操作进度推送和取消反馈完全依赖它
2. **错误码目录** 缺失——后端错误码和前端的 i18n 翻译之间没有连接点
3. **Store 状态结构** 未定义——视觉设计中的交互状态（全局禁用、网络状态、选择传递等）没有对应的代码结构
4. **测试策略** 极度匮乏——非功能需求的可验证性为零
5. **Composables 目录** 缺失——Vue 3 的核心能力未被利用

**一句话结论**：规范覆盖了"静态结构"（组件树、文件结构、路由），但缺少"动态协议"（事件、错误码、状态接口、测试）。后者的缺失是阻止从设计到代码的最后 20% 的障碍。

建议在进入 `doc/plan/` 编写阶段前，优先补齐上述 P0 项。否则 Plan 步骤中每步都会遇到"此处如何与后端交互？"的追问，打断编码节奏。
