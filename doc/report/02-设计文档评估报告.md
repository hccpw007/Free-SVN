## 二、对设计师意见的独立评估

设计师认为以下 6 项属于 Plan 阶段而非 Design 阶段。逐项独立判断：

### 1. Pinia Store 接口 → Plan 阶段

**设计师观点：** Store 的 actions/getters 属于"怎么做"，应归属 Plan。

**独立判断：✅ 正确**

- 设计文档已列出 4 个 store 的名称和职责（`workspace`、`fileList`、`settings`、`svn`）
- 这已在 Design 层面定义了"需要哪些状态管理模块"
- 具体有哪些 state 变量、actions 方法、如何调用，是 Plan 的编码细节
- 没有争议

### 2. 核心类型定义 → Plan 阶段

**设计师观点：** 编码时自然产生。

**独立判断：✅ 正确，但有一项提醒**

- Design 文档不需要定义完整的 TypeScript 接口
- 但 `FileItem` 的结构（path、status、propsChanged 等字段）决定了文件列表如何渲染、筛选、搜索
- 如果 Plan 不写出核心类型的 shape，AI 在步骤 1 定义的类型可能与步骤 5 的预期不一致
- **建议**：Plan 的第一步就定义好核心类型（这在 Plan 中完成即可，不属于 Design）

### 3. Tauri Event 协议 → Plan 阶段

**设计师观点：** Rust 后端与前端通信细节，Plan 阶段定义。

**独立判断：✅ 正确**

- 设计文档已声明"长操作通过 Tauri event 推送进度到前端"（§6.2）
- 协议名、payload 结构是纯实现细节
- Plan 阶段定义 event 接口时，只需保证前后端约定一致

### 4. XML 解析结构 → Plan 阶段

**设计师观点：** Rust 内部实现细节。

**独立判断：✅ 正确**

- `svn status --xml` 的输出结构是 SVN 自身格式
- Plan 会针对每个 command 定义对应的 Rust struct
- 文档中已声明"通过 serde 解析 XML 输出"（§6.2），足够了

### 5. "最近使用的日志"存储 → Plan 阶段

**设计师观点：** 提交面板的一个小功能。

**独立判断：✅ 正确**

- 存储方案（tauri-plugin-store 存最近 N 条 vs Pinia 运行时内存存）是 Plan 可以决定的
- 设计层面只需要知道"有这个功能"，已满足

### 6. Services 拆分策略 → Plan 阶段

**设计师观点：** 纯实现决策。

**独立判断：⚠️ 原则正确，但 Plan 必须处理此问题**

- 从职责划分上，Services 拆分确实是实现决策
- **但有一个硬约束：** Vue 规范（doc/rule/ §07）规定 services 文件超过 200 行需拆分。19 个 invoke 方法 + 类型转换逻辑，单个文件必然远超 200 行
- 如果 Plan 忽略这个问题，AI 会写出一个超长文件，然后被 knip/lint 检测触发拆分，浪费一轮编码
- **结论：** Designer 正确（Design 不需要处理），但 Plan 编写者需在第一步就决定拆分策略，不能留到编码时自然发生

### 设计师意见汇总

| 序号 | 项目 | 判断 | 备注 |
|------|------|------|------|
| 1 | Pinia Store 接口 | ✅ Plan | 无争议 |
| 2 | 核心类型定义 | ✅ Plan | 提醒：Plan 需尽早定义，避免跨步骤不一致 |
| 3 | Tauri Event 协议 | ✅ Plan | 无争议 |
| 4 | XML 解析结构 | ✅ Plan | 无争议 |
| 5 | 最近日志存储 | ✅ Plan | 无争议 |
| 6 | Services 拆分 | ⚠️ Plan（硬约束） | 必须处理，否则违反文件拆分规范 |

**总判断：设计师 6 项判断中 5 项完全正确，1 项原则正确但有隐含约束（Plan 必须处理 Services 拆分）。v1 报告将它们列为 Design 缺口是过于严格了，正式撤回这 6 条。**
