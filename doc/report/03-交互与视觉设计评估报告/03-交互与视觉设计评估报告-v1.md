# 03-交互与视觉设计评估报告

> 评估对象：`doc/design/02-交互与视觉设计/02-交互与视觉设计-v2.md`
> 对照基准：`doc/design/01-业务功能设计/01-业务功能设计-v2.md`、`doc/rule/01-Vue前端开发规范.md`、`doc/rule/02-Rust后端开发规范.md`
> 评估视角：业务完整性、用户体验、AI 可编程性

---

## 1. 总体评价

交互与视觉设计 v2 文档在纯 UI/UX 设计层面质量很高——色彩体系、字体排版、间距系统、暗色模式等设计系统定义完整，页面线框图细致到每个交互状态和颜色值，Element Plus 组件使用约定明确。

但作为 AI 编程的输入文档，与业务功能设计文档之间存在 **多处架构分歧和功能遗漏**，必须在进入 Plan 阶段前统一。

**综合评级：B+**（良好，明确可改进）

---

## 2. 严重问题

以下问题必须修复，否则 AI 编码会出现方向性错误。

### 2.1 CSS 变量实现方式与开发规范直接冲突

**问题**：视觉设计文档 §5.2 给出的 CSS 变量双主题方案：

```css
:root { --bg-primary: #F8FAFC; }
.dark { --bg-primary: #0F172A; }
```

与 `01-Vue前端开发规范.md` 第 04 节 "核心规则" 冲突——规范明确规定 **仅使用 Tailwind CSS，禁止 `<style>` 标签或任何自定义 CSS 文件**。

CSS 变量要么写在 `<style>` 标签中，要么写在 `.css` 文件中——两者都被禁止。Tailwind CSS 4 的 `@theme` 指令虽然可以在 `style.css` 中定义 Token，但 `.dark .card { @apply bg-slate-800; }` 这类写法的确需要自定义 CSS。

> **影响范围**：全部页面的主题色、卡片色、输入框背景色等约 30 处使用了 CSS 变量的元素。
>
> **修复建议**：二选一——
> - 方案 A：仅使用 Tailwind `dark:` 变体（如 `class="dark:bg-slate-800"`），完全不用 CSS 变量
> - 方案 B：放宽规范，允许在 `src/style.css` 中通过 Tailwind `@theme` / `@layer` 定义设计 Token，仍禁止独立 `<style>` 标签

### 2.2 页面/路由架构与业务设计不一致

| 项目 | 业务设计 | 视觉设计 | 分歧说明 |
|------|---------|---------|---------|
| 欢迎页 vs 主视图 | `HomePage.vue` 负责"空状态条件渲染：有工作副本→文件列表，无→欢迎页" | `HomePage.vue`（欢迎页）+ `WorkspacePage.vue`（文件列表）两个独立页面 | **架构分歧** |
| Diff 视图 | `DiffPage.vue` 路由 `/workspace/diff?file=xxx` | 仅 `DiffViewer.vue` 组件（底部面板），无独立路由 | **路由缺失** |
| 冲突解决 | `ResolvePage.vue` 路由 `/workspace/resolve?file=xxx` | `ResolveDialog.vue` 对话框组件 | **形态不同** |

三个差异意味着 AI 按视觉设计编码后，文件结构和路由会与业务设计的路由定义矛盾。**必须先在两个设计文档之间统一**。

### 2.3 组件命名与业务设计多处不一致

| 业务设计（§9.1） | 视觉设计（§7） | 问题 |
|-----------------|---------------|------|
| `pages/HomePage.vue`（主视图） | `pages/WorkspacePage.vue`（主视图） | 名称冲突：HomePage 在业务设计中是主视图，在视觉设计中是欢迎页 |
| `components/svn/CommitDialog.vue` | `components/CommitForm.vue` | 名称不同 |
| `components/svn/ResolveConflict.vue` | `components/dialogs/ResolveDialog.vue` | 名称和目录层级不同 |

AI 编程严格依赖 `doc/plan/` 中的精确文件路径，这些命名分歧会导致 Plan 步骤中写哪个路径无法确定。

### 2.4 缺少两个业务设计中定义的对话框

| 业务设计 § | 对话框 | 缺失的输入字段 |
|-----------|--------|--------------|
| §5.12 | **导出对话框** | 工作副本路径（只读）、导出到（路径+浏览）、版本号（文本输入，可选）、仅导出未版本控制的文件（复选框） |
| §5.13 | **更新到版本对话框** | 当前版本（只读）、目标版本（文本输入）、检出深度（下拉选择） |

这两个对话框有明确的输入字段定义和交互需求，但视觉设计完全没有覆盖。AI 无从获知它们的视觉形态。

## 3. 中等问题

虽不影响编码启动，但影响交付质量。

### 3.1 设置按钮位置与业务设计不一致

| 文档 | ⚙️ 设置位置 |
|------|------------|
| 业务设计 §4.3 | **工具栏**中（与提交/更新/日志等并列），"始终可用" |
| 视觉设计 §2.1 | **TopBar 右侧**，与暗色切换并列 |

放 TopBar（始终可见区域）是更合理的 UX 决策，但这是 **设计决策变更**，应在视觉设计中明确标注"此位置已在 v2 中优化"。

### 3.2 Diff 面板高度参数未定义

面板从底部滑入的动画（200ms ease-in-out）已定义，但缺少：
- 默认展开高度 / 主列表与 diff 面板的分配比例（如 6:4？）
- 面板是否可拖拽调整高度
- 最小高度/最大高度约束

### 3.3 合并向导仅画了第一步

业务设计 §5.10 定义了 4 个合并步骤，视觉设计 §3.7 仅画了"步骤 1/3：选择来源"的线框图。步骤 2-4 的视觉形态缺失。

| 步骤 | 内容 | 视觉覆盖 |
|------|------|---------|
| 1 | 选择源 URL | ✅ 有线框图 |
| 2 | 选择版本范围 | ❌ 缺失 |
| 3 | 选择合并选项（忽略祖先、记录合并信息等） | ❌ 缺失 |
| 4 | 执行合并并显示结果（冲突文件列表） | ❌ 缺失 |

### 3.4 日志搜索功能缺少能力边界提示

业务设计 §5.5 明确说明 `svn log --search` **仅支持搜索提交信息，不支持按作者或版本号过滤**。

视觉设计的日志页面只有一个通用搜索框，用户会自然尝试输入作者名或版本号。缺少对搜索能力边界的 UI 提示（如搜索框 placeholder="搜索提交信息..." 或搜索模式下拉）。

### 3.5 批量还原按钮显示条件被简化

业务设计 §4.4 定义了批量还原按钮的显示条件：**"选中 ≥ 2 个还原类文件时显示"**。视觉设计 §3.2 仅画了"批量还原"按钮，未说明显示/隐藏逻辑。

### 3.6 提交面板选择状态传递规则未体现

业务设计 §5.4 定义的关键行为：
1. 进入提交面板时继承主视图选中状态
2. 提交面板内修改选择不影响主视图的选中状态
3. 提交完成后清除两个视图的选中状态

视觉设计仅展示布局，未体现这些状态传递约束。

不影响编码启动，但应知悉。

### 4.1 最近打开列表空状态

- 存储多少条历史记录？

## 5. AI 可编程性评估

### 5.1 各维度评分

| 维度 | 评分 | 依据 |
|------|------|------|
| 设计系统完整性 | A | 色彩、字体、间距、圆角、阴影、图标、动画完整定义 |
| 页面线框图详细度 | A- | ASCII 图细到每个状态行和颜色值；合并向导步骤 2-4 缺失 |
| 交互状态覆盖 | A | hover / disabled / selected / loading / empty 均有覆盖 |
| 与业务设计一致性 | C | 4 处架构分歧 §2.2、5 处命名分歧 §2.3、2 处功能遗漏 §2.4 |
| 技术可实现性 | B- | CSS 变量与 Tailwind-only 规范的矛盾未解决 |
| 边界/异常状态覆盖 | B | 存在加载/空/错误状态，但工作副本路径不存在等边界未覆盖 |

### 5.2 AI 编码关键风险

1. **编码规范冲突**：AI 读到视觉设计中的 CSS 变量方案 → 再读到 Vue 规范禁止 `<style>` → 两条指令都能触发 AI 执行。必须以"规范优先"或"设计优先"固定其中之一。
2. **文件结构二选一**：AI 无法同时遵循业务设计的 §9.1 文件结构和视觉设计的 §7 组件树——存在 5 处命名差异 3 处架构差异。Plan 阶段必须选定一套。
3. **遗漏功能的编码风险**：导出对话框和更新到版本对话框在视觉设计中缺失，AI 按视觉设计生成的 Plan 不会包含这两个组件。

### 5.3 执行步骤可贯彻性

视觉设计在 **样式层面** 足够细致（颜色色值/字号/间距/动画参数均有数值），因此 Plan 中的样式描述可以直接引用视觉设计的 Token，无需 AI 自行判断。

但在 **行为逻辑层面**，状态传递规则（提交选择状态）、条件显示逻辑（批量还原按钮）、功能边界提示（日志搜索）等信息的缺失，需要 Plan 步骤补充细节。否则 AI 要么写错、要么做出与业务设计不一致的实现。
