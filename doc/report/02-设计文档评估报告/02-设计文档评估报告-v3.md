### 1.2 设计师提出不同意见的项

**#1: 文件命名与 Vue 规范不一致（workspace.ts 等）**

> 设计师反馈：Pinia store 文件在 Vue/TS 生态中普遍使用 camelCase，kebab-case 主要适用于组件目录和 .vue 文件。属于规范未明确的地方，Plan 阶段按实际偏好调整即可。

**独立判断：✅ 设计师正确**

- Pinia store 文件名使用 camelCase（如 `workspace.ts`、`userStore.ts`）是 Vue/TS 生态的通行做法
- Vue 规范中"TS 文件使用 kebab-case"的描述过于笼统，未区分 store/service/util 等非组件 TS 文件
- 这属于规范文档的表述精确度问题，不是设计文档的缺口
- 设计文档使用 `workspace.ts` 等命名是合理的，无需修改

**#2: 色彩体系、暗色模式、按钮层级等视觉问题**

> 设计师反馈：属于 UI 设计规范范畴，应该在独立的设计稿或 UI 规范中定义，与功能设计文档正交。

**独立判断：✅ 设计师正确**

- 功能设计文档的职责是定义"做什么"（功能、数据流、交互逻辑）
- 视觉规范（主色、暗色模式、按钮样式、图标风格）属于 UI 设计系统，是两个正交的维度
- 在 Claude Code 编程场景下，这些由 Tailwind CSS + Element Plus 的默认主题覆盖，不影响 Plan 生成
- v2 报告将 V-1~V-5 列入设计文档缺口是判断过度

### 1.3 小结

v2 报告列出的所有 Design 层缺口，设计师已全部处理或合理解释。**当前设计文档已没有任何待处理的 Design 层问题。**
