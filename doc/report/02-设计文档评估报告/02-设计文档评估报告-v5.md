## 二、运行时层盲点

以下问题发生在应用运行时，会直接导致崩溃或数据损坏，但设计文档完全没有涉及。

### 2.1 并发操作防护完全缺失

**设计文档现状**：§6.2 只定义了单个 svn 命令的执行（超时 60 秒 + 异步），**完全没有提及当多个操作同时发生时怎么做**。

| 场景 | 风险等级 | 后果 |
|------|---------|------|
| 提交进行中，用户点击更新 | 🔴 **严重** | 两个 svn 进程同时写 `.svn/` 元数据 → 工作副本损坏 |
| 更新进行中，用户点击切换分支 | 🔴 **严重** | `svn update` 未完成就执行 `svn switch` → 版本不一致 → 灾难 |
| 文件列表还在加载，用户双击文件查看 diff | 🟡 中等 | invoke 返回错乱或超时 |
| 检出进行中，用户再次点击检出到同一目录 | 🟡 中等 | 目录冲突或覆盖 |
| 用户快速连续点击刷新 | 🟢 轻微 | 重复请求浪费性能 |

> **缺少的内容**：没有定义操作队列、互斥锁（per-workspace lock）、按钮禁用态的并发规则。设计文档应该至少有一个 `并发控制策略` 章节说明：是否使用 Rust 的 `Mutex<Vec<Operation>>` 或每工作副本的 `AtomicBool`。

### 2.2 长操作取消机制未定义

**设计文档现状**：§10 要求"长操作显示实时进度"，但**没有定义用户如何取消正在进行的操作**。

| 操作 | 典型耗时 | 无取消的后果 |
|------|---------|------------|
| `svn checkout` 大仓库 | 数分钟 | 用户只能命令行 kill 进程或强关应用 |
| `svn update` 大量文件 | 30 秒+ | 无退路 |
| `svn export` | 大仓库 1-2 分钟 | 只能强关 |
| `svn merge` | 中等仓库 30 秒+ | 进度只能干等 |

> **缺少的内容**：没有定义 ProgressOverlay 上是否有关闭/取消按钮、取消后是否需要清理（`svn cleanup`）、取消时是否回滚已做的变更。Tauri 的 `AbortController` / Rust 的 `CancellationToken` 模式没有提及。

### 2.3 IPC 通信失败的降级未设计

**设计文档现状**：所有数据流图（§8.2）都假设 Tauri `invoke` 总是返回成功的结果。**没有考虑前端 invoke 失败的情况**。

| 失败场景 | 概率 | 后果（如果不处理） |
|---------|------|-----------------|
| Rust command panic | 低 | Promise rejected → 前端无处理则静默失败 |
| JSON 序列化/反序列化失败 | 低 | invoke 抛出异常 |
| Tauri 窗口关闭但后台操作还在进行 | 中 | 幽灵进程 |
| 前端调用 invoke 但 Rust 侧已卸载或崩溃 | 低 | MethodNotFound 错误 |

> **缺少的内容**：没有规定前端调用 `invoke()` 时的统一错误处理模式（全局 error handler vs 每个调用 try-catch），也没有定义 Tauri event 异常时的重试或降级行为。

### 2.4 持久化存储无损坏恢复策略

**设计文档现状**：§8.3 定义了使用 `tauri-plugin-store` 持久化 JSON，但**没有考虑文件损坏怎么办**。

| 场景 | 后果 |
|------|------|
| 用户直接编辑或误删了 store 文件 | settings 初始化失败 |
| 磁盘写入时断电导致 JSON 截断 | 解析失败，应用启动崩溃 |
| 跨版本升级后 store 格式变化 | 旧字段不存在导致逻辑错误 |

> **缺少的内容**：没有定义 store 读取失败的兜底策略（使用默认值 + 忽略损坏）。常见的健壮做法是 `try_deserialize`，如果失败则重置为默认并备份旧文件。

### 2.5 空提交/无效操作防护未定义

| 场景 | 问题 |
|------|------|
| 用户不选任何文件点击【提交】 | 按钮应禁用还是弹出提示？ |
| 用户提交信息为空 | SVN 允许空日志，但团队规范通常禁止 |
| 用户选择"全部使用我的版本"但没有冲突文件 | 按钮在无冲突时禁用？ |
| 用户在检出对话框点击【检出】但 URL 为空 | 无校验直接崩溃？ |

> **缺少的内容**：没有定义每个操作的"执行前前置条件检查"（precondition checks）。

## 三、系统集成层盲点

桌面应用不同于 Web 应用，需要与操作系统和用户环境深度集成。

### 3.1 系统托盘/后台运行未设计

**设计文档现状**：§3 定义了系统右键菜单，§5.7 设置了"开机自启动"，但 **"后台常驻"的具体行为完全没有定义**。

| 缺失的问题 | 说明 |
|-----------|------|
| 关闭窗口 = 退出还是最小化到托盘？ | 如果开机自启动，用户可能期望关闭窗口不退出 |
| 托盘图标是什么？ | 无设计 |
| 托盘右键菜单有哪些选项？ | 显示主窗口/检出.../关于/退出 → 都未定义 |
| 右键菜单触发的后台操作（更新/清理）在哪里反馈？ | 用户如果在托盘状态触发了更新，结果通知从哪里发 |
| macOS 的 Menu Bar extra | 是否需要？ |

> 这直接影响 §5.7 的"开机自启动"开关是否有意义——自启动但没有后台模式，启动后就打开全屏窗口，用户可能完全不需要。

### 3.2 多实例防护未定义

**设计文档现状**：没有提及是否允许用户同时打开多个 Free-SVN 实例。

| 场景 | 后果 |
|------|------|
| 用户从 Finder 双击两次应用图标 | 两个实例操作同一工作副本 → 损坏 |
| 系统右键菜单"SVN 更新" + 已打开的主窗口 | 后台进程操作时主窗口也在操作 |
| 打开一个实例，再从终端 `open /Applications/Free-SVN.app` 再开一个 | 没有单实例锁 |

> Tauri 2 通过 `tauri-plugin-single-instance` 插件支持单实例防护。设计文档应至少说明策略：允许单实例还是多实例？如果是单实例，第二个实例启动时应激活已有窗口还是忽略？如果已有窗口在操作中，第二个实例的右键菜单参数（--workspace）如何处理？

### 3.3 窗口状态持久化未定义

**设计文档现状**：§2.2 定义了窗口初始最小尺寸（800×500），但**没有定义每次打开时是否恢复上次关闭时的位置和大小**。

| 缺少 | 影响 |
|------|------|
| 窗口位置恢复 | 每次打开都在屏幕居中，用户需要手动拖回常用位置 |
| 窗口大小恢复 | 大屏幕上全屏操作后，下次打开全屏或小窗口？ |
| 最大化状态恢复 | macOS 上的绿色按钮状态不记录 |
| 多显示器 | 在副屏上操作后关闭，下次打开在主屏上 |

> 使用 `tauri-plugin-window-state` 即可解决，但设计文档未提及。

### 3.4 文件系统变更监听缺失

**设计文档现状**：§4.4 定义了"主窗口获得焦点时（window focus）自动刷新"。**但用户预期是实时同步外部修改，而不是推迟到窗口激活时**。

| 场景 | 问题 |
|------|------|
| 用户在终端执行 `svn update` | 需要切换回 Free-SVN 窗口激活后才能看到变化 |
| 其他 SVN GUI 工具修改了工作副本 | 同上 |
| 文件管理器删除了未版本控制的文件 | 不切窗口不会刷新 |

> 使用 Rust 的 `notify` crate 可以监听文件系统事件实时刷新。虽然不是 MVP 必需（窗口 focus 刷新已覆盖基本需求），但设计文档应注明这是已知局限。

## 四、平台适配层盲点

### 4.1 快捷键未考虑 macOS vs Windows/Linux 差异

**设计文档现状**：业务设计本身未列出快捷键（快捷键定义在视觉设计中）。视觉设计 §4.4 使用 `Ctrl+` 前缀，但未区分平台：

| 快捷键 | macOS | Windows/Linux | 问题 |
|--------|-------|---------------|------|
| `Ctrl+Enter` 提交 | macOS 用户期望 `Cmd+Enter` | ✅ | ❌ 未区分 |
| `Ctrl+D` 差异 | 同上 → `Cmd+D` | ✅ | ❌ 未区分 |
| `Ctrl+` 系列 | macOS 通常用 `Cmd+` | `Ctrl+` | ❌ 未区分 |

> Tauri 中应使用 `CmdOrCtrl` 修饰键，设计文档应注明此点。

### 4.2 平台特定窗口装饰无定义

- macOS：是否使用 `TitleBarStyle::Hidden` 实现自定义 TopBar？
- Windows：标题栏是否使用 Mica/亚克力效果？
- Linux：Wayland 下的窗口行为是否与 X11 一致？
- 窗口红绿灯按钮是否隐藏或自定义？

> 设计文档未涉及任何平台特定的窗口装饰策略。

### 4.3 内置 svn 的许可证合规未提及

**设计文档现状**：§2.3 定义了各平台 svn 的来源策略，但**完全没有提及这些二进制文件的许可证及合规要求**。

| 来源 | 许可证 | 合规风险 |
|------|--------|---------|
| Homebrew 编译的 svn | Apache 2.0 | ✅ 可分发 |
| SlikSVN | **专有许可**，需确认分发权 | ❌ **可能不允许再分发** |
| 官方 Subversion 二进制 | Apache 2.0 | ✅ 需在 NOTICE 中声明 |

> 捆绑第三方二进制文件涉及许可证合规和法律风险。如果选择了 SlikSVN（专有许可）或从非官方源获取，可能需要放弃该方案。文档应在 §2.3 中注明许可证来源和合规策略。

### 4.4 代码签名和 Gatekeeper 未提及

- macOS：内置的 svn 二进制如未签名可能被 Gatekeeper 拦截
- Windows：SmartScreen 对未签名的 exe 弹警告
- 第一次运行时 macOS 会询问权限

> 只在 v4 评估报告的 2.2 中提到过，但设计文档本身没有章节处理这些平台发布问题。

## 五、运维和可观测性层盲点

### 5.1 日志和诊断体系完全缺失

**设计文档现状**：§6.2 要求"执行前后必须记录日志"（Rust 规范也要求）。但**记录什么日志、存哪里、用户如何获取诊断信息完全没有定义**。

| 缺失项 | 说明 |
|--------|------|
| 日志存储位置 | 系统日志（`~/Library/Logs`）、用户数据目录还是 Tauri 默认日志？ |
| 日志轮转策略 | 无限增长？<100MB？保存 7 天？ |
| 用户可见的诊断入口 | 没有"帮助 → 导出日志"或"复制诊断信息"按钮 |
| 日志级别 | info/error/warn/debug 分别记录什么？ |
| SVN 原始输出是否记录 | 出问题时 svn stderr/stdout 是否包含在日志中？ |

> 没有日志体系，用户遇到问题只能截图，开发者无法复现。

### 5.2 性能退化监控未定义

§10 规定"变更文件列表加载 ≤ 1 秒（10000 文件）"，但：

- 如果工作副本有 50000 个文件怎么办？渐进式加载策略？
- 差异对比 ≤ 2 秒——diff 输出 10 万行时怎么做？
- 首次启动时大工作副本的首次 `svn status` 扫描是否要有"正在扫描..."初始状态？
- 长期运行内存泄漏监控？

### 5.3 应用更新机制未提及

- Tauri 支持 `tauri-plugin-updater` 实现自动更新
- 但设计文档完全没有提及：
  - 更新检查时机（启动时/定时/手动）
  - 更新通知 UI
  - 是否允许跳过版本
  - 是否显示更新日志
  - 差分更新 vs 全量下载

> §5.7 设置页的"关于"只显示版本号，没有"检查更新"按钮。

## 六、开发者体验和安全合规层盲点

### 6.1 错误信息本地化未定义

**设计文档现状**：§6.2 定义了中文错误消息（硬编码在 Rust 代码中）。但 §1.3 包含 `vue-i18n`。

- **前后端不一致**：前端 UI 用 i18n 多语言，后端 svn 错误消息硬编码中文。用户用英文界面时看到中文错误消息
- SVN CLI 本身的输出（英文）是否被直接展示给用户？文档要求"不抛原始 stderr"，但未翻译成用户界面语言
- 没有说明后端错误消息是否需要 i18n 化（通过错误 key 而非中文文字）

### 6.2 SVN 认证的 UX 盲点

§2.2 定义的认证策略技术上合理，但缺少 **用户首次配置的引导**：

- 用户只会看到"超时"错误提示，没有任何引导告诉用户"需要配置 SSH key 或设置 ~/.subversion/auth/"
- 没有设计帮助页面、引导弹窗或配置向导完成首次认证设置
- 超时和认证失败在 UI 上都表现为超时，用户无法区分是网络问题还是认证问题

### 6.3 多语言文案管理与 i18n 架构盲点

- 四个语言文件列出（zh-CN/en/ja/ko），但没有说明：
  - 回退语言（fallback locale）
  - 不完整翻译的处理策略
  - 动态占位符格式规范（`{0} 个文件`）
  - svn 状态标记（`M`/`✚`/`!`）是否在 i18n 范围内

### 6.4 测试策略盲点

Rust 规范 §08 要求测试，但设计文档自身缺少测试策略章节：

- **集成测试**：是否需要搭建 CI 内的 svn 测试服务器？
- **内置 svn 的测试路径获取**：测试环境如何找到内置 svn 二进制？
- **跨平台测试覆盖**：每个平台的 svn 行为是否一致？
- **前端测试**：如何 mock Tauri invoke？

### 6.5 辅助功能/无障碍盲点

设计文档完全没有涉及无障碍：
- Tab 键导航顺序
- 高对比度主题支持
- 焦点可见性
- 快捷键发现性（用户如何知道有哪些快捷键）

## 七、功能范围盲点

### 7.1 右键菜单与主窗口功能不对称

| 右键菜单（§3.1） | 主窗口（§4-5） | 问题 |
|-----------------|--------------|------|
| ✅ `更新到版本...` | ❌ **主窗口无此入口** | 主窗口的"更新"只更新到 HEAD，"更新到指定版本"没有入口 |
| ✅ `差异对比` | ❌ toolbar 无"差异"按钮（仅双击文件可开） | 主窗口工具栏缺少直接入口 |

> 主窗口缺少"更新到指定版本"的功能入口是一个功能断点。

### 7.2 svn 属性管理范围窄

§6.1 中 `property` 标记为"后续迭代"，但常用属性操作基本空缺：

- 缺少编辑已有 `svn:ignore` 的能力（当前只支持添加忽略）
- 缺少删除 `svn:ignore`（取消忽略）
- 缺少其他常用属性管理（`svn:eol-style`、`svn:mime-type`、`svn:keywords`、`svn:externals`）
- 缺少属性查看界面

### 7.3 搜索功能维度单一

| 搜索场景 | 是否支持 |
|---------|---------|
| 按文件名搜索文件 | ✅ |
| 按提交信息搜索日志 | ✅（有 svn CLI 限制） |
| 按作者过滤日志 | ❌ |
| 按版本号范围过滤日志 | ❌ |
| 按日期范围过滤日志 | ❌ |
| 在 diff 内容中搜索 | ❌ |
| 在文件内容中全文搜索 | ❌ |
| 跨条件组合搜索 | ❌ |

### 7.4 结果导出/共享缺失

| 场景 | 设计覆盖 |
|------|---------|
| 导出 diff 为文件 | ❌ |
| 复制文件路径到剪贴板 | ❌（仅 TopBar 路径可复制） |
| 复制版本号到剪贴板 | ❌ |
| 导出日志为文本/CSV | ❌ |
| 刷新/重算 diff | ❌ |

### 8.2 与 v4 评估的关系说明

v4 评估结论"评级 A，设计文档已完全满足生成 Plan 的条件"——这个结论 **在功能定义层面成立**。

但如果从 **生产级桌面应用** 的标准评估，设计文档缺失了约 **6 个关键维度**。这些不是 Plan 阶段可以 AI 自行填补的——它们是需要设计师或架构师决策的策略问题。

### 8.3 设计遗漏分类处理

## 十、总结

当前设计文档的 **功能定义层面** 足够完备（v4 结论 A 级依然有效）。但如果目标是 **生产级桌面应用**，需要在 Plan 之前补充约 6 个架构决策。建议将这些决策纳入 `doc/plan/` 的第 0 步（基础设施决策），而不是要求设计师修改业务设计文档——因为这些问题跨越编码实现、构建发布、运维支持多个阶段，更适合在 Plan 中集中处理。
