## 一、核心变更识别：从「外部依赖 svn」到「内置 svn」

v1 设计文档的核心假设是"用户自行安装 svn CLI，应用调用系统命令"。v2 将其改为"安装包捆绑各平台 svn 可执行文件，应用仅使用内置 svn"。这是一个**架构级变更**，影响范围超出单纯的功能调整。

### 1.1 变更波及范围

| 受影响模块 | 变更内容 | 状态 |
|-----------|---------|------|
| §1.2 核心理念 | "调用系统 svn CLI" → "捆绑 svn CLI" | ✅ 已更新 |
| §1.3 技术栈 | "调用系统 svn CLI" → "调用内置 svn CLI" | ✅ 已更新 |
| §2.1 前置依赖 | 要求用户预装 → 内置 svn，首次启动检测内置 | ✅ 已更新 |
| §2.3 内置 svn 版本维护（新增） | 更新策略、平台支持、打包方式、版本显示 | ✅ 新增 |
| §5.7 设置页面 | 删除"SVN 可执行路径"设置 | ✅ 已更新 |
| §6.2 实现策略 | 新增"通过 Tauri `resource_dir()` 获取内置 svn 路径" | ✅ 已更新 |
| §7 架构图 | "系统 svn CLI" → "内置 svn CLI" | ✅ 已更新 |
| §8.3 持久化存储 JSON | 删除 `"svnPath"` 字段 | ✅ 已更新 |

### 1.2 变更影响评估

| 维度 | 评估 |
|------|------|
| **前端影响** | 极小。仅设置页移除了一个选项（SVN 路径输入），前端 store/settings.ts 减少一个字段 |
| **后端影响** | 中等。`executor.rs` 需要从 Tauri `resource_dir()` 获取内置 svn 路径，而非直接调用 `svn` 命令名 |
| **构建/打包影响** | 较大。需要收集各平台 svn 二进制并放入 `resources/svn/{platform}/`，Tauri 配置 `resources` 打包 |
| **用户影响** | 正面。用户安装即用，零配置，零额外步骤 |
| **Plan 影响** | 后端实现细节需调整（svn 路径来源），但步骤拆分不变 |

## 四、变更前后的差异摘要

| 对比项 | 变更前（外部依赖） | 变更后（内置 svn） |
|--------|-----------------|-----------------|
| 用户安装步骤 | 先装 Free-SVN，再装 svn CLI，配置 PATH | 仅安装 Free-SVN |

| 运行时路径 | 调用 `svn`（依赖 PATH） | 调用 `resource_dir()/svn/{platform}/svn` |
| 设置项 | 有"SVN 可执行路径"自定义 | 无此设置项 |
| 持久化存储 | 含 `"svnPath"` 字段 | 不含 |
| 架构图 | 底层为"系统 svn CLI" | 底层为"内置 svn CLI" |

### 5.3 Plan 编写的 6 项注意点

以下合理归属 Plan 阶段，确认为编制 Plan 时必须处理的事项：

1. **核心类型定义** — Plan 第一步定义 `FileItem`、`SvnStatus`、`WorkspaceInfo` 等
2. **Services 拆分决策** — 19 个 invoke 方法超 200 行，Plan 决定单文件或拆分目录
3. **Store 接口签名** — Plan 为 4 个 store 写明 state/actions/getters
4. **Tauri Event 协议** — Plan 约定事件名与 payload
5. **XML 解析 Rust 结构体** — Plan 中对应每个 command 定义 `svn status --xml` 的输出结构
6. **内置 svn 路径封装** — Plan 确定 Rust 侧 svn 执行函数如何获取 `resource_dir()`
