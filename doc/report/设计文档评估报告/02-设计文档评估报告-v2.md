## 二、对设计师"归属 Plan 阶段"意见的独立判断

设计师认为以下 6 项属于 Plan 阶段而非 Design 阶段。以下逐个做独立判断：

### 1. Pinia Store 接口 → Plan 阶段

**设计师观点：** Store 的 actions/getters 属于"怎么做"，应归属 Plan。

**独立判断：✅ 正确**

- 设计文档已列出 4 个 store 的名称和职责（`workspace`、`fileList`、`settings`、`svn`）
- 这已在 Design 层面完成了"需要哪些状态管理模块"的定义
- 各 store 内部的 state 变量、actions 方法、getters 是编码细节，Plan 阶段定义即可
- 无争议

### 2. 核心类型定义 → Plan 阶段

**设计师观点：** 编码时自然产生。

**独立判断：✅ 正确（含一条提醒）**

- Design 文档不需要列出完整的 TypeScript 接口字段
- 但 `FileItem` 的结构（path、status、propsChanged 等）决定了文件列表的渲染和筛选逻辑
- **提醒：Plan 的第一步应定义核心类型**，否则步骤 1 定义的类型可能和步骤 5 的预期不一致，导致返工。但这一步确实在 Plan 中完成即可

### 3. Tauri Event 协议 → Plan 阶段

**设计师观点：** Rust 后端与前端通信细节。

**独立判断：✅ 正确**

- 设计文档已声明"长操作通过 Tauri event 推送进度到前端"（§6.2）
- 具体的事件名和 payload 结构是纯实现细节
- Plan 阶段定义时只需前后端约定一致

### 4. XML 解析结构 → Plan 阶段

**设计师观点：** Rust 内部实现细节。

**独立判断：✅ 正确**

- `svn status --xml` 的输出结构由 SVN 自身定义
- 文档已声明"通过 serde 解析 XML 输出"（§6.2），足矣
- Plan 中针对每个 command 定义对应的 Rust struct

### 5. "最近使用的日志"存储 → Plan 阶段

**设计师观点：** 提交面板的小功能。

**独立判断：✅ 正确**

- 存储方案（tauri-plugin-store 存最近 N 条 vs Pinia 运行时内存存）可由 Plan 决定
- 设计层面只需知道"有这个功能"

### 6. Services 拆分策略 → Plan 阶段

**设计师观点：** 纯实现决策。

**独立判断：⚠️ 原则正确，但有隐含硬约束**

- 从职责划分上，Services 拆分确实是实现决策
- **但有一个硬约束：** Vue 规范（`doc/rule/01-Vue前端开发规范.md §07`）规定 services 文件超过 200 行需拆分。19 个 invoke 方法 + 类型转换逻辑，单文件必然远超 200 行
- 如果 Plan 忽略此问题，AI 会写出一个超长文件，然后被 lint 检测触发拆分，浪费一轮编码
- **结论：** 设计师正确（Design 不需要处理），但 Plan 必须在第一步就决定拆分策略，不能"编码时自然发生"

### 汇总

| 序号 | 项目 | 判断 | 备注 |
|------|------|------|------|
| 1 | Pinia Store 接口 | ✅ Plan | 无争议 |
| 2 | 核心类型定义 | ✅ Plan | 提醒：Plan 第一步需定义 shape |
| 3 | Tauri Event 协议 | ✅ Plan | 无争议 |
| 4 | XML 解析结构 | ✅ Plan | 无争议 |
| 5 | 最近日志存储 | ✅ Plan | 无争议 |
| 6 | Services 拆分 | ✅ Plan（含硬约束） | Plan 必须处理，否则违反规范 |

**总判断：设计师 6 项判断中 5 项完全正确，1 项原则正确但有隐含约束（Plan 必须处理 Services 拆分，不能留到编码时自然发生）。**

首次评估（02 号报告）将这 6 项列为 Design 缺口是**判断过于严格**，本次评估予以正式修正。

## 四、从 A- 到 A/A+：仍需改进的设计层缺口

A- 评级的意思是"已达到 Plan 生成条件，但设计文档本身仍有可优化的空间"。以下缺口当前不影响 Plan 启动，但建议在开始 Plan 前或 Plan 过程中一并处理：

### 4.1 设计层缺口

| # | 缺口 | 涉及 | 说明 |
|---|------|------|------|
| 1 | **5 个操作有后端定义无前端 UI** | §6.1 vs §5 | Blame / Property / Lock / Unlock / Relocate 列入操作清单输入输出，但 §5 没有任何页面或对话框与这 5 个操作对应。要么补充 UI 设计，要么明确标注"后端 API 级，后续迭代实现" |
| 2 | **日志搜索能力描述与 svn CLI 实际能力不符** | §5.5 | `svn log --search` 只能搜索提交信息（message），不能按作者（author）或版本号（revision）过滤。设计文档描述的"搜索过滤（按提交信息、作者、版本号）"svn CLI 无法直接实现。需要设计决策：客户端全量加载后自行过滤 / 接受局限仅搜索提交信息 / 改用 `svn log` 多次调用 |
| 3 | **Locked 状态缺少操作按钮** | §4.4 | 状态图标表含有 `-`（locked）行，但操作按钮表中没有对应行。locked 文件应该显示什么按钮？[解锁]？是否需要特殊提示？映射不完整 |
| 4 | **错误信息展示方式未定义** | §6.2 | "不抛原始 stderr"和"友好展示"是原则，但没有定义交互形式：是用 Element Plus 的 ElMessage（toast）？弹窗对话框？状态栏文字？行内红色提示？不同方式影响前端组件设计 |
| 5 | **文件命名与 Vue 规范不一致** | §9.1 | `workspace.ts`、`fileList.ts` 不是规范要求的 kebab-case。应改为 `workspace-store.ts`、`file-list.ts`（Pan 规范 §06：TS 文件 kebab-case），否则编码阶段会被 lint 或 review 指出 |

### 4.2 处理建议

| # | 建议处理时机 | 原因 |
|---|-------------|------|
| 1 | 建议 Plan 前在 Design 中补充 | 如果直接归入 Plan，Plan 需要为这 5 个操作自行决定 UI 形态，可能偏离设计预期 |
| 2 | 建议 Plan 前在 Design 中修正 | 这是一个事实性错误（描述了 svn CLI 做不到的事），必须在设计层纠正 |
| 3 | Plan 阶段处理即可 | 操作按钮表的补齐可以在 Plan 中作为"文件列表"步骤的一部分完成 |
| 4 | Plan 阶段处理即可 | 通知交互是全局 UX 约定，Plan 第 1 步或第 2 步约定即可 |
| 5 | Plan 阶段处理即可 | 改名在 Plan 第一步定义文件结构时修正即可 |

### 4.3 关于 Shell 集成的说明

§3.3 的 Shell 集成在首次评估和本次评估中均未列入主缺口，原因：
- Shell 集成（右键菜单）属于安装/部署阶段的功能，不影响主窗口编码
- 即使后续实现有难度降级方案（从 Finder 扩展降为 URL scheme），不影响已完成的代码
- **建议：** Shell 集成从 MVP 范围中排除，作为 P2 功能在主体开发完成后独立实现

### 4.4 小结

以上 5 个缺口中，**仅 #1 和 #2 建议在 Plan 前在 Design 中处理**，其余 3 项在 Plan 阶段处理即可。

5 个缺口按严重程度排序：
- 🔴 **#2（日志搜索事实错误）** → 必须改 svn CLI 不支持的功能描述
- 🟡 **#1（5 个操作无 UI）** → 建议注明"后续迭代"
- 🟢 **#3（locked 按钮）** → Plan 中补充
- 🟢 **#4（错误交互形式）** → Plan 中约定
- 🟢 **#5（文件命名）** → Plan 中统一

如果 #1 和 #2 在 Design 中修正，评级可提升至 **A**。如果全部 5 项均优化，可达 **A+**。

## 五、用户体验（UX）与交互设计评估

前几轮评估聚焦于功能完整性和代码生成支撑度，本维度专门评估用户体验、交互细节和视觉样式层面。

### 5.1 用户体验问题

| # | 问题 | 涉及 | 说明 | 严重度 |
|---|------|------|------|--------|
| UX-1 | **主视图与提交面板存在双选机制冲突** | §4.4 + §5.4 | 主视图文件列表有复选框用于选择文件，底部显示"已选 N 个文件"；提交面板又有独立的复选框列。两者是否共享选择状态？如果用户在主页选了 3 个文件后点"提交"，这 3 个文件是否自动勾选？文档没有定义选择状态的传递规则 | 🟡 中 |
| UX-2 | **Diff 面板位置未确定** | §5.3 | "右侧/底部面板展开"二选一但未定。右侧面板适合宽屏但需缩小文件列表宽度；底部面板适合窄屏但可能信息不足。选型影响 MainLayout 的 layout 方案 | 🟡 中 |
| UX-3 | **文件列表空状态缺失** | §4.4 | 当工作副本没有变更时，文件列表区域显示什么？空白表格？"工作副本无变更，状态干净"提示？展示绿色勾号 + 版本号？这直接影响用户首次打开工具时的体验 | 🟡 中 |

| UX-10 | **状态图标仅有文本无颜色辅助** | §4.4 | 仅用 `M` `✚` `!` `🗑` `?` 文字符号标识文件状态，没有颜色辅助区分。常见的 SVN 工具会用颜色快速传递状态信息（红=冲突、绿=新增、蓝=修改），可扫描性明显不足 | 🟢 低 |
| UX-11 | **批量操作按钮缺失** | §4.4 | 底部显示"已选 3 个文件"但没有对应的批量操作按钮（如"批量还原"）或仅显示计数。用户选中多个文件后不知道下一步能做什么 | 🟢 低 |
| UX-12 | **Switch 对话框的分支列表来源未说明** | §5.8 | "提供分支列表下拉"但未说明分支列表如何获取。`svn ls ^/branches` 需要网络访问，且不同仓库的分支目录结构不同。用户可能看到空列表或等待超时 | 🟢 低 |
| UX-13 | **合并向导要求用户知道版本号** | §5.10 | "选择版本范围（起始版本 → 结束版本）"要求用户输入 SVN revision 数字，对不熟悉命令行 SVN 的用户不友好。最好能提供版本列表选择 | 🟢 低 |

### 5.3 视觉样式问题

当前设计文档在视觉层面几乎没有定义，这在"AI 编程"场景下意味着所有视觉决策都交给 AI 默认发挥，可能导致不一致：

| # | 问题 | 说明 |
|---|------|------|
| V-1 | **未定义色彩体系** | 没有主色/辅助色/成功/警告/错误色值。菜单/按钮/链接/背景色均由 AI 自由选择。Tailwind 默认色板可能和 Element Plus 主题不协调 |
| V-2 | **未定义暗色模式** | Tauri 桌面应用用户期望支持系统暗色模式，但文档完全没有提及。后期加入暗色支持时可能需要大量修改 |
| V-3 | **工具栏按钮无主次层级** | "提交"（高频操作）和"导出"（低频操作）在工具栏中视觉权重相同。没有通过图标大小、填充/描边风格来区分常用和非常用操作 |
| V-4 | **表格行交互状态未定义** | 文件列表行没有 hover、selected、active 等状态样式定义。AI 会使用 Element Plus 的默认表格样式，可能与整体视觉风格不协调 |
| V-5 | **Diff 面板响应式行为未定义** | 在窗口缩小时 diff 面板怎么做？代码会换行还是出现横向滚动条？窄屏下"右侧"面板是否自动转为"底部"？ |

### 5.4 根因分析：为何会出现这些 UX 缺口

设计文档本身的定位是"功能需求定义"（做什么），因此自然偏向功能点覆盖而非体验细节。UX 层面的缺失不意味着设计文档不合格，而是说明：

1. **UI/UX 设计阶段可以考虑独立于工程设计进行** — 视觉风格、色彩、暗色模式、微交互等可以单独输出一份 UI 设计规范
2. **MVP 阶段这些缺口可接受** — 很多 UX 细节（tooltip、responsive、加载状态）是 Plan 编码时自然可以覆盖的
3. **但部分重要 UX 决策（UX-1 双选机制、UX-2 diff 面板位置）需要在 Design 层明确**，因为它们直接影响组件结构

### 5.5 建议

| 类别 | 处理建议 |
|------|---------|
| **必须在 Design 明确的** | UX-1（选择状态传递规则）、UX-2（Diff 面板位置定一个） |
| **可在 Plan 中覆盖的** | UX-3 空状态、UX-4 加载态、UX-6 确认框、UX-8 导航方式、UX-10 颜色辅助、UX-11 批量操作、I-1 禁用 tooltip、I-2 外部工具流程 |
| **建议出独立 UI 规范** | V-1 色彩体系、V-2 暗色模式、V-3 按钮层级、V-4 表格交互态 |
