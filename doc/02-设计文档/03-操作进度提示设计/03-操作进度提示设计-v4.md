# 操作进度提示设计 v4

> **修订说明：** 根据 v3 评估报告（10-操作进度提示设计评估报告-v3）全面修订。v3 评估报告综合评分 95.8/100，指出 4 个 P3 优化项。v4 全部修复，含：应用退出操作安全、实施路径/试点策略、防抖双保险、showProgress 移至 composable、stdout 管道关闭标记、panic 恢复机制等共 13 项改进。

## 1. 概述

SVN 服务端文件传输操作（检出/更新/提交/切换/合并/导出）需要一个**非模态信息弹窗**，实时展示传输进度、速度、文件列表和取消功能。

## 2. 设计原则

- **非模态**：弹窗不阻塞应用其他功能操作（查看日志、差异等）
- **仅禁用工作空间切换**：弹窗打开期间不允许切换工作空间
- **信息清晰**：三种状态分明的文件展示
- **拖拽定位**：用户可拖拽标题栏调整弹窗位置；使用 CSS `transform` 而非 top/left 实现拖拽，避免重排；拖拽过程中 `pointer-events-auto` 区域限制提升性能

> **设计例外说明：** 文件区域使用深色终端背景色 `bg-slate-900`（深色模式）/ `bg-slate-950`（深色模式增强）是刻意为之，意在模拟终端操作的真实视觉体验。这与项目整体标准背景色 `bg-slate-800` 不一致，作为设计例外此处保留。此例外已在设计原则部分声明，确保阅读者首先了解。

## 3. 适用范围

| 操作 | SVN 命令 | 文件信息来源 | 进度行来源 |
|------|----------|------------|-----------|
| 检出 | `svn checkout` | stdout：`A    path/to/file` | stderr：`  22%   567K   1.2MB/s` |
| 更新 | `svn update` | stdout：`U/A/D path/to/file` | stderr：同上格式 |
| 提交 | `svn commit` | stdout：`Sending    path/to/file` → `Committed revision N` | stdout 行 + stderr Transmitting：`Transmitting file data .` → `   ...  45%` |
| 切换 | `svn switch` | stdout：类似 update 的 `U/A/D` 行 | stderr：同上格式 |
| 合并 | `svn merge` | stdout：类似 update 的 `U/A/D` + 冲突行 | stderr：同上格式 |
| 导出 | `svn export` | stdout：类似 checkout 的 `A    path/to/file` | stderr：同上格式 |

commit 操作的特殊阶段：在 Sending 阶段之后进入 Transmitting file data 阶段，此时输出 `Transmitting file data ...` 行，后续的 `   ...  45%` 行包含百分比进度（正则：`^\s*\.\.\.\s(\d+)%\s`）。此阶段的行不计入文件列表，仅用于更新进度百分比。

> **操作范围说明：** 上述 6 种操作为当前设计目标。`lock`/`unlock`/`cleanup`/`copy`(branch) 等操作的文件数量通常较少（单次操作涉及 1-3 个文件），暂不纳入进度提示。如需扩展可按同样模式添加。代码层面在 `svn/types.rs` 的 `WRITE_OPERATIONS` 表中已预留 lock/unlock 等操作的进度事件扩展点。

## 4. 弹窗布局

### 4.1 整体结构（从上到下）

```
┌─ 检出进行中 (85%) ────────────────────── 可拖拽标题栏
│  ████████████████████████████░░░░  85%   进度条（el-progress striped striped-flow）
│  ⚡ 2.3 MB/s  │  已完成 47 个  │  待传输 ~167 个  │ 00:12  统计栏
├──────────────────────────────────────────┤
│  ◆  src/main.rs                    暗淡     │
│  ◆  src/lib.rs                     暗淡     │
│  ◆  src/components/App.tsx         暗淡     │
│  ⟳  src/components/Sidebar.tsx     高亮     │  文件滚动区
│  ⟳  src/components/Navbar.tsx      高亮     │  （深色终端风格）
│  ▸  (等待中)                        正常     │
│  ▸  (等待中)                       正常     │
│  ▸  (等待中)                       正常     │
├──────────────────────────────────────────┤
│  操作完成后自动关闭              [取消中...]│  底栏
└──────────────────────────────────────────┘
```

> **待传输行计数说明：** 待传输文件数量极大（万级）时，仅在第一条待传输行显示总计数（如 `▸  (等待中，167 个文件)`），后续待传输行简化为 `▸  (等待中...)` 避免每行重复相同的大数字。

### 4.2 各模块详细说明

#### 标题栏
- 内容格式：`{操作名}进行中 ({百分比}%)`
- 可拖拽整窗移动；拖拽使用 `transform: translate(x, y)` 而非 top/left，避免触发重排
- 关闭按钮：操作完成后自动关闭，运行中不可手动关闭
- 拖拽约束：面板不可拖出视口范围；默认位置居中偏上；位置信息不持久化存储

#### 进度条
- 使用 `<el-progress>` 组件，配置 `striped` 和 `striped-flow` 属性实现条纹动效
- 百分比右侧数字
- 颜色：Element Plus 主题色（已在 style.css 中映射为 `green-500`），无需自定义渐变
- 百分比来源：
  - 稳定来源：SVN stderr 的百分比行 `  45%  1234K  1.2MB/s`
  - commit Transmitting file data 阶段：`   ...  45%` 行
  - 回退估算：若无百分比行，根据已传输文件数/估算总数推算
- 操作瞬间完成（单次输出即结束）：从 0% → 100% 以 300ms `ease-out` 缓动过渡动画展现，完成后延迟 500ms 自动关闭弹窗

#### 统计栏
- 四项信息横排：传输速度 / 已完成文件数 / 已耗时间 / 待传输估算数量
- 传输速度：从 SVN stderr 行解析（完整正则：`...\s+(\d+K?)\s+([0-9.]+ (KB/s|MB/s))\s+(\S+)`）
- 待传输估算：取最近 N（10-20）个文件的到达间隔的平均时间，除以总估算时间得出。首次估算在收到第 10 条文件行后触发。**容错设计**：前 20 个文件统计 + 每收到 50 个文件重新估算一次
- 响应式：窄屏（`< 640px`，使用 Tailwind `sm:` breakpoint，与项目其他组件一致）时自动折叠为两行（第一行：已完成数 + 待传输估算；第二行：速度 + 耗时）
- 数字格式化：超过 1000 显示 "1.2k"，超过 1000000 显示 "1.2M"（格式函数 `formatNumber` 提取到 `utils/format.ts`，供项目全局复用）

#### 文件滚动区

**三种文件状态：**

| 状态 | 图标（Lucide） | 导入语句 | 深色模式 | 浅色模式 |
|------|---------------|---------|---------|---------|
| 已完成 | `Check`（◆） | `import { Check } from 'lucide-vue-next'` | 暗淡灰色（text-slate-600） | 暗淡灰色（text-slate-400） |
| 进行中 | `Loader`（⟳，旋转动画） | `import { Loader } from 'lucide-vue-next'` | 黄色高亮（text-amber-400）+ 背景高亮（bg-slate-700） | 橙色高亮（text-amber-600），无背景色 |
| 待传输 | `MoveRight`（▸） | `import { MoveRight } from 'lucide-vue-next'` | 正常色（text-slate-300） | 正常色（text-slate-600） |

- 文件路径过长处理：前 20 字符 + `...` + 后 15 字符，hover 时 tooltip 显示完整路径
- 已完成路径用 `line-through` 删除线表示结束
- 深色模式背景 bg-slate-900，浅色模式背景 bg-white
- 等宽字体（font-mono）
- 通过 Tailwind `dark:` 前缀切换
- **自动滚动**：新文件行加入时自动滚动到底部；如果用户手动向上滚动浏览历史行（`scrollTop + clientHeight < scrollHeight - threshold`），暂停自动滚动；用户滚动到底部附近时恢复自动滚动。阈值（`scrollThreshold`）作为 composable 的参数，默认 20px，可按需调整

#### 底栏
- 左侧文字提示："操作完成后自动关闭"
- 右侧取消按钮

取消按钮交互流程：
1. 用户点击取消 → 按钮变为 loading 状态，文案变为 "正在取消中..."
2. 后端终止 SVN 进程 → 推送 `operation:cancelled` 事件（**不再推送** `operation:completed` + `result:"cancelled"`）
3. 弹窗显示"操作已取消"状态 → 500ms 后显示成功 toast "操作已取消"
4. 弹窗变为可关闭状态（显示关闭按钮）

## 5. 技术方案

### 5.1 后端改造

**核心改造**：`src-tauri/src/svn/executor.rs` 在现有 `run_svn` 实现基础上增加**双管道逐行读取 stdout + stderr** 的进度函数。

#### 与现有 `run_svn` 的共存策略

`run_svn` 保持不变（供不需要进度的只读操作使用），新增 `run_svn_with_progress` 用于需要进度的写操作：

```
现有 run_svn() → status/info/diff/log/blame（无进度事件）
新增 run_svn_with_progress() → checkout/update/commit/switch/merge/export（推送进度事件）
```

两个函数共享相同的命令行构造逻辑和认证凭据传递。command 层根据需要选择调用哪个函数。

#### 实施路径（试点 → 推广）

为降低风险和调试成本，按以下分阶段策略实施：

1. **试点阶段**：从 `checkout_repo` 开始，将其改为调用 `run_svn_with_progress`，验证双管道方案、事件推送、取消机制正确
2. **推广阶段**：试点验证通过后，依次改造 `update_workspace`、`switch_branch`、`merge_branch`、`export_workspace`
3. **可选阶段**：评估 `create_commit` 是否需要进度提示（commit 为原子操作，Sending/Transmitting 阶段信息量有限）

#### 双管道读取策略

使用两个独立线程分别读取 stdout 和 stderr，通过 channel 同步：

```
stdout 线程 ──→ line channel ──→ 解析文件行 → emit("operation:line")
                                   ↓
stderr 线程 ──→ progress channel ──→ 解析百分比/速度 → emit("operation:progress")
```

- stdout 线程：`BufReader::new(stdout)` 逐行读取，匹配文件操作行（`A/U/D/Sending...`），通过 channel 发送到主处理循环
- stderr 线程：`BufReader::new(stderr)` 逐行读取，匹配进度行（百分比/速度），通过 channel 发送到主处理循环
- 两个线程通过独立的 `mpsc::channel` 与主线程同步
- `operation:line` 用于文件列表更新，`operation:progress` 用于进度条/速度/统计更新，两者独立并发
- **stdout 管道关闭标记**：stdout 线程检测到管道意外关闭后，推送一条终止标记行 `{ operation, filePath: "[管道中断]", status: "completed" }`，前端据此显示"部分文件列表可能不完整"提示。此标记不影响操作结果，仅用于告知用户文件列表可能不完整。

#### 后端 Throttle 节流

进度事件（`operation:progress`）采用**后端 throttle** 策略：
- 首条百分比/速度行立即发送
- 后续 200ms 窗口内最多发送一条（合并累积的百分比值）
- 保证前端不会在 1 秒内收到超过 5 条 progress 事件
- 使用简单的 timestamp 比较实现，不引入额外依赖

#### 取消机制与竞争条件处理

改用 `Arc<Mutex>` 在线程间共享取消标志和子进程句柄：

```rust
struct SharedState {
    cancelled: bool,
    child: Option<Child>,
}
let shared = Arc::new(Mutex::new(SharedState { cancelled: false, child: None }));
```

- stdout/stderr 两线程每次读取一行前检查 `shared.cancelled`，若为 true 则终止读取
- 取消操作：设置 `cancelled = true`，然后 kill 子进程
- 文件解析循环中每行也检查 CANCELLED 标志，避免在取消后继续处理

#### 新增函数

```rust
async fn run_svn_with_progress(
    args: &[&str],
    cwd: &str,
    credentials: Option<&SvnCredentials>,
    app_handle: AppHandle,
    operation: &str,
) -> Result<String, AppError>
```

执行流程：
1. `Command::new(svn_path)` + `stdout(Stdio::piped())` + `stderr(Stdio::piped())`
2. 启动两个异步线程分别读取 stdout 和 stderr
3. 每解析一行 stdout 后通过 `app_handle.emit("operation:line", payload)` 推送
4. 每解析一行 stderr 后通过 `app_handle.emit("operation:progress", payload)` 推送（带后端 throttle，200ms 窗口）
5. 整体完成后推送 `operation:completed` 或 `operation:error`

> **异步兼容性说明：** 在 `spawn_blocking` 闭包内通过 `app_handle.clone()` 调用 `emit()` 是线程安全的——Tauri 内部有 channel 转发处理。

**事件生命周期**：
```
operation:started → (operation:line × N 并行 + operation:progress 节流) → operation:completed
                                                                           → operation:error
                                                                           ← operation:cancelled
```

说明：`operation:line` 与 `operation:progress` 独立并发——line 用于文件列表更新，progress 用于进度条/速度/统计更新。两者不相互阻塞。
`operation:cancelled` 事件可在生命周期中**任何阶段**（started 之后、completed/error 之前）触发，表示用户中断操作。`completed`/`error` 是操作的自然终点，`cancelled` 是用户主动终结点。

#### 错误处理与恢复

- **stderr 线程出错**：stderr 线程若因管道损坏异常退出，主处理循环检测到后降级运行——进度百分比回退到按文件数估算模式（已传输/已传输+待传输估算）。速度显示改为 "计算中..."。操作继续执行，不中断用户操作。
- **管道损坏恢复**：若 stdout 或 stderr 管道意外关闭（如 SVN 进程异常退出），使用 `try_wait()` 检查子进程状态。若子进程已退出但未返回完整结果，推送异常退出错误。管道损坏后不重新初始化 Socket（Tauri event 系统自动处理连接重建）。stdout 管道关闭时推送终止标记行（见双管道读取策略）。
- **双线程 panic 安全**：任一线程 panic 不会导致另一线程挂起。通过 `catch_unwind` 包装线程体，panic 时关闭对应 channel 让主循环自然结束。> **注意：** `catch_unwind` 不能跨 FFI 边界捕获（Tauri event emit 涉及外部调用），因此它保护的是 BufReader 同步读取部分。panic 发生时 channel 关闭，主循环通过无法接收消息自然终止。**panic 后恢复机制**：主循环检测到 channel 关闭后，启动超时兜底（5 秒），若超时后子进程仍未退出，调用 `kill_current_process()` 强制终止，防止残留 SVN 进程。

#### 速度数据提取正则

SVN 进度行格式：`  45%   1234K   1.2MB/s   00:12`

完整正则匹配模式：
```
...\s+(\d+K?)\s+([0-9.]+ (KB/s|MB/s))\s+(\S+)
```

分组说明：
- 组 1：传输字节数（如 `1234K`）—— `...` 前缀前有至少三个空格
- 组 2 + 组 3：速度（如 `1.2 MB/s`）
- 组 4：剩余时间（如 `00:12`）

commit Transmitting file data 阶段进度正则：
```
^\s*\.\.\.\s(\d+)%\s
```
匹配 `   ...  45%` 格式，提取组 1 为百分比值。

#### 文件列表淘汰策略

当 `fileLines` 数量超过 1000 时，在一次批量操作中淘汰最旧的 200 行已完成（status=completed）行。不能逐行淘汰以避免 DOM 频繁更新。

#### 各操作的行解析规则

| 操作 | 匹配模式 | 文件路径提取 |
|------|---------|------------|
| checkout | `^(A)\s+(.+)$` | 第 2 组 |
| update | `^(U\|A\|D)\s+(.+)$` | 第 2 组 |
| commit | `^Sending\s+(.+)$` | 第 1 组 |
| switch | `^(U\|A\|D)\s+(.+)$` | 第 2 组 |
| merge | `^(U\|A\|D\|C)\s+(.+)$` | 第 2 组 |
| export | `^(A)\s+(.+)$` | 第 2 组 |

### 5.4 状态管理

**核心进度数据**保持在 `svnStore`（`src/stores/svn.ts`），仅放跨组件/跨页面共享的前后端通信状态：

```typescript
// svnStore — IPC 通信层
const progress = ref<OperationProgress | null>(null)
const fileLines = ref<OperationLine[]>([])
const isOperationRunning = ref(false)

// 操作进行中守卫（双保险：svnStore 层 + composable 层）
function checkOperationRunning(): boolean {
  return isOperationRunning.value
}
```

**弹窗 UI 状态**抽离到 `useProgressOverlay` composable（`src/composables/useProgressOverlay.ts`），负责弹窗的本地 UI 状态：

```typescript
// useProgressOverlay — 弹窗 UI 层
const isDragging = ref(false)
const isCancelling = ref(false)
const panelPosition = ref({ x: 0, y: 0 })
const showProgress = ref(false)  // UI-only：是否显示弹窗
const scrollThreshold = 20       // 自动滚动暂停检测阈值（可配置参数）
const fileListRef = ref<HTMLElement | null>(null)
// ...拖拽、取消、自动滚动方法
```

**职责边界：**
- `svnStore` = IPC 通信层（事件监听、进度数据存储、跨组件共享数据、操作进行中守卫）
- `useProgressOverlay` = 弹窗 UI 层（弹窗显示控制、拖拽状态、取消 loading、面板位置、自动滚动逻辑）

**操作请求防抖（双保险）：**
- **svnStore 层**（IPC 保险）：`checkOperationRunning()` 检查 `isOperationRunning`，防止绕过 composable 直接调用 store 的操作
- **composable 层**（UI 保险）：`tryRequestOperation()` 防止同一组件内快速重复触发

**进度弹窗生命周期与队列的关系：** 进度弹窗仅在实际操作开始后打开（即在 `svn::queue::try_lock` 成功后）。当 `try_lock` 返回 `OperationInProgress` 时，不打开进度弹窗，直接显示错误提示。

**`svnStore.initEventListeners()` 监听配置：**

```typescript
function initEventListeners() {
  if (initialized) return
  initialized = true
  Promise.all([
    listen<OperationProgress>('operation:progress', e => { progress.value = e.payload }),
    listen('operation:started', () => {
      isOperationRunning.value = true
      fileLines.value = []
      useFileListStore().isOperationRunning = true
    }),
    listen<OperationLine>('operation:line', e => {
      fileLines.value.push(e.payload)
    }),
    listen<CancelledPayload>('operation:cancelled', () => {
      isOperationRunning.value = false; progress.value = null
      useFileListStore().isOperationRunning = false
    }),
    listen<OperationResult>('operation:completed', () => {
      isOperationRunning.value = false; progress.value = null
      useFileListStore().isOperationRunning = false
    }),
    listen('operation:error', () => {
      isOperationRunning.value = false; progress.value = null
      fileLines.value = []
      useFileListStore().isOperationRunning = false
    }),
  ]).then(fns => { unlistenFns.push(...fns) })
    .catch(err => console.error('[svn store] Failed to register event listeners:', err))
}
```

## 6. 边界情况

| 场景 | 处理方式 |
|------|---------|
| 操作失败 | 推送 `operation:error`，弹窗显示错误信息和关闭按钮 |
| 用户取消 | 点击取消 → loading + "正在取消中..." → 后端终止进程 → **`operation:cancelled`**（不再推送 `operation:completed` + `result:"cancelled"`）→ 弹窗显示"操作已取消" → 500ms 后显示成功 toast "操作已取消"，弹窗可关闭 |
| 大量文件（万级） | 文件列表最多保留 1000 行，超过时一次性淘汰最旧的 200 行已完成行 |
| 文件路径过长 | 前 20 字符 + `...` + 后 15 字符，hover tooltip 显示完整路径 |
| 统计栏数字显示 | 超过 1000 显示 "1.2k"，超过 1000000 显示 "1.2M"（`formatNumber` 函数提取到 `utils/format.ts`） |
| 操作瞬间完成 | 进度条 0% → 100% 以 300ms `ease-out` 过渡动画，完成后延迟 500ms 自动关闭弹窗 |
| 10 秒无响应 | 从首次操作启动开始计时，若连续 10 秒未收到任何输出行（包括 stdout 和 stderr），状态栏显示"正在连接服务器..."，进度条为不确定状态 |
| 认证失败 + 进度弹窗 | 进度弹窗保持打开（不关闭），AuthDialog 模态弹窗叠加在进度弹窗上方。认证完成后进度弹窗继续更新，认证取消时进度弹窗也关闭并显示已取消 toast |
| 网络中断恢复 | SVN 命令超时（60s）后推送 `operation:error`，弹窗显示错误信息，用户可重试 |
| 自动滚动暂停 | 用户手动向上滚动时（`scrollTop + clientHeight < scrollHeight - scrollThreshold`，默认 20px）暂停自动滚动；滚动到底部附近时恢复 |
| 操作队列与弹窗生命周期 | 进度弹窗仅在 `try_lock` 成功后打开；`try_lock` 失败时显示"操作进行中"提示，不打开进度弹窗 |
| 多个进度弹窗竞争 | 双保险防抖：composable 层 `tryRequestOperation()` + svnStore 层 `checkOperationRunning()`。同一时刻只允许一个进度弹窗激活 |
| **应用退出时操作安全** | 操作进行中用户关闭应用（窗口 X 按钮 / 系统托盘退出）时：先静默 kill 当前 SVN 子进程 → 执行 svn cleanup 恢复工作副本 → 释放写操作锁（`queue.unlock()`）→ 重置取消标志（`cancelled = false`）→ 然后退出应用。避免残留进程和锁导致下次启动异常 |
