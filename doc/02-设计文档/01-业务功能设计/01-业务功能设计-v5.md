# 01-业务功能设计 v5

> Free-SVN — 基于 Tauri + Vue 的跨平台 SVN 图形化工具
> 更新说明：v5 新增 SVN 认证凭据管理三场景（检出录入、失败重试、凭据管理），重写 §2.2 认证策略，新增 §2.8 凭据安全规范、§5.14 认证重试对话框、§5.15 凭据管理对话框、§6.2 认证命令

## 1. 概述

### 1.1 目标

提供一个跨平台（macOS / Windows / Linux）的 SVN 图形化客户端，核心体验对标 TortoiseSVN，包含系统右键菜单快捷操作和主窗口完整功能。

### 1.2 核心理念

- **单工作副本模式** — 每次只关注一个工作副本，切换通过下拉菜单选择已检出的副本
- **变更文件列表为默认视图** — 打开软件直接看到所有变更，提升效率
- **捆绑 svn CLI** — 安装包内置各平台 svn 可执行文件，应用仅使用内置 svn；用户安装即用，零额外步骤
- **内置差异查看器** — 默认基于 `diff2html` + `highlight.js` 实现语法高亮 diff，可选配外部工具

### 1.3 技术栈

| 层 | 技术 |
|----|------|
| 前端 | Vue 3 + TypeScript + Vite + Tailwind CSS + Element Plus + Pinia + Vue Router |
| 后端 | Tauri 2 (Rust) + serde + thiserror |
| 存储 | tauri-plugin-store（本地配置持久化） |
| I18n | vue-i18n（按需引入） |
| SVN | 调用内置 `svn` CLI 解析 XML/文本输出 |

### 2.1 前置依赖

- **内置 svn**：安装包捆绑对应平台的 svn 可执行文件（约 2-3MB），应用仅使用内置 svn，不依赖系统安装
- **首次启动**：检测内置 svn 是否可执行，异常时提示用户重新安装 Free-SVN
- **内置 svn 的认证缓存**：内置 svn 的 `--non-interactive` 模式下若配合 `--password-from-stdin` 传参首次认证成功，凭据会自动写入 `~/.subversion/auth/` 目录，后续不带凭据的操作可直接使用缓存

### 2.2 SVN 认证交互策略（v5 重写）

svn CLI 可能触发交互式认证提示（用户名/密码）。v5 改为分层策略，支持用户在 UI 中输入凭据。

#### 认证优先级

```
用户操作需要认证
  ├─ ① 当前会话中已有凭据（本次运行已输入过）
  │    → 复用会话内存凭据，不重复弹窗
  ├─ ② 用户主动填写的表单凭据
  │    → 检出对话框/认证重试对话框/凭据管理对话框 输入的凭据
  ├─ ③ ~/.subversion/auth/ 缓存凭据
  │    → svn 自动使用（需在首次认证时勾选"保存"或系统自动缓存）
  └─ ④ 无可用凭据
       → svn --non-interactive 返回认证失败
```

#### 凭据传入方式

| 方式 | 是否采用 | 说明 |
|------|---------|------|
| `--password` 命令行参数 | ❌ 禁止 | 密码出现在进程列表（`ps aux` 可见），不安全 |
| `--password-from-stdin` 管道输入 | ✅ 强制使用 | 通过 stdin 管道传密码，不出现在进程列表和日志中 |
| `--username` 命令行参数 | ✅ 使用 | 用户名不敏感，可通过命令行参数传递 |

**安全规则：**
1. 密码**只**通过 `--password-from-stdin` 传递
2. 密码不在应用日志中记录（日志中记录为 `***` 掩码）
3. 密码不在进程命令行字符串中暴露
4. 密码在内存中不做跨会话持久化（重启后需要重新输入，除非勾选保存）
5. 勾选"保存凭据"时通过 svn 自动写入 `~/.subversion/auth/`

#### 有凭据 vs 无凭据的执行模式

| 条件 | `--non-interactive` | `--username` | `--password-from-stdin` | 缓存行为 |
|------|--------------------|-------------|------------------------|---------|
| 无凭据 | ✅ 保留 | ❌ 不传 | ❌ 不传 | 走已有缓存 / 返回 `SVN_AUTH_FAILED` |
| 有凭据 + 保存 | ❌ 移除 | ✅ 追加 | ✅ 管道 | 认证成功后 svn 自动写入 `~/.subversion/auth/` |
| 有凭据 + 不保存 | ✅ 保留（阻止缓存写入） | ✅ 追加 | ✅ 管道 | 本次会话内有效，重启后不保留 |

#### 认证失败处理

当 svn 执行返回认证相关错误（stderr 包含"Authentication realm"、"Password:"、"certificate" 等关键词，或非零退出码），后端返回 `AppError::SvnAuthFailed`，前端行为：

1. 检出/更新等操作失败 → 弹**认证重试对话框**（见 §5.14）
2. 用户输入凭据 → 重试 → 成功则继续流程
3. 用户取消 → 终止操作

#### 首次认证引导

- 检出对话框（§5.6）提供凭据输入区供首次认证
- 弹认证重试对话框时，提示信息应区分"网络问题"和"认证问题"
- 认证失败提示文案示例："SVN 服务器需要认证，请输入用户名和密码，或确认 ~/.subversion/auth/ 中已有有效的缓存凭据"

### 2.3 内置 svn 版本维护与许可证合规

| 方面 | 策略 |
|------|------|
| 更新时机 | 内置 svn 跟随 Free-SVN 发版更新，不提供应用内动态升级 |
| 平台支持 | macOS：使用 Homebrew 或官方编译的二进制（Apache 2.0）；Windows：使用官方 Apache 二进制（Apache 2.0），**不采用 SlikSVN**（专有许可不可再分发）；Linux：使用静态编译二进制（Apache 2.0）|
| 打包方式 | 各平台 svn 二进制存放在 `resources/svn/{platform}/` 目录，由 Tauri `resources` 配置打包 |
| 版本显示 | 底部状态栏和关于页显示内置 svn 版本号 |
| 许可证声明 | 应用关于页面和发行说明中声明 Apache 2.0 许可证和 Subversion 版权归属 |
| 合规检查 | 发行前确认所有捆绑二进制的许可证允许再分发 |

### 2.4 并发操作控制策略

**背景**：多个 SVN 操作同时执行可能导致 `.svn/` 元数据损坏、工作副本状态不一致。

**控制规则**：

| 分类 | 操作 | 并发策略 | 说明 |
|------|------|---------|------|
| 只读操作 | status, info, diff, log, blame | 可并发执行 | 多个只读操作可同时运行，不互斥 |
| 写操作 | commit, update, switch, merge, checkout, cleanup, revert, resolve, add, delete, ignore, lock, unlock, copy, export, relocate | 互斥执行 | 同一时间每个工作副本只允许一个写操作 |
| 杂项 | property | 互斥执行 | 属性修改归类为写操作 |

**实现方式：**
1. 后端（Rust）：使用 `Mutex` 包裹操作执行器，写操作加锁，只读操作跳过锁
2. 前端：写操作进行时，对应工具栏按钮和菜单项禁用，ProgressOverlay 显示中
3. 队列行为：写操作进行中触发的新写操作 → 提示"当前有操作正在进行"（不排队）
4. 写操作锁仅作用于**同一工作副本**；不同工作副本的操作互不影响
5. `status` 等只读操作若在写操作执行期间被调用，返回上次缓存的列表或等待写操作完成（前端优先显示缓存，写操作完成后自动刷新）

### 2.5 长操作取消机制

**支持取消的操作**：checkout、update、export、merge（commit 是原子操作，不可取消）

**取消交互**：
- ProgressOverlay 上显示 [取消] 按钮
- 点击取消后 → 按钮变为"正在取消..."（禁用态）→ 发送终止信号

**取消实现**：
1. 前端点击取消 → invoke('cancel_operation') → Tauri event `operation:cancelled` 通知后端
2. Rust 侧发送 `SIGINT`/`SIGTERM` 终止 svn 进程
3. 进程终止后自动执行 `svn cleanup` 清理工作副本状态
4. 取消完成后 → Tauri event 通知前端 → 恢复 UI 状态
5. 清理过程本身不可取消

**取消后状态**：
- 如果 svn 已部分写入（如 update 已更新部分文件），cleanup 会恢复到一致状态
- 前端刷新文件列表反映当前实际状态

### 2.6 操作前置条件校验

每个操作在执行前必须经过前置条件检查，不通过则按钮禁用或弹出提示：

| 操作 | 前置条件 | 不满足时行为 |
|------|---------|-------------|
| 提交 | 至少选中一个文件、有工作副本 | 提交按钮禁用 |
| 更新 | 有工作副本 | 按钮禁用 |
| 日志 | 有工作副本 | 按钮禁用 |
| 差异 | 已选中一个文件 | 差异按钮禁用 |
| 检出 | URL 非空、目标路径非空 | 检出按钮禁用 |
| 切换 | 目标 URL 非空、有工作副本 | 按钮禁用 |
| 还原 | 至少选中一个文件 | 还原按钮禁用 |
| 合并 | 源 URL 非空、有工作副本 | 合并按钮禁用 |
| 清理 | 有工作副本 | 按钮禁用 |
| 导出 | 目标路径非空、有工作副本 | 导出按钮禁用 |
| 忽略 | 至少选中一个可忽略的文件 | 忽略按钮隐藏 |
| 解锁 | 至少选中一个锁定文件 | 解锁按钮隐藏 |

### 2.7 持久化存储异常恢复

tauri-plugin-store 持久化 JSON 文件可能因写入中断、用户误编辑等原因损坏。

**恢复策略**：
- 读取 store 时使用 try_deserialize，失败时：
  1. 将损坏文件备份为 `.store.json.bak`
  2. 重置为默认配置
  3. 应用正常启动，不崩溃
- 常规操作不需要备份数据（SVN 工作副本是源数据，丢失 store 只是丢失最近列表）
- 跨版本升级时，store 中不存在的字段采用默认值，不做 migration（tauri-plugin-store 自然处理）

### 2.8 认证凭据管理安全规范（v5 新增）

#### 凭据存储

Free-SVN **不自建加密存储**，凭据保存完全依托 svn 自身的 `~/.subversion/auth/` 缓存机制：

| 机制 | 说明 |
|------|------|
| 缓存位置 | `~/.subversion/auth/svn.simple/`（按服务器 realm 哈希命名的文件）|
| 存/取 | 由 svn CLI 内部自动管理 |
| 加密 | svn 在 Linux/macOS 上不加密存储明文密码（Windows 上使用 DPAPI 加密）|
| 清除 | 删除 `~/.subversion/auth/` 中对应文件 |

**安全考虑**：svn 的缓存机制在不同平台上安全级别不同：
- Windows：使用 DPAPI 加密
- macOS/ Linux：明文存储在 `~/.subversion/auth/`（权限 600）

应用不在设计层面对此做额外加密（避免引入复杂性），但在关于/设置页面提供安全提示。

#### 凭据生命周期

```
应用启动
  └→ 会话凭据池为空
     ├→ 用户通过检出/凭据管理/重试对话框输入凭据
     │    ├→ 存入会话凭据池（仅内存）
     │    ├→ 传入 run_svn → svn 执行
     │    │    ├→ 认证成功 + 勾选保存 → svn 缓存到 ~/.subversion/auth/
     │    │    ├→ 认证成功 + 不保存 → 仅会话内存有效
     │    │    └→ 认证失败 → 弹重试对话框
     │    └→ 应用重启 → 清空（除非已保存到 ~/.subversion/auth/）
     └→ 用户清除缓存
          └→ 删除 ~/.subversion/auth/ 中对应文件
```

#### 日志规则

| 是否记录 | 内容 |
|---------|------|
| ❌ 禁止 | 密码原文 |
| ✅ 可记录 | 用户名、操作命令、退出码、耗时 |
| ✅ 掩码后记录 | `--password-from-stdin [***]` |

## 3. 系统右键菜单

> 与 v4 保持一致。

### 3.1 菜单内容

右键菜单按目标位置区分：

**在 SVN 工作副本目录上右键（完整菜单）：**

```
SVN 提交...              → 打开主窗口提交面板
SVN 更新                 → 后台执行 + 系统通知
SVN 查看日志             → 打开主窗口日志视图
SVN 更新到版本...        → 打开版本选择对话框
───────────────────────────────────────────
SVN 差异对比             → 打开差异视图
SVN 还原...              → 确认后还原
───────────────────────────────────────────
SVN 切换(Switch)...      → 打开切换分支对话框
SVN 分支/标签...         → 打开分支/标签创建对话框
SVN 合并...              → 打开合并向导
───────────────────────────────────────────
SVN 清理                 → 后台执行 + 系统通知
SVN 导出...              → 选择目录后导出
```

**在普通文件夹/空目录上右键（仅检出）：**

```
SVN 检出...              → 打开检出对话框，目标目录自动填入
```

**在工作副本内文件上右键（跟目录菜单相同，但操作作用于当前文件）：**

同目录菜单，但提交/差异等操作限定为选中文件。

### 3.2 行为逻辑

| 条件 | 行为 |
|------|------|
| 主窗口未打开 | 启动主窗口，设置当前工作副本为该目录 |
| 主窗口已打开，但当前副本不同 | 切换到该目录作为当前工作副本 |
| 操作无需主窗口交互（更新/清理） | 后台执行，结果通过系统通知展示 |
| 操作需主窗口交互（提交/日志/差异） | 焦点回到主窗口，打开对应视图 |

### 3.3 跨平台实现

| 平台 | 实现方式 |
|------|----------|
| macOS | Finder 扩展 / Quick Action / 通过 Tauri 注册 URL scheme 回调 |
| Windows | 注册表 Shell 扩展（右键菜单 GUID 注册） |
| Linux | Nautilus 脚本 / Thunar UCA 配置 |

### 3.4 系统托盘与后台模式

**设计背景**：§5.7 设置了"开机自启动"，但关闭窗口行为未定义。如果自启动后没有后台模式，启动就打开全屏窗口，用户体验不好。

**关闭窗口行为**：
- 用户点击关闭按钮（红绿灯 / [×]）→ **最小化到系统托盘**，应用继续运行
- 用户右键托盘图标 → 选择"退出"→ 完全退出
- macOS 上：点击 Dock 关闭窗口按钮 → 同关闭按钮行为（最小化到托盘）

**托盘图标**：
- 默认图标：应用图标的小型版本（单色/简约版）
- 操作进行时：图标显示小徽章或动画表示有后台操作

**托盘右键菜单**：

```
显示主窗口
──────────────────
SVN 更新当前工作副本      → 后台执行 + 系统通知
SVN 清理当前工作副本       → 后台执行 + 系统通知
──────────────────
关于 Free-SVN
退出
```

**后台操作反馈**：
- 后台操作（更新/清理）完成时 → 系统通知（macOS Notification Center / Windows Toast / Linux notify-send）
- 用户点击通知 → 激活主窗口并定位到相关结果

**开机自启动的关系**：
- 自启动时应用启动 → 最小化到托盘（不打开主窗口）
- 若有关联右键菜单操作传递给首实例的参数，则打开主窗口并跳转

### 3.5 单实例运行防护

**策略**：使用 `tauri-plugin-single-instance` 确保只运行一个实例。

**第二实例启动行为**：
1. 如果主窗口最小化到托盘 → 激活主窗口并前置
2. 如果第二实例携带了右键菜单参数（如 --workspace=/path）→ 传递给首实例处理，首实例切换到该工作副本
3. 忽略第二实例的其他参数

**为什么不允许多实例**：
- 两个实例操作同一工作副本 → `.svn/` 元数据损坏
- 后台右键菜单操作与主窗口操作冲突
- macOS / Windows 的用户直觉是"一个应用一个窗口"

### 3.6 窗口状态持久化

**策略**：每次关闭窗口时记录位置和大小，下次打开时恢复。

| 持久化项 | 说明 |
|---------|------|
| 窗口位置 (x, y) | 保存并恢复 |
| 窗口大小 (width, height) | 保存并恢复 |
| 是否最大化 | 记录是否最大化状态 |
| 多显示器 | 如果上次关闭的显示器已不存在，回退到主显示器居中显示 |

使用 `tauri-plugin-window-state` 实现。

### 4.1 整体结构

```
┌────────────────────────────────────────────────────────┐

│  顶部状态栏                                            │
```

### 6.2 实现策略

**CLI 调用方式（v5 增加 credentials 支持）：**
```rust
// 统一执行模式
fn svn_exec(command: &str, args: &[&str], cwd: &Path, credentials: Option<&SvnCredentials>) -> Result<SvnOutput, SvnError>
```

- 使用内置 svn 可执行文件
- 调用 `svn` 命令时使用 `--xml` 参数获取 XML 输出
- 设置超时（默认 60 秒）
- 异步执行
- 长操作通过 Tauri event 推送进度
- 写操作通过 Mutex 互斥
- **有凭据时**：移除 `--non-interactive`，追加 `--username`，通过 stdin 管道写入密码
- **无凭据时**：保持 `--non-interactive`

**密码 stdin 管道实现：**
```rust
// 有凭据时通过 stdin 传密码
if let Some(creds) = &credentials {
    let mut child = Command::new(svn_path)
        .args(args)
        .current_dir(cwd)
        .stdin(Stdio::piped())
        .stdout(Stdio::piped())
        .stderr(Stdio::piped())
        .spawn()?;

    // 通过 stdin 写入密码
    if let Some(mut stdin) = child.stdin.take() {
        stdin.write_all(creds.password.as_bytes())?;
        stdin.flush()?;
    }
    let output = child.wait_with_output()?;
    // ...处理结果
} else {
    // 无凭据：正常执行，不写 stdin
}
```

**错误处理（v5 新增认证错误检测逻辑）：**
```rust
// 在 svn 执行失败时，额外检测是否为认证错误
fn classify_error(stderr: &str) -> AppError {
    if is_auth_error(stderr) {
        // 提取 realm 信息
        let realm = extract_realm(stderr);
        AppError::SvnAuthFailed(realm)
    } else if is_timeout(stderr) {
        AppError::Timeout(...)
    } else {
        AppError::SvnCommand(...)
    }
}

fn is_auth_error(stderr: &str) -> bool {
    let keywords = ["Authentication realm", "认证领域", "authorization failed",
                    "认证失败", "access denied", "No credentials", "密码"];
    keywords.iter().any(|k| stderr.contains(k))
}
```

### 6.3 日志与诊断体系

**日志安全规则（v5 新增）：**
- INFO 级别记录命令参数时，密码部分掩码为 `[***]`
- 认证错误的密码内容不记录到日志
- `~/.subversion/auth/` 文件内容不记录

## 7. 路由设计


## 8. 数据流

### 8.1 前端状态管理 (Pinia Store)

```
stores/
  ├── workspace.ts    -- 当前工作副本信息、最近副本列表
  ├── fileList.ts     -- 文件状态列表、选中状态、筛选/搜索
  ├── settings.ts     -- 设置项
  └── svn.ts          -- SVN 操作封装（invoke commands）
```

### 8.2 主要数据流

```
认证失败 → 重试（v5 新增数据流）

  用户执行服务端操作（update/commit/log/diff 等）
    → invoke('svn_command')
    → 后端返回 AppError::SvnAuthFailed(realm)
    → 前端检测到错误码 = 'SVN_AUTH_FAILED'
    → 弹 AuthDialog.vue 认证重试对话框
    → 用户输入用户名/密码 + 可选勾选保存
    → [重试] → 携带 credentials 重新 invoke 原命令
      → 成功 → 关闭对话框 → 触发正常完成回调
      → 失败 → 保留对话框 → 更新错误信息
    → [取消操作] → 关闭对话框 → 原始操作视为已取消
```

### 8.3 IPC 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| Rust command panic | 前端 catch 后显示通用错误提示 |
| JSON 序列化失败 | 同上 |
| invoke 超时 | 前端超时提示 |
| **SVN 认证失败（v5 新增）** | **弹出认证重试对话框（§5.14），不直接显示错误 toast** |

### 8.4 持久化存储

使用 `tauri-plugin-store`。凭据**不**持久化到此 — 仅通过 `~/.subversion/auth/` 管理。
