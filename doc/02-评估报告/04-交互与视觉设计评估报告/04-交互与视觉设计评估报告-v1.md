## 1. 总体评价

交互与视觉设计 v2 文档在纯 UI/UX 设计层面质量很高——色彩体系、字体排版、间距系统、暗色模式等设计系统定义完整，页面线框图细致到每个交互状态和颜色值，Element Plus 组件使用约定明确。

但作为 AI 编程的输入文档，与业务功能设计文档之间存在 **多处架构分歧和功能遗漏**，必须在进入 Plan 阶段前统一。

**综合评级：B+**（良好，明确可改进）

## 2. 严重问题

以下问题必须修复，否则 AI 编码会出现方向性错误。

### 2.1 CSS 变量实现方式与开发规范直接冲突

**问题**：视觉设计文档 §5.2 给出的 CSS 变量双主题方案：

```css
:root { --bg-primary: #F8FAFC; }
.dark { --bg-primary: #0F172A; }
```

与 `01-Vue前端开发规范.md` 第 04 节 "核心规则" 冲突——规范明确规定 **仅使用 Tailwind CSS，禁止 `<style>` 标签或任何自定义 CSS 文件**。

CSS 变量要么写在 `<style>` 标签中，要么写在 `.css` 文件中——两者都被禁止。Tailwind CSS 4 的 `@theme` 指令虽然可以在 `style.css` 中定义 Token，但 `.dark .card { @apply bg-slate-800; }` 这类写法的确需要自定义 CSS。

> **影响范围**：全部页面的主题色、卡片色、输入框背景色等约 30 处使用了 CSS 变量的元素。
>
> **修复建议**：二选一——
> - 方案 A：仅使用 Tailwind `dark:` 变体（如 `class="dark:bg-slate-800"`），完全不用 CSS 变量
> - 方案 B：放宽规范，允许在 `src/style.css` 中通过 Tailwind `@theme` / `@layer` 定义设计 Token，仍禁止独立 `<style>` 标签

### 2.2 页面/路由架构与业务设计不一致

| 项目 | 业务设计 | 视觉设计 | 分歧说明 |
|------|---------|---------|---------|
| 欢迎页 vs 主视图 | `HomePage.vue` 负责"空状态条件渲染：有工作副本→文件列表，无→欢迎页" | `HomePage.vue`（欢迎页）+ `WorkspacePage.vue`（文件列表）两个独立页面 | **架构分歧** |
| Diff 视图 | `DiffPage.vue` 路由 `/workspace/diff?file=xxx` | 仅 `DiffViewer.vue` 组件（底部面板），无独立路由 | **路由缺失** |
| 冲突解决 | `ResolvePage.vue` 路由 `/workspace/resolve?file=xxx` | `ResolveDialog.vue` 对话框组件 | **形态不同** |

三个差异意味着 AI 按视觉设计编码后，文件结构和路由会与业务设计的路由定义矛盾。**必须先在两个设计文档之间统一**。

### 2.5 字体加载方案存在离线兼容和规范双重问题

**问题一：Google Fonts CDN `@import` 不适用于桌面离线场景**

视觉设计 §1.4 的字体加载方案：

```css
@import url('https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700&display=swap');
@import url('https://fonts.googleapis.com/css2?family=JetBrains+Mono:wght@400;500&display=swap');
```

Free-SVN 是 Tauri 桌面应用，用户可能离线操作 SVN。CDN 加载字体在无网络时失败，fallback 到系统字体（macOS 上的 Times/Helvetica），字号和行高不同将打乱整个排版。

> **影响范围**：所有页面的标题/正文/代码/状态栏文字——约 10 个排版层级全部依赖在线加载。

**问题二：`@import` 写法违反 Vue 开发规范**

`@import url(...)` 必须写在 `<style>` 标签或 `.css` 文件中，而 Vue 规范明确禁止两者。

**问题三：字号使用 px 而非 Tailwind 类，且与自身无障碍要求矛盾**

视觉设计 §1.4 全部字号使用 px（32px / 20px / 16px / 14px / 13px / 12px），但 §6 无障碍设计明确要求"使用 rem/em 而非 px"。自身要求冲突。

**问题四：缺少字体 fallback 栈**

仅指定字体名（Inter / JetBrains Mono），未提供 fallback。正确写法应为：

```css
font-family: 'Inter', system-ui, -apple-system, 'Segoe UI', Roboto, sans-serif;
font-family: 'JetBrains Mono', 'Fira Code', 'Cascadia Code', 'Source Code Pro', Consolas, monospace;
```

**问题五：缺少字号到 Tailwind 类的映射**

所有 px 值都可以映射到 Tailwind 内置类，但文档未做映射，导致 AI 需要自行换算：

| 约定层级 | 设计 px | Tailwind 类 |
|---------|---------|------------|
| Display | 32px | `text-3xl` |
| H1 | 20px | `text-xl` |
| H2 / Body | 16px / 14px | `text-base` / `text-sm` |
| Body Small / Mono / Label | 13px | `text-sm` |
| Mono Small / Tooltip | 12px | `text-xs` |

> **修复建议**：
> - 方案 A：将字体文件内嵌打包到 Tauri 资源中，通过 `@font-face` 本地加载（推荐）
> - 方案 B：使用 Tailwind `font-sans` / `font-mono` 类+系统字体栈，完全不需要额外加载
> - 所有字号改用 Tailwind 类（`text-sm`、`text-base` 等）
> - 补充完整的 fallback 字体栈

## 3. 中等问题

虽不影响编码启动，但影响交付质量。

### 3.1 设置按钮位置与业务设计不一致

| 文档 | ⚙️ 设置位置 |
|------|------------|
| 业务设计 §4.3 | **工具栏**中（与提交/更新/日志等并列），"始终可用" |
| 视觉设计 §2.1 | **TopBar 右侧**，与暗色切换并列 |

放 TopBar（始终可见区域）是更合理的 UX 决策，但这是 **设计决策变更**，应在视觉设计中明确标注"此位置已在 v2 中优化"。

### 3.2 Diff 面板高度参数未定义

面板从底部滑入的动画（200ms ease-in-out）已定义，但缺少：
- 默认展开高度 / 主列表与 diff 面板的分配比例（如 6:4？）
- 面板是否可拖拽调整高度
- 最小高度/最大高度约束

### 3.3 合并向导仅画了第一步

业务设计 §5.10 定义了 4 个合并步骤，视觉设计 §3.7 仅画了"步骤 1/3：选择来源"的线框图。步骤 2-4 的视觉形态缺失。

| 步骤 | 内容 | 视觉覆盖 |
|------|------|---------|
| 1 | 选择源 URL | ✅ 有线框图 |
| 2 | 选择版本范围 | ❌ 缺失 |
| 3 | 选择合并选项（忽略祖先、记录合并信息等） | ❌ 缺失 |
| 4 | 执行合并并显示结果（冲突文件列表） | ❌ 缺失 |

### 3.4 日志搜索功能缺少能力边界提示

业务设计 §5.5 明确说明 `svn log --search` **仅支持搜索提交信息，不支持按作者或版本号过滤**。

视觉设计的日志页面只有一个通用搜索框，用户会自然尝试输入作者名或版本号。缺少对搜索能力边界的 UI 提示（如搜索框 placeholder="搜索提交信息..." 或搜索模式下拉）。

### 3.5 批量还原按钮显示条件被简化

业务设计 §4.4 定义了批量还原按钮的显示条件：**"选中 ≥ 2 个还原类文件时显示"**。视觉设计 §3.2 仅画了"批量还原"按钮，未说明显示/隐藏逻辑。

### 3.6 提交面板选择状态传递规则未体现

业务设计 §5.4 定义的关键行为：
1. 进入提交面板时继承主视图选中状态
2. 提交面板内修改选择不影响主视图的选中状态
3. 提交完成后清除两个视图的选中状态

视觉设计仅展示布局，未体现这些状态传递约束。

| 仅此目录复选框 | ✅ | 对应 `--depth empty` | ✅ |
| 执行中进度 | ❌ 未出现在对话框中 | "后台执行 + 进度显示 → 完成后切换" | ❌ 缺失进度反馈 |

合并向导在步骤计数上存在两个文档之间的 **直接矛盾**：

| 项目 | 视觉设计 | 业务设计 |
|------|---------|---------|
| 步骤总数标注 | "步骤 **1/3**" | 4 个步骤（来源→版本范围→选项→执行） |
| 步骤 2-4 线框图 | ❌ 全部缺失 | ✅ 完全定义 |

### 3.10 冲突解决对话框细节差异

| 审查项 | 视觉设计 | 业务设计 | 问题 |
|-------|---------|---------|------|
| 冲突类型展示 | 仅显示 "text conflict" | 明确列出 "tree conflict / text conflict" | ❌ 缺少 tree conflict，这将导致树冲突出现时用户看到空白/错误信息 |
| 操作按钮顺序 | 外部工具提示 → 取消 → 标记已解决 | 全部使用我的 → 全部使用他们的 → 标记已解决 | ⚠️ 业务设计强调"建议操作"区域居中展示，视觉的按钮布局未体现 |

### 3.11 忽略对话框和通用确认对话框缺少线框图

视觉设计组件树（§7）中列出了 `IgnoreDialog.vue` 和 `ConfirmDialog.vue`，但在 §3.7 对话框设计中没有这两个对话框的线框图：

- **忽略对话框**：业务设计 §4.4 要求"弹出选择忽略模式"，需要用户选择忽略模式（文件名/扩展名/目录等）
- **通用确认对话框**：视觉设计 §4.3 只画了"确认还原"一种模板，但业务设计要求确认的场景包括：还原、删除、切换分支。是否共用同一模板未说明

### 3.12 设计遗漏：并发操作控制完全未考虑

这是一个 **架构级缺失**，而非细节问题。视觉设计未定义任何关于多个 SVN 操作同时触发的处理策略：

| 场景 | 风险 | 说明 |
|------|------|------|
| 提交中点击更新 | 两个 svn 进程同时操作同一工作副本，可能导致 `.svn` 元数据损坏 | 无操作队列或互斥锁设计 |
| 更新中点击切换分支 | 同上，svn update 未完成就执行 svn switch | 无进度态按钮禁用规则 |
| 检出中再次点击检出 | 同一目录同时检出两次 | 无操作去重 |
| 文件列表加载中再触发刷新 | 重复请求 | 无 loading 态防抖 |

> 真实开发中这会导致 **崩溃的 svn 工作副本**。AI 按目前的设计编码不会包含并发控制，上线后会遇到用户误操作导致工作副本损坏的问题。

### 3.13 设计遗漏：长操作缺少取消机制

业务设计 §10 定义了"长操作显示实时进度"，但视觉设计中的 ProgressOverlay（全屏进度遮罩）**没有提供取消/中止按钮**。

| 操作 | 典型耗时 | 无取消的影响 |
|------|---------|------------|
| 检出大型仓库 | 数分钟 | 用户只能强制关闭应用 |
| 更新大量文件 | 30 秒+ | 无退路 |
| 合并冲突 | 手动解决时 | ProgressOverlay 不应阻止其他操作 |
| 导出 | 大仓库 1 分钟+ | 进度只能等 |

> 缺少取消按钮在桌面应用中是不可接受的。ProgressOverlay 线框图中必须加上取消入口。

### 3.14 设计遗漏：输入校验体系完全缺失

视觉设计的所有对话框线框图中，**没有任何一个文本框标明了输入格式校验规则**：

| 对话框 | 需要校验的字段 | 校验规则 |
|-------|--------------|---------|
| 检出 | 仓库 URL | 必须是 svn:// https:// svn+ssh:// 格式 |
| 检出 | 检出到 | 路径必须存在且是目录 |
| 分支/标签 | 目标路径 | 必须在 branches/ 或 tags/ 下 |
| 更新到版本 | 目标版本 | 必须是正整数 |
| 导出 | 导出到 | 路径必须可写 |
| 设置页 | 差异命令模板 | 必须包含 `<file1>` `<file2>` 占位符 |

> 不定义校验规则，AI 编码时要么不校验（导致 crash），要么自行猜测规则（可能和业务设计不一致）。

### 3.15 设计遗漏：搜索/筛选空结果无界面设计

变更文件列表的搜索框（§3.2）和筛选下拉均有过滤功能，但 **筛选出零结果时没有设计空状态界面**。

与此对比，"工作副本无变更"的空状态有完整的线框图（§3.2 空状态），"欢迎页无历史"也有设计。唯独搜索结果为空被遗漏了。用户输入搜索词后如果无匹配，会看到一个空白表格，体验突兀。

### 3.16 设计遗漏：工具栏按钮溢出未处理

视觉设计 §2.1 的布局图中工具栏固定为 8 个按钮（提交/更新/日志/切换/分支标签/合并/清理/导出），但窗口最小宽度仅 800px。

| 场景 | 问题 |
|------|------|
| 窗口宽度 800px | 8 个按钮每个约 80px 宽 + 左右间距，已接近极限 |
| 窗口宽度 < 800px | 按钮必然溢出（换行或隐藏） |
| 加上国际化 | 日文/韩文按钮文字更长，可能更早溢出 |

> 未定义溢出策略（更多菜单 `···` / 折叠到二级菜单 / 响应式隐藏）。

### 4.5 操作反馈/Toast 规范逐项审查

视觉设计 §4.2 定义了 8 种操作反馈，经逐项比对业务设计和 SVN 操作清单，发现以下不足：

**现有 8 项的审查：**

| 触发 | 状态 | 问题 |
|------|------|------|
| 提交成功 | ✅ 完整 | |
| 提交失败 | ✅ 完整 | |
| 更新完成 | ✅ 完整 | |
| 还原 | ✅ 确认→toast 两段流程完整 | |
| 删除文件 | ⚠️ 有歧义 | 业务设计 §4.4 描述"直接删除新增文件（需要确认）"，是单文件操作。视觉设计描述为"删除文件"未注明是单个删除还是批量 |
| 检出中 | ⚠️ 缺失败态 | ProgressOverlay 在检出失败时应展示错误摘要和重试入口，未定义 |
| 网络超时 | ⚠️ 缺说明 | 业务设计 §2.2 处理策略是"超时退出并提示用户配置凭据缓存"，视觉设计写的是"重试按钮"，两者不一致。超时的根本原因可能是认证问题而非网络 |
| 右键菜单更新 | ✅ 完整 | |

**视觉设计未覆盖的操作反馈（按业务设计 §6.1 操作清单）：**

| 缺失的操作 | 建议反馈形式 |
|-----------|------------|
| 更新失败 | toast（red-500），显示 svn 错误信息 |
| 切换分支成功 | toast（green-500）+ 自动刷新列表 |
| 切换分支失败 | toast（red-500），保持当前分支 |
| 合并完成 | toast 或 ProgressOverlay 完成态，显示冲突文件数量 |
| 合并失败/冲突 | toast（warning）+ 进入冲突解决界面 |
| 清理完成 | toast（green-500） |
| 清理失败 | toast（red-500） |
| 还原成功 | toast（green-500） |
| 解锁成功 | toast（green-500） |
| 忽略成功 | toast（green-500） |
| 检出失败 | ProgressOverlay 错误态（red-500 摘要 + 重试按钮） |

> **影响**：共约 11 种操作反馈在视觉设计中缺失。AI 按视觉设计编码将缺少这些反馈的 UI 部分，需要 Plan 阶段补充。

### 4.6 窗口状态持久化未设计

Tauri 桌面应用的窗口位置、大小、是否最大化，在重启后不会自动恢复。常见做法是使用 `tauri-plugin-window-state` 或自行存储。视觉设计和业务设计都未提及这一点——但用户会期望关闭再打开后窗口回到之前的位置。

### 4.7 文件路径/文件名溢出截断未定义

工作副本路径可能很长（如 `/Users/me/work/projects/client/company-rebrand-2026`），视觉设计中的 TopBar（48px）和 StatusBar（28px）高度有限，当路径过长时：

- TopBar 中的路径如何截断（左侧截断/右侧截断/中间省略）？
- StatusBar 的三项信息在窗口窄时如何压缩？
- 文件列表中的文件路径列宽度多少？过长时如何处理？
- 操作按钮列在文件名很长时是否会挤压？

> 未定义截断策略（CSS `text-overflow: ellipsis` + 哪些字段可截断）。

### 4.8 多选交互（Shift+Click / Ctrl+Click）未定义

文件列表支持复选框多选（视觉设计 §3.2），但用户还会自然尝试：

- `Shift+Click` 范围选择（点击第一行，Shift+点击第 5 行 → 选中 1-5 行）
- `Ctrl+Click` / `Cmd+Click` 增删选择（保持已有选中，切换单个）

这些是表格多选的标准交互模式，但视觉设计既没有说明支持也没有明确禁止。AI 编码时可能不会实现，导致用户多选效率低。

### 4.9 文件列表排序交互未定义

变更文件列表和日志列表都使用表格展示数据。标准表格行为是点击列头（状态/文件/版本/日期等）按该列排序。视觉设计未定义：

- 是否支持点击表头排序？
- 排序指示器样式（升序/降序箭头）？
- 排序后的再次点击是切换方向还是取消排序？

### 4.10 外部工具不可用的降级处理未定义

设置页（§3.6）允许用户配置外部差异/合并工具（VS Code / Beyond Compare / Kaleidoscope），但：

- 用户选择了 VS Code 但系统未安装时，点击"差异"按钮会发生什么？
- 是弹出错误 toast、回退到内置 diff2html、还是显示配置错误提示？
- 外部工具命令执行失败时（如权限不足），如何反馈给用户？

### 4.11 空提交/无效提交防护未定义

提交面板中：

- 用户未选中任何文件时点击【提交】→ 应该按钮禁用还是点击后提示？
- 提交信息为空时是否可以提交？（SVN 允许空日志提交，但通常建议禁止）
- 提交信息超长时（超过 72 字符/行规范）是否有警告？

### 4.12 Diff 视图缺少行内搜索和换行控制

差异对比面板（§3.3）缺少以下常见功能：

- **搜索功能**：在 diff 内容中搜索关键词（Ctrl+F），高亮匹配行
- **换行切换**：代码行过长时，是否自动换行（word-wrap）还是不换行水平滚动
- **复制行内容**：点击行号或右键复制该行的内容

### 4.13 状态栏信息不可交互

视觉设计 §4.5 的状态栏显示"版本号 / 最新提交 / SVN 版本"，但开发者常见期望是：

- 点击版本号 → 复制 revision 号到剪贴板
- 点击最新提交 → 复制提交 hash 或直接打开日志视图
- 鼠标悬停 → 显示完整信息 tooltip

这些交互未定义，但如果 AI 不实现，用户会觉得功能不完整。

### 4.14 欢迎页最近打开列表缺少右键菜单

欢迎页的"最近打开的工作副本"列表只设计了"移除"按钮，但用户常见的其他操作未覆盖：

- 右键点击历史项 → 期望看到"打开所在目录"/"复制路径"/"移除"等菜单
- 是否可以拖拽历史项改变顺序？
- 历史记录上限多少条？（业务设计未限定）

### 4.15 无初始设置引导（First-Run Experience）

首次启动 Free-SVN 时，用户会面对欢迎页，但没有任何引导帮助用户：

- 没有提示用户设置默认检出目录
- 没有提示如何配置 SVN 认证（SSH key 或 auth 缓存）
- 没有"快速开始"或"入门"链接
- 设置页"SVN"分组为空，但用户首次需要配置默认检出目录

> 这会增加新用户的认知负担。至少应在欢迎页或首次设置中加一行引导文字。

## 5. AI 可编程性评估

### 5.1 各维度评分

| 维度 | 评分 | 依据 |
|------|------|------|
| 设计系统完整性 | A | 色彩、间距、圆角、阴影、图标、动画完整定义；但字体加载方案（CDN @import）有离线兼容和规范冲突问题 |
| 页面线框图详细度 | A- | ASCII 图细到每个状态行和颜色值；合并向导步骤 2-4 缺失、忽略/确认对话框无线框图 |
| 交互状态覆盖 | A | hover / disabled / selected / loading / empty 均有覆盖；但缺少并发操作/操作取消等运行时交互 |
| 与业务设计一致性 | C | 4 处架构分歧 §2.2、5 处命名分歧 §2.3、2 处功能遗漏 §2.4、对话框输入项差异多处 §3.7-3.11、合并向导步骤计数矛盾 §3.9 |
| 技术可实现性 | B- | CSS 变量与 Tailwind-only 规范矛盾未解决；Google Fonts CDN 不适用于离线桌面场景 |
| 操作反馈覆盖 | C | 视觉设计只覆盖了 8 种操作反馈，但全部 21+ 种 SVN 操作中约 11 种缺失（§4.5） |
| 运行时/安全设计覆盖 | D | 并发操作控制、操作取消、输入校验、工具栏溢出等完全未考虑（§3.12-3.16），是架构级缺失 |

### 5.2 AI 编码关键风险

1. **编码规范冲突**：AI 读到视觉设计中的 CSS 变量方案 → 再读到 Vue 规范禁止 `<style>` → 两条指令都能触发 AI 执行。必须以"规范优先"或"设计优先"固定其中之一。
2. **字体离线不可用**：Google Fonts CDN `@import` 在桌面离线场景会失败，AI 按设计使用 px + CDN 字体将导致排版异常。必须改为内嵌字体或系统字体栈 + Tailwind 类。
3. **文件结构二选一**：AI 无法同时遵循业务设计的 §9.1 文件结构和视觉设计的 §7 组件树——存在 5 处命名差异 3 处架构差异。Plan 阶段必须选定一套。
4. **遗漏功能的编码风险**：导出对话框和更新到版本对话框在视觉设计中缺失，AI 按视觉设计生成的 Plan 不会包含这两个组件。
5. **操作反馈代码缺失**：约 11 种 SVN 操作的操作反馈在视觉设计中未定义，按视觉设计生成的代码将缺少这些反馈的 UI 部分。
6. **并发安全风险**：视觉设计未定义并发操作控制，AI 直接编码将不会包含操作队列或互斥锁，用户误操作可能导致 svn 工作副本损坏。
7. **长操作无取消**：ProgressOverlay 无取消按钮，AI 不会生成取消逻辑，用户对长操作只能强制关闭应用。
8. **输入校验缺失**：所有对话框的输入校验规则均未定义，AI 要么不校验（导致运行时错误），要么自行猜测（可能和实际需求不一致）。

### 5.3 执行步骤可贯彻性

视觉设计在 **样式层面** 足够细致（颜色色值/字号/间距/动画参数均有数值），因此 Plan 中的样式描述可以直接引用视觉设计的 Token，无需 AI 自行判断。

但在 **行为逻辑层面**，状态传递规则（提交选择状态）、条件显示逻辑（批量还原按钮）、功能边界提示（日志搜索）等信息的缺失，需要 Plan 步骤补充细节。否则 AI 要么写错、要么做出与业务设计不一致的实现。

此外，还存在 **设计完全未覆盖的运行时领域**（§3.12-3.16、§4.6-4.15），包括并发控制、操作取消、输入校验、工具栏溢出、窗口持久化、文件溢出截断等。这些不是通过补充设计即可解决的——它们是设计阶段未被纳入 scope 的领域。Plan 阶段必须逐一决策是否在 MVP 中实现，不能留给 AI 自行判断。
