## 二、与设计师的评估分歧回应

v1 报告中有 4 项评估结论被设计师指出"不认同"，以下逐一回应：

### 2.1 §3.12 并发操作控制 → 接受异议，撤回此条

> 设计师意见：并发控制是后端架构/工程设计的范畴，视觉设计文档不需要定义。

**专家回应：接受。** 操作队列、Mutex 锁、去重逻辑确实是后端架构设计决策。从视觉文档角度，v3 已做了正确的事情——通过 ProgressOverlay + 取消按钮覆盖了"操作进行中的 UI 状态"这一视觉层面。并发控制的决策应移至业务功能设计文档或 Plan 阶段。

### 2.2 §3.14 输入校验体系 → 部分接受，仅保留"错误展示样式"维度

> 设计师意见：具体校验规则（URL 格式、版本号格式）是功能设计范畴，视觉设计做"界面怎么展示校验错误"——v3 已补充。

**专家回应：部分接受。** 校验规则本身（URL 需以 svn:// / https:// / svn+ssh:// 开头）确实是业务功能设计的职责，视觉设计不必定义规则。v3 在检出对话框补充了"格式非法时显示红色错误提示"、在版本号输入补充了"非数字时显示错误"，属于正确的视觉错误覆盖。**最终结论：此项通过。**

### 2.3 §4.6 窗口状态持久化 → 接受异议，撤回此条

> 设计师意见：这是实现细节，属于 Plan 阶段或后端实现考量的范畴。

**专家回应：接受。** 窗口位置/大小恢复是 `tauri-plugin-window-state` 一行配置可解决的实现细节，确实不需要在视觉设计文档中定义。**最终结论：此项撤回。**

### 2.4 §4.14 右键菜单 / §4.15 首次引导 → 调整结论为"后续迭代"

> 设计师意见：合理的改进建议，但属于 MVP 后续迭代的 UX 增强。v3 已添加基础的右键菜单（复制路径）。

**专家回应：调整结论。** v3 已为最近列表补充了"hover 显示移除和复制路径按钮（右键也可弹出菜单）"，基础功能到位。首次引导确实属于非关键 UX 增强。**最终结论：这两项从"遗漏"降为"可选的后续迭代优化"，不作为当前版本缺陷。**

## 三、v3 仍存在的剩余问题（3 项）

### 3.1 快捷键未区分平台差异

**位置**：§4.4 快捷键表 / §3.4 提交面板

**问题**：所有快捷键统一使用 `Ctrl+` 前缀，但 macOS 平台上多数操作的通行快捷键是 `Cmd+`（如 `Cmd+Enter` 提交、`Cmd+,` 设置）。未说明是否在 Tauri 中使用 `CmdOrCtrl` 修饰键处理平台差异。

**影响**：AI 直接编码会在 macOS 上产生反直觉的快捷键（如 `Ctrl+Enter` 提交而非 `Cmd+Enter`）。Tauri 中可使用 `CmdOrCtrl` 修饰键适配，但设计文档未指定，AI 可能按默认 `Ctrl+` 实现。

**建议**：在 §4.4 的表头或附注中注明"实现时使用 Tauri 的 `CmdOrCtrl` 修饰键"。

### 3.2 操作进行中的工具栏按钮禁用态未定义

**位置**：§4.1 工具栏 / §4.5 ProgressOverlay

**问题**：当 ProgressOverlay（全屏进度遮罩）正在展示时，工具栏按钮和文件列表操作按钮是否应全部禁用？ProgressOverlay 虽然阻挡了文件列表区域的点击，但键盘快捷键（如 `Ctrl+R` 刷新、`Ctrl+D` 差异）和工具栏按钮是否仍然可用没有定义。

**场景**：用户在提交过程中（ProgressOverlay 显示），下意识按 `Ctrl+R` 尝试刷新。文档未定义此场景的行为。

**建议**：在 §4.5 或 §4.1 中补充一行：操作进行中，所有工具栏按钮和文件行操作按钮禁用，键盘快捷键不生效（Esc 取消除外）。

### 3.3 设置页输入项缺少校验反馈

**位置**：§3.6 设置页面

**问题**：v3 的对话框设计几乎都有校验规则（检出 URL 格式、版本号数字、导出路径可写），但设置页的输入项缺少对应的校验反馈：

| 设置项 | 可能的无效输入 | 校验反馈 |
|-------|--------------|---------|
| 默认检出目录 | 路径不存在或不可写（如 `/nonexistent`） | ❌ 未定义 |
| 全局忽略模式 | 格式非法（如 `*.o *.`） | ❌ 未定义 |
| 差异命令模板 | 缺少 `<file1>` `<file2>` 占位符 | ❌ 未定义 |

与对话框不同，设置页的保存动作会持久化到 `tauri-plugin-store`，无效输入可能导致运行时错误。

**建议**：在 §3.6 交互说明中补充保存按钮的校验反馈策略——或在校验失败时显示字段级错误提示，或在保存按钮上方显示全局错误条。

## 四、设计盲点评估（v3 仍未涉及的领域）

以下问题使用与 v1 评估相同的"独立盲点扫描"方法论发现，属于设计文档无论哪个版本都尚未触及的新盲点：

### 4.1 文件列表在大数据量下的滚动条交互

**盲点**：§3.2 文件列表使用虚拟滚动支持大量数据，但：

- 表格底部有"已选 N 个文件"的底栏 + 底部的 StatusBar——当用户滚动到列表底部时，"全选底栏"是否固定在列表底部还是与内容一起滚动？
- 文件列表列的"操作"按钮列（差异/还原等）在水平滚动时是否固定在最右侧？不固定则用户需左右滚动找操作按钮。

这些都是真实场景中影响效率的问题，但未定义。

### 4.2 日志详情变更文件列表的交互不足

**盲点**：§3.5 日志视图展开详情后显示变更文件列表，每行有 [查看差异] 按钮。但：

- 如果一次提交变更了 30 个文件，变更文件列表是否应该也有虚拟滚动？
- 展开详情的文件行 hover 是否有交互态（如背景变色）？
- [查看差异] 点击后底部面板打开 diff——此时日志页面的展开详情是否保持打开态？

### 4.3 合并向导执行结果的冲突查看流程

**盲点**：§3.7 合并向导步骤 4 显示"⚠️ 合并完成，N 个文件存在冲突 → [查看冲突]"，但点击 [查看冲突] 后的跳转流程未定义：

- 是直接关闭向导并打开冲突解决对话框？
- 还是在向导内嵌入冲突文件列表？
- 如果用户点 [完成] 后如何继续处理冲突？

### 4.4 对话框在遮罩层上的动画和关闭策略不一致

**盲点**：v3 大部分对话框使用 `<el-dialog>` 组件（§8），其中"遮罩点击关闭"是全局约定。但：

- ProgressOverlay（全屏遮罩）是否可以点击遮罩关闭？还是必须通过 [取消] 按钮？
- 检出对话框嵌入进度条后，遮罩点击是否允许关闭？如果操作中的进度条被遮罩关闭销毁，后端操作是否继续？
- 合并向导的多步骤流程中，遮罩点击关闭是否会丢失已填写的步骤数据？

建议在 §4.5 或 §8 中补充"操作进行中的对话框遮罩点击行为"说明。

### 4.5 i18n 的布局适配缺省

**盲点**：§3.6 设置页配置了 4 种语言，但无任何对文字长度变化的视觉适配：

- 日文的工具栏按钮（"コミット"、"更新"、"ログ"）比中文宽 30-50%，8 个按钮会更早溢出
- 英文按钮溢出行为可能不同（"Create Branch/Tag" 比 "分支/标签" 长一倍）
- 不同语言的 toast 文案长度不同，toast 宽度是否自适应？

建议在 §2.2 工具栏溢出处附注"不同语言下的溢出阈值可能不同"。
