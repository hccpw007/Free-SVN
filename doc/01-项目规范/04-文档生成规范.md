# 文档生成规范

## 01-目录结构

所有文档存放在 `doc/` 目录下，按类别分目录、以序号开头：

| 目录 | 用途 |
|------|------|
| `doc/01-项目规范/` | 规范、规则 |
| `doc/02-设计文档/` | 设计文档 |
| `doc/03-执行计划/` | 开发计划 |
| `doc/99-评估报告/` | 评估报告 |

文件名以 `01-` 序号开头，使用中文命名。

## 02-设计文档 & 评估报告版本命名

### 核心原则

所有设计文档和评估报告从 v1 开始就放在子文件夹中，后续修改一律**新增版本文件，不修改旧版本**。

### 规则

```
doc/02-设计文档/
├── 01-业务功能设计/
│   ├── 01-业务功能设计-v1.md   # 初始版本
│   ├── 01-业务功能设计-v2.md   # 修改时新增，不修改 v1
│   └── 01-业务功能设计-v3.md   # 再次修改继续新增（版本号连续递增）
```

1. **子文件夹带序号** — 如 `01-业务功能设计/`
2. **统一基底文件名** — 文件夹内所有文件共用基底名（含序号），如 `01-业务功能设计-v1.md`
3. **版本号后缀** — 依次 `-v1`、`-v2`、`-v3`（v1 不省略）
4. **首次即子文件夹** — 禁止平级单文件（如 `01-业务功能设计.md`）
5. **新增不修改** — 不得修改已存在版本文件
6. **版本号连续递增** — 目录最大版本号 +1

### 评估报告独立文件夹

每个评估主题使用独立子文件夹，不得混放：

```
doc/99-评估报告/
├── 05-阶段一执行步骤评估报告/    # 主题 A
└── 06-阶段二执行步骤评估报告/    # 主题 B
```

## 03-执行计划文件命名

```
doc/03-执行计划/
└── 01-整体开发计划/
    ├── 01-开发计划/              # 整体计划文档（带版本号）
    │   ├── 01-开发计划-v1.md
    │   └── 01-开发计划-v2.md
    └── 02-执行步骤/              # 具体执行步骤
        ├── phase-1/
        │   ├── 1.1-中文描述.md   # {阶段}.{步骤}-{中文描述}
        │   └── 1.2-中文描述.md
        └── phase-2/
```

规则：
1. `doc/03-执行计划/` 下统一为 `01-整体开发计划/`
2. 步骤按阶段分目录 `phase-1/`、`phase-2/`（不使用中文"阶段一"）
3. 步骤文件命名：`{阶段序号}.{步骤序号}-{中文描述}.md`
4. 整体计划文档带 `-v1`、`-v2` 版本号后缀

## 04-文档角色分工

### 设计文档（doc/02-设计文档/）

定义"做什么"——每个功能定义必须包含充分的执行信息（输入/输出/行为/边界等）。**仅用于生成执行步骤，不直接用于编码**。

### 执行步骤文件（doc/03-执行计划/01-整体开发计划/02-执行步骤/phase-N/）

定义"怎么做"——将设计文档细化为 AI 能具体编码的步骤文件。

**引用规则：**
- AI 编码以执行步骤文件为准，**而非设计文档**
- 执行步骤与设计文档不一致时，**以执行步骤为准**
- 执行步骤细节不足无法编码时，**不回读设计文档**，直接向用户提问
- 写步骤时必须逐条对照设计文档，确认覆盖了全部输入/输出/行为/边界

## 05-步骤文件格式

每个执行步骤文件必须包含以下三个字段：

| 字段 | 要求 |
|------|------|
| 涉及文件 | 列出所有新建/修改的文件路径，标注操作类型 |
| 操作细节 | 具体代码结构、函数签名、修改位置、逻辑描述。**禁止仅写需求目标，禁止"...其余xx"省略写法** |
| 验收标准 | 可验证的完成条件（如"登录表单包含账号/密码输入框和登录按钮"） |
