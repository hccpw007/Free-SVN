# 多语言代码注释规范

覆盖前端（TypeScript / Vue / React）和后端（Rust / Java）。

## 01-总则

1. **功能变更同步更新注释** — 过期注释比没注释更危险
2. **不言自明的代码不注释** — 变量名已表达用途、循环索引、初始化值自说明时省略
3. **注释"为什么"而非"是什么"** — 代码表达"做了什么"，注释表达"为什么要这么做"
4. **不注释掉代码** — 不需要直接删，git 可找回

## 02-核心规则（语言无关）

| 元素 | 注释位置 | 说明 |
|------|----------|------|
| Class / Struct / Interface | 上方 | `// 描述`，较长可用 `/* */` |
| 函数 / 方法 | 上方 | 有参数/返回值时说明用途 |
| 变量 / 常量 | 上方 | 复杂逻辑/标志位/魔法数字/配置值**必须**注释 |
| `if` | 正上方 | `// 条件` |
| `else` / `else if` | 右侧同行 | `} else { // 条件` |
| `switch` / `match` | 正上方 | `// 条件` |
| `for` / `while` | 正上方 | `// 条件` |
| `try` | 正上方 | `// 描述` |
| `catch` | 右侧同行 | `} catch { // 条件` |
| Rust `Err` 分支 | 右侧同行 | `Err(e) => { // 原因` |
| 枚举变体 | 上方或右侧 | `// 描述` |
| Vue Template 区块 | 上方 | `<!-- 功能块描述 -->` |
| `v-if` / `v-else` | 正上方 / 右侧同行 | `<!-- 条件 -->` |
| 按钮 / 事件处理 | 上方 | `<!-- 点击后的操作 -->` |
| React JSX | `{/* */}` | 规则同上 |

注释前空行规则：`//` 上一行如果是同级代码（非注释/空白），上方必须空行。连续多条注释只需第一条前空行。

## 03-各语言注释格式

| 场景 | TypeScript | Rust | Java |
|------|-----------|------|------|
| 行内 | `//` | `//` | `//` |
| 公开函数 | `//` 或 `/** */` | `///` | `/** */` |
| 私有函数 | `//` | `//` | `//` |
| Class / Struct | `//` 或 `/** */` | `///` | `/** */` |
| 模块/文件 | `//` | `//!` | `/** */` |
| 接口 / Trait | `//` 或 `/** */` | `///` | `/** */` |
| 参数说明 | `@param` 或行内 | `/// # 参数` | `@param` |
| 返回值说明 | `@returns` 或行内 | `/// # 返回` | `@return` |

### TypeScript 补充

- 默认 `//`，参数较多（≥3）时允许 `/** */` doc block
- Vue 每个 `.vue` 文件 `<script setup>` 顶部注释组件功能
- Vue `<template>` 中每个功能块上方用 `<!-- -->` 注释
- React JSX 中用 `{/* */}`
- Pinia store 定义上方加注释
- 所有 `ref()` / `reactive()` / `useState()` 上方加注释

### Rust 补充

- 公开函数/struct/enum/trait: `///` doc comment
- 模块级文档: `//!`（mod.rs 或文件顶部）
- 私有函数/变量/控制流: `//` 行内注释
- 结构体字段: 每个字段用 `///` 单独注释
- 可省略：getter 方法、trait 实现的显而易见方法、自文档化测试函数名

### Java 补充

- 公开 API: `/** */` Javadoc（含 `@param`、`@return`、`@throws`）
- 私有方法: `//` 行内注释
- 可省略：标准 getter/setter、逻辑简单的 `@Override` 方法、自文档化 `@Test` 方法

## 04-常见反模式

| 反模式 | 正解 |
|--------|------|
| 注释与代码重复（噪声） | 省略，不写废话 |
| `if` 注释在同行 / `else` 注释在上方 | 修正位置 |
| 控制流无注释 | 每个控制流加注释说明条件 |
| 注释掉的代码 | 直接删，git 可找回 |
| 修改代码不更新注释 | 修改时同步更新 |
| Vue Template 无 `<!-- -->` 注释 | 按 05 细则逐级检查补充 |

## 05-Vue Template 注释细则

> **为什么需要独立细则？** Vue Template 是最容易因"看起来简单"而跳过注释的地方，但 template 中的逻辑分支和交互元素是用户直接接触的 UI 表面，缺少注释会让后续维护者难以理解功能边界和交互意图。AI 编码时必须按本章节逐级检查，不能因"组件只有几行"而跳过。

### 5.1 注释分区层级

Vue Template 从外到内分为 **5 级**，每级都有对应的注释规则。AI 编码时必须从 L1 开始逐级检查，确保每级的每个"副本"（同一级出现多次的元素）都有注释：

| 层级 | 元素 | 注释规则 |
|------|------|----------|
| **L1-区块** | `div`、`<template>`、`<Transition>` 等顶层分块容器 | `<!-- 功能块描述 -->` 正上方 |
| **L2-控制流** | `v-if` / `v-else-if` / `v-else` / `v-for` | `<!-- 条件 -->` 正上方 |
| **L3-UI组件** | `el-table`、`el-dialog`、`el-form` 等 Element Plus 组件 | `<!-- 用途 -->` 正上方 |
| **L4-交互元素** | `<button>`、链接、输入框 | `<!-- 点击/输入后的操作 -->` 正上方 |
| **L5-文本/图标** | `<span>`、`<p>`、`<svg>` 等纯展示元素 | **省略**，不言自明时跳过 |

> **强制规则：** AI 编码时必须遍历 template，对每个 L1-L4 元素检查是否缺少注释。L5 可省略。

### 5.2 各层级详细规则

#### L1-区块注释

Template 中每个顶层视觉分区上方必须加 `<!-- -->` 注释。区块按功能划分，常见场景：

| 场景 | 写法 |
|------|------|
| 页面顶部导航 | `<!-- 顶部导航条 -->` |
| 左右布局容器 | `<!-- 左侧 Tab 列表 -->` / `<!-- 右侧设置内容 -->` |
| 表单分组 | `<!-- 基本信息 -->` / `<!-- 高级选项 -->` |
| 模态框区域 | `<!-- 弹窗主体内容 -->` |
| 步骤容器 | `<!-- 步骤 1：选择来源 -->` |

**正确示例：**
```vue
<!-- 顶部导航条 -->
<div class="h-12 px-4 flex items-center gap-2 border-b">
  ...

<!-- 左侧竖排 Tab -->
<div class="w-44 shrink-0">
  ...
```

**错误示例（缺少区块注释）：**
```vue
<div class="h-12 px-4 flex items-center gap-2 border-b">
  ...
<div class="w-44 shrink-0">
  ...
```

#### L2-控制流注释

`v-if` / `v-else-if` / `v-else` / `v-for` 正上方必须加 `<!-- 条件 -->` 注释：

| 控制流 | 注释位置 | 示例 |
|--------|----------|------|
| `v-if="condition"` | 正上方 | `<!-- 有工作副本时显示路径 -->` |
| `v-else` | 正上方 | `<!-- 无工作副本时显示应用名 -->` |
| `v-for="item in list"` | 正上方 | `<!-- 遍历分支列表 -->` |

**正确示例：**
```vue
<!-- 无工作副本时：显示应用名 -->
<template v-else>
  <span class="text-sm font-semibold">Free-SVN</span>
</template>

<!-- 搜索无匹配结果 -->
<div v-if="isSearchEmpty" class="flex items-center justify-center">
  <p>{{ t('workspace.noSearchResult') }}</p>
</div>
```

**错误示例（控制流无注释）：**
```vue
<div v-if="isSearchEmpty">
  ...
</div>
```

#### L3-UI 组件注释

Element Plus 等 UI 组件上方必须加 `<!-- 用途 -->` 注释，说明该组件在整个页面中的功能角色：

```vue
<!-- 版本号输入 -->
<el-input-number v-model="revision" :min="1" size="small" />

<!-- 文件列表表格 -->
<el-table :data="filteredFiles" size="small" style="width:100%" row-key="path">
  ...
</el-table>

<!-- 展开列：显示日志详情 -->
<el-table-column type="expand" width="30">
  ...
</el-table-column>
```

#### L4-交互元素注释

`<button>`、`<a>` 等可点击元素上方加 `<!-- 点击后的动作 -->`：

```vue
<!-- 开始合并操作 -->
<el-button type="primary" :loading="isMerging" @click="handleMerge">
  {{ t('workspace.executeMerge') }}
</el-button>

<!-- 取消并关闭对话框 -->
<el-button @click="emit('close')">{{ t('common.cancel') }}</el-button>
```

### 5.3 模板结构注释模板——直接套用

#### 页面模板（含多个视觉区块）

```vue
<template>
  <!-- 顶部导航条 -->
  <div class="...">...</div>

  <!-- 操作栏 -->
  <div class="...">...</div>

  <!-- 主内容区 -->
  <div class="...">
    <!-- 加载状态 -->
    <div v-if="isLoading">...</div>
    <!-- 空状态 -->
    <div v-else-if="isEmpty">...</div>
    <!-- 数据展示 -->
    <div v-else>...</div>
  </div>

  <!-- 底栏 -->
  <div class="...">...</div>
</template>
```

#### 对话框模板

```vue
<template>
  <!-- 弹窗主体 -->
  <el-dialog ...>
    <!-- 表单区域 -->
    <div class="flex flex-col gap-4">
      <!-- URL 输入 -->
      <div>...</div>
      <!-- 路径选择 -->
      <div>...</div>
    </div>

    <!-- 底部按钮 -->
    <template #footer>
      <el-button>{{ t('common.cancel') }}</el-button>
      <el-button type="primary">{{ t('common.confirm') }}</el-button>
    </template>
  </el-dialog>
</template>
```

### 5.4 注释豁免规则

以下场景**可以省略** template 注释：

| 场景 | 原因 |
|------|------|
| `<RouterView />` 等单一顶级组件 | 结构自明 |
| 不超过 3 行的单区块 `<div>` | 结构过于简单 |
| 纯图标/文本展示（无点击、无逻辑） | 不言自明 |
| 纯 CSS 布局容器（无功能语义） | 属于样式而非功能 |

### 5.5 AI 编码检查清单

每次编码或修改 Vue 组件后，必须逐条检查：

```
□ L1-区块：每个顶层视觉分区上方有 <!-- --> 注释？
□ L2-控制流：每个 v-if / v-else / v-for 上方有 <!-- 条件 -->？
□ L3-UI组件：每个 el-* 组件上方有 <!-- 用途 -->？
□ L4-交互元素：每个 <button> 上方有 <!-- 动作 -->？
□ 注释位置：在正上方，而非同行或下方？
□ 注释内容：说明"为什么"而非"是什么"？
□ 豁免判断：L5 层级是否真的不言自明？L5 以上是否真的需要豁免？
```
