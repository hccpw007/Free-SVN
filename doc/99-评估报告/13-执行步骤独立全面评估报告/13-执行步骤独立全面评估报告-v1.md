## 第四部分：发现的问题清单

### P1 问题（影响业务功能完整性和运行正确性，编码前必须修复）

#### P1-01：`check_network` 未注册为 Tauri Command，前端 services/svn.ts 无法调用

**问题描述：**
步骤 7.2（services/svn.ts 完整封装）定义了一个 `checkNetwork()` 方法，该方法通过 `wrappedInvoke` 调用后端 `check_network` 命令。但后端 **从未创建** `commands/check_network.rs` 或将该命令注册到 `generate_handler!` 中。`check_network` 函数仅在 `svn/executor.rs` 中存在，只能在 Rust 内部调用。

**受影响范围：**
- `src/services/svn.ts` — `checkNetwork()` 方法会运行时崩溃
- `src/composables/useNetworkStatus.ts` — 依赖 `checkNetwork`，会被阻断
- `workspaceStore.isOffline` 状态永远无法更新

**建议修复：**
在 `commands/` 目录下创建 `check_network.rs` Tauri Command，包装 `svn::executor::check_network()`，并注册到 `generate_handler!`。

**或**：将前端网络检测改为前端独立的 HTTP HEAD/fetch 探活，不依赖后端 command。

---

#### P1-02：`load_settings` / `save_settings` 未注册为 Tauri Command

**问题描述：**
步骤 4.2C（settings Store）的 `load()` 和 `save()` 方法调用 `invoke<AppSettings>('load_settings')` 和 `invoke('save_settings', ...)`。但后端 `config/store.rs` 中的 `load_settings()` 和 `save_settings()` 是普通 Rust 函数，**没有 `#[tauri::command]` 注解**，也没有在 `generate_handler!` 中注册。

**受影响范围：**
- `src/stores/settings.ts` — `load()` 和 `save()` 方法会在运行时崩溃
- 所有依赖 settingsStore 的组件（SettingsPage、App.vue 语言切换联动等）
- 设置页面完全不可用

**建议修复：**
在 `commands/` 下创建 `settings.rs`，添加 `load_settings` 和 `save_settings` 两个 Tauri Command，包装 `config::store::load_settings()` 和 `config::store::save_settings()`，注册到 `generate_handler!`。

---

#### P1-03：`@tauri-apps/plugin-fs` 未安装但 Step 6.1 使用 `exists()`

**问题描述：**
步骤 6.1 HomePage.vue 的 `onMounted` 中动态导入 `@tauri-apps/plugin-fs` 并调用 `exists()` 进行工作副本路径有效性检测。但步骤 1.2 的依赖安装命令中没有包含 `@tauri-apps/plugin-fs`。

**同时问题：** Rust 侧也未注册 `tauri_plugin_fs`。

**建议修复：**
- 在步骤 1.2 的 cnpm install 中添加 `@tauri-apps/plugin-fs`
- 在 Cargo.toml 中添加 `tauri-plugin-fs`
- 在 lib.rs 中注册 `tauri_plugin_fs::init()`

### P2 问题（影响功能完整性或编码效率，建议编码前修复）

#### P2-01：generate_handler 命令数量不一致

**问题描述：**
- 步骤 3.4 标题和描述说"25 个 command"，实际代码生成 25 个（正确）
- 步骤 7.3 描述说"generate_handler! 共 26 个，含 check_network 和 property_ops"
- 但 `check_network` 不是 Tauri Command（是普通函数），所以不应在 generate_handler 中
- 开发计划 v2 也说"24 种操作 + 2 日志 = 26"，但实际列表只有 25 项

**建议：**
统一描述为"25 个 Tauri Command"，删除所有提到 26 的数字。在 7.3 中移除 `check_network` 作为 command 的描述（或单独创建 check_network command 后再计为 26）。

#### P2-02：7.4 Linux Nautilus 脚本多文件路径参数传递问题

**问题描述：**
步骤 7.4 linux.rs 的 12 项 SVN 操作脚本使用：
```bash
#"{}" --svn-cmd {} "$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS"
```

`$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS` 是换行分隔的多路径字符串，直接作为单参数传递会导致所有文件路径被当作一个参数。只有 checkout 脚本使用了正确的 `while IFS= read -r` 逐行读取方式。

**建议修复：**
所有脚本改为 `while IFS= read -r` 逐行读取模式。

---

#### P2-03：LogPage/CommitPage Store 调用参数类型不匹配

**问题描述：**
步骤 6.5 LogPage 调用 `svnStore.getLog({ path, limit, search })` 传入**单个对象**，但步骤 4.2D svn store 的定义是：
```ts
async function getLog(path: string, limit?: number, revision?: number, search?: string)
```
需要**四个独立参数**。这将导致 TypeScript 类型错误。

**同类问题：**
步骤 6.4 CommitPage 调用 `svnStore.commit({ paths, message, keepLocks })`，但 4.2D 定义是：
```ts
async function commit(paths: string[], message: string, keepLocks?: boolean)
```
需三个独立参数。两个调用方式均不匹配。

**建议修复：**
统一调用方式，建议 svn store 的方法统一接受对象参数（更具可读性且不易出错）。

#### P2-04：DiffViewer 的 `openDiff` 中 `r.revision1/r.revision2` 不存在

**问题描述：**
步骤 6.3 DiffViewer 的 `openDiff()` 方法中：
```ts
diffFile.value = { path, content: r.content, isBinary: r.isBinary, rev1: r.revision1, rev2: r.revision2 }
```

但 `DiffResult` 接口（步骤 4.1B）和 Rust `DiffResult`（步骤 3.1B）均不包含 `revision1` / `revision2` 字段。这将导致 `rev1` 和 `rev2` 始终为 `undefined`，面板标题显示为"路径 (undefined → undefined)"。

**建议修复：**
在 `openDiff` 中保存调用时传入的版本号参数，或扩展 `DiffResult` 增加 `revision1` / `revision2` 字段。

#### P2-05：Settings 前端 diff/merge 模板字段无法持久化

**问题描述：**
步骤 4.2C settings Store 定义了 `diffCommandTemplate` 和 `mergeCommandTemplate` 两个 ref，UI 可编辑。但 `save()` 方法发送给后端的 settings 对象**不包含这两个字段**（Rust Settings 结构体也没有对应字段）。用户编辑后保存，模板内容丢失。

**建议修复：**
- Rust 侧 Settings struct 增加 `diff_command_template` 和 `merge_command_template` 字段
- 前端 save 时发送这两个字段

#### P2-06：SwitchDialog 调用的 `svnStore.listBranches()` 未实现

**问题描述：**
步骤 6.7A SwitchDialog 在 `onMounted` 中调用 `svnStore.listBranches()` 获取分支列表，但：
- 该函数在 svn store（步骤 4.2D）中未定义
- 后端无对应的 `list_branches` Tauri Command
- 开发计划中提到使用 `svn ls ^/branches`，但从未真正实现

**影响：**
切换对话框的分支下拉列表为空，用户必须手动输入完整 URL。

**建议修复：**
后端新增 `list_branches` command，调用 `svn ls ^/branches --xml`，前端 svn store 增加对应方法。
