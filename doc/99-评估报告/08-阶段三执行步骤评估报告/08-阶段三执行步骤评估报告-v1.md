## §3 逻辑缺陷

### 🔴 P1 — `update_workspace` 锁序缺陷（3.2B）

```rust
state.try_lock()?;        // ← 先获取写锁

let info_xml = run_svn(&["info", "--xml"], &path).await?;  // 可能失败
let info = parse_info(&info_xml)?;                          // 可能失败
check_network(&info.url).await?;                            // 可能失败
// 以上任何一个 ? 提前返回 → state.unlock() 永不执行 → 队列永久死锁
```

**问题**：`try_lock()` 获取写锁后，后续连续三处可能失败的调用都没有保护。任一处 `?` 返回都会导致 `unlock()` 不被执行，整个 SvnQueue 永久死锁。

**对比 checkout_repo（正确顺序）：**
```rust
check_network(&params.url).await?;   // 先检测网络
state.try_lock()?;                    // 后获取锁
```

**修复建议**：将 `try_lock()` 移到所有前置检查之后：

```rust
// 先获取 URL 和检测网络（只读操作不需要写锁）
let info_xml = run_svn(&["info", "--xml"], &params.path).await?;
let info = parse_info(&info_xml)?;
check_network(&info.url).await?;
state.try_lock()?;  // 所有检查通过后才获取写锁
```

## §4 已知设计取舍（已确认可接受）

### 4.1 长操作无逐行进度推送

仅使用了 `operation:started/completed/error`，未使用 `operation:progress`。步骤已说明"完整逐行进度解析留待后续增强"。✅

### 4.2 长操作取消依赖 CANCELLED 标志

`run_svn_sync()` 阶段二已决定不保存 child 句柄到 `CURRENT_CHILD`。`cancel_operation` 的 `kill_current_process()` 无法找到子进程。取消主要通过超时机制。✅ 已知取舍，后续阶段优化。

### 4.3 generate_handler 25 个 command

验收标准写"26 个 command"，核对后端模块函数后实际为 25 个。25 是正确数量。✅ 文字笔误，不影响编译。
