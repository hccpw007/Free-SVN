## §1 业务需求覆盖度

### P0 缺项（必须修复）

#### 1.1 SVN 认证交互策略未覆盖

| 来源 | 业务设计 v4 §2.2 |
|------|-----------------|
| 内容 | svn CLI 可能触发交互式认证提示（用户名/密码），处理策略包括：设置超时、不处理交互式认证（建议 SSH key 或 `~/.subversion/auth/` 缓存）、超时/认证失败时区分"网络问题"和"认证问题"并给出配置指引 |
| 现状 | 开发计划仅提及全局 60s 超时（技术参数清单），未提及认证提示检测、错误类型区分、首次认证引导 |
| 影响 | 用户遇到认证问题时只能看到"SVN 命令执行超时"，无法区分是网络问题还是认证问题，也无法获得配置指引 |
| 建议修复 | 在阶段二（2.2 executor.rs）中增加：① 检测 svn stderr 中的认证提示模式（"Authentication realm" / "Password" / "Username"）；② 区分 `SVN_AUTH_FAILED` 错误码；③ 前端对应 i18n 翻译和引导文案 |

#### 1.2 文件行右键菜单未实现

| 来源 | 业务设计 v4 §4.4 |
|------|-----------------|
| 内容 | 变更文件列表每行支持右键菜单：复制文件路径、复制相对路径、在文件管理器中显示、用外部编辑器打开 |
| 现状 | 开发计划 6A.3（FileListTable）仅提到底栏操作，未明确实现文件行右键菜单 |
| 影响 | 用户无法通过右键快速获取文件路径或使用外部编辑器打开文件，影响日常操作效率 |
| 建议修复 | 在阶段 6A（FileListTable 实现步骤）中增加右键菜单实现：`@contextmenu.prevent` + 自定义弹出菜单（四项），使用 navigator.clipboard 复制路径 |

#### 1.3 设置页"导出日志"按钮入口缺失

| 来源 | 业务设计 v4 §6.3 |
|------|-----------------|
| 内容 | 设置页 → 关于 → [导出日志] 按钮，导出包含：日志文件 + 应用版本 + 系统信息 + 内置 svn 版本 |
| 现状 | 开发计划 3D.2 实现了后端 `export_logs`，6E.2 关于区提到了"版本号+SVN版本+许可证"，但未提"导出日志"按钮 |
| 影响 | 后端功能已规划但前端无入口，用户无法在 UI 中导出诊断信息 |
| 建议修复 | 在 6E.2 关于区增加"导出日志"按钮，调用 `svnStore.exportLogs()` |

### P1 缺项（建议修复）

#### 1.4 提交按钮启用条件与业务设计不一致

| 来源 | 业务设计 v4 §5.4 vs 交互设计 v4 §3.4 |
|------|--------------------------------------|
| 内容 | **业务设计**：至少选中一个文件且提交日志非空时提交按钮才启用。**交互设计**：未选中任何文件时禁用；提交信息为空时弹出确认框 |
| 现状 | 开发计划 6B.2 按交互设计实现（空提交弹出确认对话框），与业务设计存在分歧 |
| 影响 | 部分用户可能误提交空信息（虽然被确认框拦截，但与业务设计的"非空才可提交"原则不符） |
| 建议 | 统一原则：以交互设计为准（弹确认框优于硬禁用），因用户可能只想用最近使用的日志。建议在 CLAUDE.md 或开发规范中明确此决策依据 |

#### 1.5 `svn status` 缓存回退策略未定义

| 来源 | 业务设计 v4 §2.4 |
|------|-----------------|
| 内容 | "status 等只读操作若在写操作执行期间被调用，返回上次缓存的列表或等待写操作完成（前端优先显示缓存，写操作完成后自动刷新）" |
| 现状 | 开发计划未提及后端缓存策略或前端缓存显示，仅提到写操作锁（2.4 queue.rs）和自动刷新策略 |
| 影响 | 写操作期间用户无法查看文件状态，场景体验断档 |
| 建议 | 在 2.4 或 4.2 fileListStore 中增加缓存策略：store 中保留上次 `files` 快照，写操作期间 `refresh()` 时先返回缓存再轮询 |

#### 1.6 工作副本路径失效处理未提及

| 来源 | 业务设计 v4 §4.4 |
|------|-----------------|
| 内容 | "如果检测到工作副本路径不存在或被删除/移动，显示提示信息并切换到欢迎页（保留该路径在历史记录中供用户修正）" |
| 现状 | 开发计划各阶段均未提及此边界处理 |
| 影响 | 工作副本被外部删除后应用可能崩溃或卡在无效状态 |
| 建议 | 在阶段 4（workspace store）或阶段 6A（HomePage）增加路径有效性检测逻辑 |

## §5 正面认定

以下方面设计合理、覆盖充分，予以正面认定：

### 5.1 阶段划分与并行策略

前后端可并行开发的策略（阶段二↔阶段四、阶段三↔阶段五）符合 Tauri 项目特点，可有效缩短开发周期。

### 5.2 核心功能链路完整

检出 → 变更列表 → 差异对比 → 提交 → 日志 的核心链路所有组件和后端命令均已覆盖，依赖关系清晰。

### 5.3 技术参数清单

30 项技术参数全部与设计文档对齐，无错漏。

### 5.4 全部操作清单

24 项 SVN 操作的输入/输出/优先级全部与业务设计 §6.1 表对齐，无遗漏。

### 5.5 并发控制与取消机制

写操作互斥、只读操作并行、长操作可取消三档策略与业务设计 §2.4/§2.5 完全一致。取消的完整流程（设标志 → kill → cleanup → 重置 → 通知）明确。

### 5.6 数据模型与错误码

12 个错误码变体与后端规范 §03 表逐项对齐，无遗漏。

### 5.7 前端组件树

所有 9 个对话框、7 个 SVN 组件、3 个通用组件、4 个页面组件均与交互设计 §7 组件树一致。

### 5.8 国际化策略

4 语言框架（中/英/日/韩）预留 + 后端错误码 key + 前端 i18n 的策略正确，MVP 聚焦中/英合理。

### 5.9 工具栏按钮启用/禁用逻辑

与交互设计 §4.1 表逐项映射，禁用条件 + tooltip 文案完整。

### 5.10 空状态三态区分

欢迎页 / 工作副本无变更 / 搜索无结果 三种空状态分别处理，符合交互设计 §3.2 要求。
