## 三、设计师反馈意见评估（§11 — 未采纳项及理由）

### 3.1 性能退化监控

> 理由：CI/QA 职责，非设计文档内容

**判定：✅ 合理**

性能退化监控确实需要持续集成环境和基准测试工具保障，不是功能设计文档需要定义的内容。但有个值得补充的点：v3 §10 规定了"10000 文件 ≤ 1 秒"和"10000+ 使用虚拟滚动"，这两者是对的。问题是"50000 文件怎么办"——这个边界在 v3 中仍然未讨论。建议在 §10 性能行中补充："50000+ 文件场景：首次 svn status 扫描展示「正在扫描…」占位状态，文件列表采用增量加载"。

### 3.2 自动更新机制

> 理由：依赖发版策略和签名流程，Plan 阶段决定

**判定：✅ 合理**

自动更新涉及：更新服务器、代码签名证书、更新 UI 设计。这些确实是构建/发版阶段的基础设施决策。

### 3.3 测试策略

> 理由：开发规范/编码阶段的事

**判定：✅ 合理**

CLAUDE.md 的编码规范已规定测试。设计文档定义"做什么"，测试策略是"怎么做"。分离合理。

### 3.4 辅助功能/无障碍

> 理由：开发者工具 MVP 过度设计

**判定：⚠️ 部分合理，但有风险**

设计师说 MVP 过度设计——对于开发者工具这个判断 **在业务层面成立**。但需要注意两个问题：

- **Tab 键导航和键盘完整性是桌面应用的基本可用性，不是无障碍过度设计**。开发者用户群本身是键盘重度用户，完全无法使用 Tab 导航的桌面应用会显著影响效率
- 桌面应用（非 Web）不受 WCAG 约束，但基本操作键盘可达是桌面平台用户的核心预期

建议：不要求设计师修改设计文档，但 Plan 阶段的编码规范中应加入"所有可交互元素可通过 Tab + Enter 访问"的要求。

### 3.5 i18n 架构细节

> 理由：Plan 阶段按需决定

**判定：✅ 合理**

v3 §6.2 的"错误码 key + 前端 i18n 渲染"策略已经解决了 v5 最关键的批评（后端硬编码中文错误消息）。占位符格式、fallback locale 等细节确实可以在 Plan 阶段决定。

### 3.6 平台窗口装饰

> 理由：视觉设计范畴

**判定：✅ 合理**

窗口装饰属于视觉设计文档范畴。业务功能设计中不做这一层是正确的分工。

### 3.7 搜索维度扩展

> 理由：已标注 svn CLI 限制，后续迭代

**判定：✅ 合理**

v3 §5.5 已注明 CLI 限制。搜索维度扩展需要自定义实现或服务端支持，MVP 不做是正确的优先级判断。

### 3.8 小结

7 项反馈中 **6 项合理，1 项部分合理但有风险**。设计师的决策判断整体成熟，未出现明显的功能遗漏。唯一值得关注的争议点是"辅助功能"——建议不强推设计文档修改，但 Plan 阶段应确保键盘导航可用性。

## 四、v3 版本中仍存在的盲点

以下问题在 v5 中未明确指出、v3 也未覆盖。

### 4.1 ⚠️ 网络离线状态未处理

**问题描述**：桌面应用（尤其是笔记本电脑）需要处理网络中断场景。当前所有需要与服务器交互的 SVN 操作（log/diff/update/checkout/switch/merge 等）都假设网络可用。实际上只有 `svn status` 可离线执行。

| 场景 | 后果 |
|------|------|
| 用户打开 Free-SVN，WiFi 断开 | 工具栏按钮全部可用但操作必然超时 |
| 编辑文件后想提交，但无网络 | 提交按钮可用但必然失败 |
| svn update 到一半断网 | 操作进入互斥锁，窗口卡住 |

**建议**：§10 的非功能需求中增加一条 `SVN 操作执行前检测网络可达性，服务端操作不可达时禁用对应按钮并提示`。或至少标注为已知局限。

### 4.2 ⚠️ 二进制文件差异对比无定义

**问题描述**：SVN 仓库中常见非文本文件（图片、PDF、`.exe`、`.jar` 等），`svn diff` 对二进制文件输出 `Cannot display: file marked as a binary type`。

| 场景 | 当前设计 | 问题 |
|------|---------|------|
| 文件列表中有 `.png` | 显示 [差异] 按钮 | 点击后 diff2html 试图渲染二进制 diff → 渲染错误 |
| `.jar` 文件被修改 | 差异视图显示 diff 失败 | 无降级提示 |

**建议**：§5.3 差异对比视图中补充一行：`二进制文件点击差异时显示「该文件为二进制文件，无法显示差异」提示，不展开差异视图`。

### 4.3 ⚠️ 外部工具合并路径传递未定义

**问题描述**：§5.11 冲突解决的 MVP 策略是"查看差异 + 标记已解决"，推荐使用外部工具手动编辑解决冲突。但如何将冲突文件的三个版本（mine/base/theirs）传递给外部工具完全没有定义。

| 冲突类型 | 需要传递的参数（按 §5.7 命令模板） |
|---------|----------------------------------|
| text conflict | `<file.mine> <file.rOLD> <file.rNEW> <output>` |

**当前问题**：用户点击"使用外部工具"后，应用是自动调用合并工具传入四个文件？还是只提示用户去手动找冲突文件？§5.7 定义了外部差异/合并工具的命令模板，但冲突解决场景没有说如何使用这些模板。

**建议**：在 §5.11 中补充一行：`冲突解决的外部工具调用：根据预设的合并工具命令模板（见 §5.7），自动传入 <mine> <base> <theirs> <output> 四个参数`。这个决策如果在设计文档中不做，编码时缺少依据。

### 4.4 ⚠️ svn:externals 处理策略未定义

**问题描述**：很多 SVN 仓库使用 `svn:externals` 引入外部依赖。在 checkout/update/export 时是否处理 externals 会显著影响用户体验。

| 操作 | 当前设计 | 问题 |
|------|---------|------|
| checkout | 未提及 externals | 检出大仓库可能自动拉取外部依赖，花费额外时间 |
| update | 未提及 externals | 同上 |
| export | 未提及 | 导出是否包含 externals？ |

**建议**：§5.6 检出对话框增加 `忽略外部定义(svn:externals)` 复选框，默认不勾选。同时在 §6.1 checkout/update 参数中列出 `ignoreExternals` 选项。

### 4.5 🔷 日志分页策略不完整

**问题描述**：§5.5 定义"分页加载版本历史（默认 50 条/页）"，但 svn log 的分页实现策略未定义。`svn log --limit 50` 只能获取最新 N 条，翻页需要知道当前最后一版的 revision 号。

**影响**：纯实现细节，不影响设计完备性。Plan 阶段注意 `第二页使用 svn log --limit 50 -r <last_revision-1>:0` 即可。

### 4.6 🔷 recentWorkspaces 无上限

**问题描述**：§8.4 中 `recentWorkspaces` 数组无限增长。如果用户切换了 100 个工作副本，JSON 持续膨胀。

**影响**：轻度（增加约 1KB/条）。Plan 阶段设为"最多保留 20 个"即可。

### 4.7 🔷 输入校验边界

**问题描述**：URL 输入框、版本号输入框无格式校验。§2.6 仅校验非空。

**影响**：用户输入错误格式（全角冒号、版本号输入字母）后执行 svn 才报错，体验差。Plan 阶段基本前端校验（正则 + 纯数字限制）即可解决。

## 六、结论

### 6.1 设计师反馈评估结论

设计师的 7 项未采纳理由中 **6 项合理、1 项部分合理**。设计师整体判断成熟，v3 体现了对 v5 评估的系统性响应。唯一值得关注的争议点是"辅助功能"：虽然高对比度/屏幕阅读器在 MVP 阶段确实过重，但桌面应用的 Tab 键导航和键盘操作完整性应视为基本可用性而非无障碍过度设计。

### 6.2 总体评估

v3 是对 v2 的重大升级。从 v5 到 v3 的修复覆盖率达到 **16/23 ≈ 70% 的评估发现项**（16 项完全修复，7 项合理推迟）。新发现 4 个功能盲点（网络离线、二进制 diff、外部合并路径传递、svn:externals 策略），均为非关键性问题。
