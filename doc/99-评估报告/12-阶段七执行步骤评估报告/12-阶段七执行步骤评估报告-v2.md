#### P1 — translateError 缺失后端 AppError 错误码前缀匹配（新增）

`translateError` 函数仅匹配 SVN CLI 原始 E- 错误码（E155037、E200009 等），但后端 AppError 通过 `#[error("ERROR_CODE: {0}")]` 返回的是带前缀的字符串。

后端返回的实际格式（后端规范 §03）：

```
"SVN_EXEC_FAILED: E155037: ..."       ← AppError::SvnCommand
"SVN_TIMEOUT: svn 命令执行超时"        ← AppError::Timeout
"SVN_CANCELLED"                        ← AppError::Cancelled
"NETWORK_UNREACHABLE"                  ← AppError::NetworkUnreachable
"SVN_NOT_FOUND"                        ← AppError::SvnNotFound
```

需要先按**前缀**匹配，再按 E- 码匹配：

```ts
function translateError(msg: string): string {
  // 1. 优先匹配后端错误码前缀
  if (msg.startsWith('SVN_TIMEOUT')) return 'error.operationTimeout'
  if (msg.startsWith('SVN_CANCELLED')) return 'error.operationCancelled'
  if (msg.startsWith('SVN_NOT_FOUND')) return 'error.svnNotFound'
  if (msg.startsWith('NETWORK_UNREACHABLE')) return 'error.networkUnreachable'
  if (msg.startsWith('SVN_OPERATION_IN_PROGRESS')) return 'error.operationInProgress'
  if (msg.startsWith('INVALID_INPUT')) return 'error.invalidInput'
  if (msg.startsWith('IO_ERROR')) return 'error.ioError'
  if (msg.startsWith('REPO_ERROR')) return 'error.repoError'
  if (msg.startsWith('SVN_PARSE_FAILED')) return 'error.parseFailed'

  // 2. 匹配命令执行错误中的 SVN CLI E- 码
  if (msg.includes('E155037')) return 'error.workingCopyClean'
  // ... 其余 E- 码

  return 'error.unknown'
}
```

#### P1 — translateError 缺少 6 个后端 AppError 错误码（新增）

后端 `AppError` 有 11 个变体，`translateError` 当前映射覆盖不足：

| 缺失的错误码 | 后端枚举变体 | 影响 |
|-------------|-------------|------|
| SVN_EXEC_FAILED | SvnCommand | CLI 执行失败无对应翻译 |
| SVN_CANCELLED | Cancelled | 用户取消操作无对应翻译 |
| SVN_NOT_FOUND | SvnNotFound | svn 不可用时无提示 |
| SVN_OPERATION_IN_PROGRESS | OperationInProgress | 写操作冲突无翻译 |
| NETWORK_UNREACHABLE | NetworkUnreachable | 网络不可达无翻译 |
| SVN_PARSE_FAILED | ParseFailed | XML 解析失败无翻译 |

#### P2 — onOperationError 使用 as any（新增）

```ts
return listen('operation:error', (e) => cb(e.payload as any))
```

`as any` 违反 TypeScript strict 模式和前端规范 §03"禁止使用 any"。

**建议：** 定义类型并做类型收窄：

```ts
export function onOperationError(cb: (e: { errorCode: string; message: string }) => void) {
  return listen<{ errorCode: string; message: string }>('operation:error', (e) => cb(e.payload))
}
```

#### P2 — ElMessage 导入未使用（新增）

```ts
import { ElMessage } from 'element-plus'
```

`wrappedInvoke` 捕捉到错误后重新抛出，未使用 `ElMessage` 做用户提示。该导入是死代码。

**建议：** 删除未使用的 `ElMessage` 导入，或由 `wrappedInvoke` 内部直接展示错误 toast：

```ts
async function wrappedInvoke<T>(cmd: string, args?: Record<string, unknown>): Promise<T> {
  try { return await invoke<T>(cmd, args) }
  catch (e) {
    const msg = typeof e === 'string' ? e : e instanceof Error ? e.message : '未知错误'
    const translated = translateError(msg)
    ElMessage.error(translated)  // 统一错误 toast
    throw translated
  }
}
```

#### P2 — 方法数量声明不准确（新增）

步骤标题声称"26 个 invoke 方法"，但方法清单仅列出 25 个（#1-#25）。缺少 `checkNetwork`（对应后端 `check_network` 命令，被 `useNetworkStatus` composable 调用）。

#### P1 — Windows 缺少文件级右键菜单注册（新增）

业务设计 §3.1 明确要求：
- **工作副本目录上右键** → 12 项完整菜单
- **工作副本内文件上右键** → 同目录菜单，但操作作用于当前文件

当前 7.4 的 Windows 实现仅在 `HKEY_CLASSES_ROOT\Directory\Background\shell\` 下注册（目录背景右键），缺少 `HKEY_CLASSES_ROOT\*\shell\`（文件右键）的注册。

**建议：** `register()` 中补充文件级注册：

```rust
const FILE_KEY: &str = r"HKEY_CLASSES_ROOT\*\shell";
// 对每个菜单项，同时在 WC_KEY、FILE_KEY 下注册
```

#### P2 — Windows reg query 输出解析依赖英文区域（新增）

`reg query` 输出格式随 Windows 区域设置变化（中文系统输出包含"注册表项"等文字）。`line.contains("SVN-")` 匹配后 `reg delete` 的参数可能包含非路径文字。

**建议：** 在注释中标注已知局限，或改用 WinAPI 操作注册表。

#### P2 — macOS .workflow bundle 文档不完备（新增）

macOS 注册代码中 `document.wflow` 写入的是空 XML，实际的 AppleScript 执行脚本未包含。Automator 工作流需要 `<key>AMWorkflowActions</key>` 包裹的实际操作定义才能生效。

**建议：** 补充 `document.wflow` 的完整 XML 结构示例，或注明需手动在 Automator 中配置。

## 跨步骤一致性问题

### P1 — services/svn.ts 与 stores/svn.ts 职责冲突

| 来源 | 描述 | 职责 |
|------|------|------|
| 前端规范 §05 Pinia 示例 | `stores/svn.ts` — "封装全部 invoke 调用" | Store 直接调用 invoke |
| 步骤 7.2 | `services/svn.ts` — "前端唯一调用后端入口" | Service 封装 invoke |
| 开发计划 v2 §4.2 | stores/svn.ts 和 services/svn.ts 都提到 invoke 封装 | 两者重复 |

AI 按照前端规范示例会创建 `stores/svn.ts` 直接调用 invoke，但按步骤 7.2 应调用 `services/svn.ts`。二者冲突导致 AI 产生重复的 invoke 封装，绕过 `wrappedInvoke` 的统一错误处理。

**建议：** 在步骤 7.2 的操作细节中**显式声明**：

```ts
// stores/svn.ts 应从 services/svn.ts 导入，不直接调用 invoke
import { getStatus, createCommit } from '@/services/svn'
```

或调整开发计划，明确 `stores/svn.ts` 是 store 层（状态管理），`services/svn.ts` 是 service 层（无状态 invoke 封装）。
