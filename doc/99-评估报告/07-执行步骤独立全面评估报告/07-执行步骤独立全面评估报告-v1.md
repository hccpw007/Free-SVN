## 二、问题列表

### 🔴 P1 — 必须修正（影响编码正确性）

**P1-01：check_network 未注册为 Tauri Command**

- **涉及步骤**：`3.4/lib.rs` → `7.2/services/svn.ts`
- **问题描述**：`7.2` 的 services/svn.ts 方法清单第 24 项声明了 `checkNetwork` 方法（调用后端 `check_network` 命令），但 `3.4` 的 `generate_handler!` 中**未注册** `check_network` 命令。后端 `svn/executor.rs` 中的 `check_network` 是普通公有函数（`pub async fn`），未标注 `#[tauri::command]` 且未出现在 `generate_handler!` 中。前端调用 `invoke('check_network')` 将导致运行时错误。
- **影响范围**：`useNetworkStatus.ts` 依赖此命令实现网络检测，整个网络离线检测功能无法工作
- **建议修复**：在 `commands/` 中新增 `check_network.rs`（或追加到已有 command 文件），注册为 Tauri Command，并在 `generate_handler!` 中注册

**P1-02：settings store 中使用了不存在的 Rust Command**

- **涉及步骤**：`4.2/stores/settings.ts`
- **问题描述**：`settingsStore.load()` 调用 `invoke<AppSettings>('load_settings')`，`settingsStore.save()` 调用 `invoke('save_settings')`，但后端 `config/store.rs` 中的 `load_settings()` 和 `save_settings()` 是普通函数（非 `#[tauri::command]`），且未注册在 `generate_handler!` 中。后端规范 §02 的 command 清单中也未包含这两个命令。
- **影响范围**：设置页面的保存/加载功能完全无法工作
- **建议修复**：在 `commands/` 中新增 settings 相关的 command 文件，或者调整前端直接使用 `@tauri-apps/plugin-store` 的 JavaScript API（取消除 store 的命令外的前端直接调用）

### 🟡 P2 — 建议修正（影响代码质量或一致性）

**P2-01：generate_handler! 收录 command 数量与设计文档不一致**

- **涉及步骤**：`3.4/lib.rs`
- **问题描述**：
  - `3.4` 中声明 generate_handler! 包含 **25** 个 command
  - 后端规范 §02 命令清单列出 **26** 个操作（含 check_network）
  - `7.2` services/svn.ts 方法清单列出 **26** 个方法（含 checkNetwork）
  - 数量不一致说明存在遗漏（check_network 未注册）
- **建议修复**：统一为 26 个（补充 check_network 注册）

**P2-02：`property_ops` 参数结构与前端调用不匹配**

- **涉及步骤**：`3.3/ignore.rs` vs `4.2/svn.ts`
- **问题描述**：后端 `property_ops` 声明为 `#[tauri::command] pub async fn property_ops(path: String, prop_name: Option<String>, action: Option<String>, prop_value: Option<String>, state: ...)`，使用了**分别的参数注入**（flat params）。但前端 `stores/svn.ts` 调用方式为 `call<string>('property_ops', { path, prop_name: propName, action, prop_value: propValue })`，外部又套了一层 `{ params: ... }` 包装。Tauri 2 的 flat params 调用格式需确认：后端定义是 flat params 还是 struct Params。
- **影响范围**：property_ops 的调用可能因参数格式不匹配而失败
- **建议修复**：统一参数传递方式，建议后端统一使用 `#[derive(Deserialize)]` 结构体（与 `3.1`/`3.2` 中其他 command 一致），前端按结构体传参

**P2-03：commit 的 args 构造缺少 `cwd` 参数合理性验证**

- **涉及步骤**：`3.2/commit.rs`
- **问题描述**：`create_commit` 中调用 `run_svn(..., &params.paths[0])` 作为 cwd，但 `params.paths` 是文件路径数组，第一个文件路径可能不在顶层工作目录下，导致 svn 的 cwd 设置为子目录。`svn commit` 通常需要在工作副本根目录或文件所在目录执行。
- **建议修复**：增加逻辑获取工作副本根目录作为 cwd，或确认 Tauri State 中保存的 workspace 根路径

**P2-04：StatusBar 的 `lastCommitTime` 更新时机未明确**

- **涉及步骤**：`5.4/StatusBar.vue`
- **问题描述**：StatusBar 显示 `ws.lastCommitTime`，但该字段的更新时机在步骤中未明确——是 `get_info` 时一并获取？还是 commit 完成后更新？步骤只说"由 workspaceStore 管理"，但 workspaceStore 中 refresh 方法的调用时机不完整。
- **影响范围**：StatusBar 可能持续显示空状态
- **建议修复**：补充说明刷新时机（如 `onMounted` + `get_info` + commit 完成回调中更新）

**P2-05：Linux Nautilus 脚本多文件路径处理问题**

- **涉及步骤**：`7.4/linux.rs`
- **问题描述**：Nautilus 脚本中 `commit` 等脚本使用 `"$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS"` 传递多个文件路径，但 Nautilus 脚本的该变量是多行字符串（每行一个路径），如果按单字符串处理，会将所有路径作为一个参数传给应用。注册脚本中缺乏逐行读取的处理逻辑。
- **建议修复**：在脚本中写 `while IFS= read -r file; do ... done <<< "$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS"` 模式来处理多文件选择

**P3-02：「更新到版本」按钮在工具栏中的位置与视觉设计不一致**

- **涉及步骤**：`5.3/ToolBar.vue`
- **问题描述**：视觉设计 §4.1 的工具栏中未包含「更新到版本」按钮（仅在业务设计 §4.3 中新增），但 `5.3` 的按钮列表中包含了它。功能补充是正确的，但建议确认该按钮在溢出策略中的优先级排序（当前 priority=2，容易被折叠到溢出菜单）。
- **建议修复**：确认优先级安排，或考虑与「更新」按钮合并为下拉菜单

- **问题描述**：ja.json/ko.json 当前为预留骨架（空字符串），但 `main.ts` 的 `createI18n` 中同时导入了 4 个语言包，如果用户切换到日语/韩语，UI 将大面积显示空字符串。建议在骨架阶段至少复制英文翻译，或添加语言切换的可用性限制提示。
- **建议修复**：MVP 阶段将 ja/ko 的 fallback 设为 `en`（当前已是），但建议在设置页的语言选项中隐藏日语/韩语选项直到翻译完成
