| **整改建议** | 建议将此项验收标准移至 phase-6 6E 步骤中，或标注为"前向参考（后续阶段实现）" |

#### D-2：SVN_AUTH_FAILED locale 依赖

| 项目 | 内容 |
|------|------|
| **问题描述** | `svn store` 的 `call<T>()` 方法中，通过 `msg.includes('SVN_AUTH_FAILED')` 检测认证失败。但 `getErrorMessage()` 会将错误码翻译为 locale 文本（中文 locale 下返回"认证失败，请检查..."），该文本中不包含 `'SVN_AUTH_FAILED'` 字符串。因此 auth detection 在非英文 locale 下失效。同理 `msg.includes('E170001')` 也可能在已翻译消息中不可用 |
| **实际风险** | 在用户使用中文界面时需要 SVN 认证时，authContext 不会被填充，authFailed 保持为 false，因此 AuthDialog 将无法自动弹出。用户可能持续看到通用错误提示而非认证重试对话框 |
| **整改建议** | 方案 A（推荐）：在 `getErrorMessage()` 中同时返回原始错误码和翻译文本，或增加一个 `getErrorCode()` 函数返回未翻译的错误码。方案 B：在 `call()` 的 catch 中直接检查 Tauri invoke 抛出的原始 error 对象的 `error` 属性（如 `(err as any)?.error === 'SVN_AUTH_FAILED'`），绕过错码翻译层进行检测 |

#### D-3：Event listener 未清理

| 项目 | 内容 |
|------|------|
| **问题描述** | `initEventListeners()` 调用了 4 次 `listen()`（'operation:progress'/'operation:started'/'operation:completed'/'operation:error'），但未存储返回的 `UnlistenFn` 句柄。没有在 store 销毁或组件卸载时执行 `unlisten()` |
| **实际风险** | 低风险。`initialized` guard 防止了重复注册，因此在正常使用生命周期中不会产生多个监听实例。但在 HMR 热重载或组件频繁重建场景下可能导致孤立监听 |
| **整改建议** | 在 `initEventListeners()` 中将 4 个 `UnlistenFn` 存入数组，提供对应的 `destroy()` 或 `cleanup()` 方法供 `onUnmounted` 调用 |
