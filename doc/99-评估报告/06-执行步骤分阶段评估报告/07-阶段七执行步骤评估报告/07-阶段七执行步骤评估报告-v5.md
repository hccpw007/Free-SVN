## 评估结论

```
发现 P1 问题：2 项
发现 P2 问题：3 项
```

v4 终评认定"零 P1 零 P2"，但本轮独立评审发现以下被遗漏的问题：

- **P1-01**：7.2 translateError 缺少 `SVN_AUTH_FAILED` 错误码映射 → 业务设计 §2.2 和开发计划 v2 明确要求区分"认证失败"与"网络超时"
- **P1-02**：7.1 useNetworkStatus 引入 `@/services/svn`，但 7.1 执行先于 7.2，导致步骤执行顺序矛盾
- **P2-01**：7.3 声称 generate_handler! 共 25 个，实际应为 26 个（缺 `check_network` + `property_ops`）
- **P2-02**：7.4 Linux Nautilus 脚本 `$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS` 多选时换行分隔路径会作为单参数传递
- **P2-03**：7.5 暗色模式 grep 命令仅覆盖已知 Tailwind 类，遗漏 arbitrary value 语法和其他颜色类

**总体评价：** 所有 6 个步骤文件结构完整，"涉及文件/操作细节/验收标准"三个字段齐全，业务功能覆盖率和视觉设计覆盖率接近 100%。上述 5 项问题修复后可直接用于 AI 编码，无需再次评审。

## v4 遗漏问题详情

### P1-01 — 7.2 translateError 缺少 SVN_AUTH_FAILED

**业务来源：** 业务功能设计 §2.2（SVN 认证交互策略）明确定义了认证失败处理，开发计划 v2 新增 `SVN_AUTH_FAILED` 错误码及其 i18n 翻译。

**问题描述：** 步骤 7.2 `services/svn.ts` 的 `translateError` 函数映射了 11 个错误码，但未包含 `SVN_AUTH_FAILED`。当后端返回此错误码时，前端无法翻译，用户看到的是原始错误码文字。

**影响：** 业务设计 §2.2 要求区分"网络问题"和"认证问题"并给出配置指引，缺失此映射后认证失败场景的提示无法正确显示。

**修复方案：** 在 translateError 函数中加入：
```ts
if (msg.startsWith('SVN_AUTH_FAILED')) return 'error.authenticationFailed'
```
同时确保 locales/{zh-CN,en,ja,ko}.json 中包含：
- `"error.authenticationFailed": "认证失败，请检查凭据配置或使用 SSH key / ~/.subversion/auth/ 缓存"`（中文）
- 其余语言的对应翻译

### P1-02 — 7.1 与 7.2 执行顺序矛盾

**问题描述：** 步骤 7.1 的 `useNetworkStatus.ts` 包含：
```ts
import { checkNetwork } from '@/services/svn'
```
但步骤 7.1 的执行排期在 7.2 之前（7.1 → 7.2 → ...），而此时 `services/svn.ts` 尚未创建。当 AI 按顺序执行步骤 7.1 时，会遇到 TypeScript 编译错误。

**修复方案：** 方案 A：交换 7.1 和 7.2 的执行顺序（推荐）。方案 B：在 7.1 中暂时使用 `import { invoke } from '@tauri-apps/api/core'` 直接调用，并在步骤 7.2 说明需要重构此引用。

推荐方案 A，因为 `useNetworkStatus` 除 `checkNetwork` 外还间接依赖 services/svn.ts 的整体错误处理体系。

### P2-01 — 7.3 generate_handler! 数量不准确

**问题描述：** 步骤 7.3 描述"command 注册（generate_handler! 共 25 个）"，但实际后端命令应有 26 个（对照业务设计 §6.1 + 步骤 7.2 方法表）。

**差异明细：**
- 后端规范示例的 generate_handler! 列出 24 个 command
- `check_network`（方法 #24）是一个独立 command（需新增注册）
- `property_ops`（开发计划 3C.7）是一个独立 command（需新增注册，与 set_ignore 共享同一文件但函数不同）
- 因此实际应为 26 个 command，而非 25 个

**修复方案：** 将"共 25 个"更正为"共 26 个（含 check_network、property_ops）"，并确保 generate_handler! 注册清单包含全部 26 个 command。

### P2-02 — 7.4 Linux Nautilus 多路径传递问题

**问题描述：** 步骤 7.4 的 `linux.rs` 中，Nautilus 脚本内容为：
```bash
#!/bin/bash
"/path/to/free-svn" --svn-cmd commit "$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS"
```
当用户多选文件时，`$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS` 变量包含**换行符分隔**的文件路径列表。被双引号包裹时会作为单参数传递，导致应用收到的 `"$@"` 参数异常，路径解析失败。

**修复方案：** 改用 `while IFS= read -r file; do ... done <<< "$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS"` 逐路径处理，或改为将所有路径合并为空格分隔的参数列表：
```bash
#!/bin/bash
FILES=()
while IFS= read -r file; do FILES+=("$file"); done <<< "$NAUTILUS_SCRIPT_SELECTED_FILE_PATHS"
"/path/to/free-svn" --svn-cmd commit "${FILES[@]}"
```

### P2-03 — 7.5 暗色模式 grep 搜索命令不完整

**问题描述：** 步骤 7.5 提供了 7 条 grep 命令搜索已知 Tailwind 类（bg-white、bg-slate-50、text-slate-900 等），但有以下遗漏：
1. 遗漏 `text-blue-600`（差异操作按钮色）、`text-amber-600`（还原按钮色）等组件特定色
2. 遗漏 `bg-blue-50`（选中行背景色）已在替换表中但不在搜索命令中
3. 遗漏 Tailwind 任意值语法（如 `bg-[#123456]`）的搜索
4. 遗漏 `border-l-3` 等冲突行指示类
5. 仅搜索 `.vue` 和 `.ts` 文件但部分 Tailwind 类可能出现在其他文件类型中

**修复方案：** 补充以下 grep 命令：
```bash
# 追加搜索目标 — 操作按钮颜色类
grep -rn 'text-blue-600\|text-amber-600\|text-red-600\|text-violet-600' src/ --include='*.vue'
# 追加搜索目标 — 行状态背景类
grep -rn 'bg-blue-50\|bg-red-50\|bg-green-50' src/ --include='*.vue'
# 追加搜索目标 — 任意值语法
grep -rn 'bg-\[.*\]' src/ --include='*.vue'
# 追加搜索目标 — 冲突行指示
grep -rn 'border-l-\|border-red' src/ --include='*.vue'
```
