#### 🔴 P1-v3-1：OperationProgress 结构体字段与实际代码严重不一致

**严重程度：高**

步骤 8.1 在 "OperationProgress 事件载荷" 节显示了以下结构体：

```rust
// 步骤 8.1 中展示的字段（错误版本）
pub struct OperationProgress {
    pub operation: String,
    pub percent: f64,            // ← 实际是 u8
    pub speed: String,           // ← 实际是 Option<String>
    pub completed_count: u32,
    pub pending_count: u32,
    pub elapsed: String,         // ← 实际是 Option<String>
}
```

但 `src-tauri/src/svn/types.rs` 中的实际定义：

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct OperationProgress {
    pub operation: String,
    pub percent: u8,
    pub stage: String,              // ← 步骤缺失此字段
    pub file_count: u32,           // ← 步骤缺失此字段
    pub completed_count: u32,
    pub pending_count: u32,
    pub speed: Option<String>,     // ← 步骤中用 String
    pub elapsed: Option<String>,   // ← 步骤中用 String
    pub current_lines: Vec<String>, // ← 步骤缺失此字段
}
```

**关键差异：**

| 字段 | 步骤 8.1 | 实际 types.rs | 影响 |
|------|---------|-------------|------|
| `percent` | `f64` (0.0~100.0) | `u8` (0~100) | AI 按步骤编码会产生类型不匹配 |
| `speed` | `String` | `Option<String>` | 序列化行为不同 |
| `elapsed` | `String` | `Option<String>` | 序列化行为不同 |
| `stage` | **缺失** | `String` | **步骤 8.4 实际代码使用 `progress?.stage`** |
| `file_count` | **缺失** | `u32` | 影响文件计数统计 |
| `current_lines` | **缺失** | `Vec<String>` | 影响当前行显示 |

**结论：** 步骤 8.1 为了简化而展现了一个"概念性"的 OperationProgress，但该结构与代码实际定义不一致。AI 编码时若以步骤为准而非设计文档或实际代码，可能：
1. 修改 types.rs 产生编译错误（字段增减导致下游 consumer 不兼容）
2. 前端无法正确接收 `stage` 字段（步骤 8.4 的实际代码依赖此字段）
3. `speed` 和 `elapsed` 的 None/空字符串边界处理不一致

**修复建议：** 将步骤中 OperationProgress 的字段定义替换为与实际 types.rs 一致的定义（含全部 9 个字段的正确类型和 serde 属性），或删除步骤中的结构体定义，改为引用设计文档 §5.5。
